## VOD 트랜스코딩이 왜 필요한가

<img width="1083" height="759" alt="image" src="https://github.com/user-attachments/assets/5d14f81d-737e-4ea8-9e47-2b768ad25510" />


VOD 서비스는 다양한 기기와 네트워크 환경에서도 영상이 안정적으로 재생되도록 콘텐츠를 미리 가공한다.

이 가공 과정에서 컨테이너 포맷, 해상도, 코덱, 비트레이트를 서비스 정책에 맞게 다시 만드는 일을 트랜스코딩이라고 정의한다.

트랜스코딩은 “내용은 같지만 재생에 더 적합한 새 영상”을 만드는 과정이다.

## 트랜스코딩이 오래 걸리는 이유

<img width="1165" height="357" alt="image" src="https://github.com/user-attachments/assets/b7f241d4-0342-4958-94df-3a32b58ea18d" />


트랜스코딩 파이프라인은 크게 “분리 → 복원 → 처리 → 재압축 → 포장” 흐름으로 진행된다.

먼저 Demuxer가 컨테이너에서 비디오 스트림과 오디오 스트림을 분리한다.

분리된 스트림은 대개 코덱으로 압축된 비트스트림이므로 Decoder가 디코딩해서 원시 프레임과 오디오 샘플로 복원한다.

그 다음 필요한 필터링이나 리사이즈 같은 처리를 수행한 뒤 Encoder가 다시 압축하고 Muxer가 컨테이너로 포장한다.

이 과정에서 가장 시간이 많이 드는 구간은 보통 비디오 인코딩이며 이는 코덱 연산량이 매우 크기 때문이다.

따라서 순차 방식으로 전체 영상을 처음부터 끝까지 처리하면 영상 길이에 비례해서 처리 시간이 늘어난다.

## 분산 트랜스코딩

분산 처리는 큰 작업을 작은 작업으로 쪼개 병렬로 처리하고 결과를 모아 최종 결과를 만드는 방식이다.

분산 트랜스코딩은 원본 영상을 여러 구간으로 나누고 구간별로 트랜스코딩한 뒤 시간 순서대로 다시 병합하는 방식이다.

초기에는 전체 시간의 대부분을 비디오 인코딩이 차지하므로 비디오 중심으로 분산을 먼저 적용했다.

## 분산 트랜스코더의 구조

- Distributor: 원본 영상을 분할하고 분배한다. 구간별 트랜스코딩 결과를 병합한다.
- Worker: Distributor가 분배한 구간의 영상(세그먼트)을 트랜스코딩한다.
- Shared Storage: NAS(network attached storage)를 사용해 영상을 저장한다.

## 프로토타입의 실행 과정과 오버헤드

<img width="1133" height="767" alt="image" src="https://github.com/user-attachments/assets/7506f531-0d2a-4dbd-b6df-4a51e111aec1" />


1. Distributor가 NAS에서 원본 영상을 읽어 일정한 시간 길이로 분할한다.
2. Distributor가 분할한 영상을 NAS에 저장한다.
3. Distributor가 각 세그먼트를 트랜스코딩할 Worker에 해당 세그먼트의 위치를 전달한다.
4. Worker가 NAS에서 분배된 세그먼트를 읽어 트랜스코딩을 실행한다.
5. Worker가 세그먼트를 트랜스코딩한 결과를 NAS에 저장한다.
6. Distributor가 세그먼트의 트랜스코딩 결과를 NAS에서 읽어 하나의 파일로 병합한다.

위의 과정은 모두 순차로 실행된다. 예를 들어 1번 과정에서 분할된 원본 영상이 모두 NAS에 저장되기 전에는 3번 과정이 시작되지 않는다. 그리고 4번 과정과 5번 과정의 세그먼트 인코딩과 저장이 모두 완료되기 전에는 6번의 병합 과정이 시작되지 않는다.

### 문제점

1. NAS I/O가 과도하게 많고, 트래픽이 NAS에 집중된다.
2. 원본 영상을 세그먼트로 분할하는 과정에서 영상 전체를 읽어야 한다.
3. 분할된 원본 영상을 별도의 파일로 전달하는 방법에는 제약 사항이 있다.

## 분산 트랜스코더의 구조와 프로세스

<img width="1202" height="766" alt="image" src="https://github.com/user-attachments/assets/e1b4fd26-bc04-46d5-8d10-15c5aaf9a32b" />


1. Distributor가 NAS에 있는 원본 영상에서 최소한의 정보만 가져와 분할 구간을 결정한다.
2. Distributor가 트랜스코딩해야 하는 구간에 대한 정보(time offset)만 Worker에 전달한다.
3. Worker가 Distributor가 할당한 구간을 NAS의 원본 영상에서 읽고 트랜스코딩을 실행한다.
4. Worker가 트랜스코딩 결과를 실시간으로 Distributor로 전송한다.
5. Distributor가 Worker로부터 받은 트랜스코딩 결과를 임시 파일로 저장하면서 동시에 병합을 실행한다.

### 이점

1. 분할 구간이 반드시 키 프레임으로 시작해야 한다는 제약이 사라진다.
2. 키 프레임 제약이 없기 때문에 분할 구간을 자유롭게 결정할 수 있다.
3. 분할 구간을 자유롭게 결정할 수 있기 때문에 분할 구간을 조절한 최적화가 가능해진다.
4. Distributor가 원본 영상의 헤더 정보만 가져와도 분할에 필요한 데이터는 충분해진다.

## 분할 구간의 길이를 조절한 최적화

<img width="1178" height="1038" alt="image" src="https://github.com/user-attachments/assets/527dfc3c-f809-44dc-88a4-49c673e4b3ca" />


Worker 장비 성능이 완전히 동일하지 않거나 운영 상태가 다르면 처리 시간이 달라진다.

같은 장비라도 동시에 다른 작업이 돌면 트랜스코딩 속도가 흔들린다.

따라서 “N등분으로 균등 분할”하면 어떤 Worker는 빨리 끝나 놀고 어떤 Worker는 늦게 끝나 전체 완료가 지연된다.

그래서 구간을 짧게 잘라 먼저 끝난 Worker에게 다음 구간을 추가로 할당하는 방식이 전체 리소스 활용을 높인다.

구간이 짧을수록 균형 분배가 쉬워지고 전체 완료 시간을 줄일 가능성이 커진다.

## 결론

원본을 분할 파일로 만들지 않고 Worker가 원본에 직접 접근하게 하며 결과를 실시간으로 병합하는 구조로 바꿔 NAS 병목과 키 프레임 제약을 함께 줄이는 방향으로 최적화했다.
