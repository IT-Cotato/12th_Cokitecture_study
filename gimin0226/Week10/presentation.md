# 카카오가 S2Graph를 만든 진짜 이유: RDB와 기존 그래프DB로는 안 되던 것들

카카오는 한동안 **MySQL + 애플리케이션 레벨 수동 샤딩**으로 소셜 그래프를 버텨왔다.

유저 수가 늘어나고, 친구 관계가 폭증하고, 서비스도 계속 추가되면서 단순히 DB 스펙을 높이거나 서버를 몇 대 더 두는 것만으로는 감당이 어려워졌다.

결국 물리적인 한계 때문에 “이건 저장하지 말자”라고 포기하는 데이터도 늘어났다.

**기존 그래프DB로는 카카오 수준의 소셜 그래프를 실시간으로 다루는 데 한계가 있었다.**

그래서 결국 S2Graph를 직접 만들게 된다.

---

## 문제점 1: RDB에 억지로 우겨 넣기 힘든 소셜 그래프

이걸 전통적인 RDB로 모델링한다고 가정해 보면, 보통 이런 테이블 구조를 떠올리게 된다.

- `user` 테이블: 2억 행
- `relation` 테이블 (m:n 관계): 수십억 행 (예시로 20억 행 수준만 잡아도 이미 빡빡하다)
- `activity` 테이블: **하루에만** 1,000만 행 이상 증가한다고 가정해도 금방 수억·수십억 행이 된다

여기서 한 가지 깔고 들어가는 제약이 있다.

대부분 DB에서 많이 쓰는 **32비트 정수(INT)**의 최대값은 약 21억이다.

즉, `AUTO_INCREMENT INT`로 PK를 만들면 최대 21억 개까지만 유니크한 ID를 찍을 수 있다.

활동 테이블은 하루에 1,000만 행씩 쌓인다고 가정하면,

단순 계산으로 200일 안에 20억을 넘어버린다.

그래서 이런 규모에서는 **PK, FK를 전부 64비트 정수(BIGINT)**로 설계해야 한다.

PK뿐만 아니라, 그 PK를 참조하는 FK들도 전부 64비트여야 한다.

## 문제점2: 기존 그래프DB의 한계

그렇다면 “그래프 데이터베이스”는 원래 이런 문제를 풀라고 만든 것 아닐까?

의외로 그렇지 않은 경우가 많다.

**기존 그래프DB/시맨틱 웹 계열 시스템들**은 대체로 이런 용도로 설계되었다.

- 미리 모아 둔 **정적인 데이터** (지식 베이스, 온톨로지, 카탈로그 등)를
- SPARQL 같은 쿼리와 **의미 추론(Reasoning, Inference)**으로 분석하는 용도

이런 시스템들은 보통 다음을 전제로 설계된다.

- 데이터는 한 번에 크게 넣지만, **자주 바뀌지는 않는다**
- 그 대신, **읽을 때 복잡한 질의와 추론을 허용**한다
- 쿼리 시간이 길어도 괜찮다. 몇 초~몇 분 단위의 OLAP(분석) 용도에 가깝다

하지만 카카오 같은 소셜 서비스는 완전히 반대다.

- 데이터는 **매일, 매초, 실시간으로 바뀐다**
    - 친구 추가/삭제, 채팅방 초대, 좋아요, 차단, 팔로우 등
- 새로운 관계와 활동이 매일 수십억 건씩 늘어난다
- 쿼리는 **사용자가 앱을 사용하는 순간마다 바로 응답해야 하는** 실시간 API다

즉, **기존 그래프DB는 “정적인 지식 그래프를 똑똑하게 읽는 용도”에 더 가깝고,
카카오 같은 “극단적 규모의 실시간 소셜 그래프”에는 잘 맞지 않았던 것**이다.

## 문제점3: 성능의 문제

카카오가 필요한 핵심 기능은 한 줄로 말하면 **실시간 BFS**이다.

- “이 유저 A의 친구 목록”
- “친구의 친구 중에서 조건을 만족하는 사람들”
- “이 컨텐츠에 반응한 사람들의 이웃”

**위와 같은 것들이 특정 노드 주변의 이웃을 너비 우선으로 확장해 가는 탐색(BFS)**이다.

카카오는 

> 1초에 6만 5천 번이나 BFS를 요청받는데,
그 중 하나당 0.05초 안에 답을 줘야 하는 상황이다.
> 

### RDB로 구현하면 벌어지는 상황

그래프를 RDB로 표현하면 보통 이렇게 된다.

- `user` 테이블
- `relation(src_id, dst_id, type, …)` 테이블

“친구의 친구” 같은 BFS를 SQL로 짜면 결국 다음과 같이 흘러간다.

- `relation`을 여러 번 **자기 자신과 JOIN**
- 중복 제거를 위한 `DISTINCT` 또는 `GROUP BY`
- 유저 상태/조건 필터를 위해 다른 테이블들과 추가 JOIN

즉, 전형적인 **분석용(OLAP) 쿼리** 모양이 된다.

문제는 카카오가 이 쿼리를 보고서가 아니라 **실시간 API**로 쓰고 싶다는 점이다.

게다가 피크 타임에는 이 무거운 쿼리를 **동시에 수천 개씩** 처리해야 한다.

이 상황은, “대형 데이터 웨어하우스급 쿼리를 카카오톡 API 서버처럼 쓰고 싶다”는 말과 크게 다르지 않다.

또한, 카카오가 과거에 했던 것처럼 **MySQL을 수동 샤딩**했다면 문제는 더 꼬인다.

- 유저 ID 범위에 따라 DB를 여러 개로 쪼갠 상황에서는
- “BFS를 하면서 필요한 이웃들이 여러 샤드에 흩어져 있을 수 있고”
- 애플리케이션이 여러 샤드에 직접 쿼리를 보내고, 결과를 합쳐서 정렬해야 한다

## 목표: 서버를 늘리면 성능도 같이 늘어나야 한다

트래픽이 폭증할 때, **한 서버의 스펙을 올리는 것(scale-up)**만으로는 한계가 있다.

결국 **서버 대수를 늘리는 수평 확장(horizontal scaling)**이 필수적이다.

여기서 중요한 것은:

- 서버를 2대에서 4대로 늘렸을 때
- 처리 가능한 QPS도 거의 2배 → 4배로 오르는지,
    
    즉 **선형에 가깝게 확장되는지**이다.
    

그런 구조를 만들려면

- 데이터를 여러 노드에 적절히 나누어 저장하고
- 특정 유저/관계에 대한 BFS 쿼리를 여러 노드에 분산해서 처리하고
- 노드 수가 늘어날수록 처리량도 비례해서 늘어나게 설계해야 한다.

## 목표: 실시간 업데이트

소셜과 모바일의 결합이 가져온 가장 큰 변화는 실시간성이다.

매일 혹은 매시간 배치로 데이터를 분석하고, 이렇게 분석된 정보를 기반으로 추천해서는 사용자들의 소비 속도를 맞출 수 없다. 

### 동적 랭킹 로직 지원

- 푸시 전략: 랭킹 로직을 동적으로 변경하기 어렵다.
- 풀 전력: 다양한 랭킹 로직을 적용할 수 있다.

### 푸시 방식

트위터는 피드를 구현하기 위해서 푸시 방식을 사용한다.

이 방식은 컨텐츠를 생상할 때 그 컨텐츠를 소비할 사용자들의 피드에 해당 컨텐츠를 추가한다.

장점: 

- 컨텐츠 소비 처리(타임라인을 읽을 때)가 간편하고 빠르다

단점: 

- 피드의 내용과 순서를 변경하기 어렵다
- 불필요한 데이터 추가(트위터를 접은 팔로어의 피드에도 새 트윗을 추가)로 인한 자원 낭비가 있다

### 풀방식

페이스북은 풀 방식을 사용한다.

사용자들이 볼 때 컨텐츠의 내용과 순서를 결정한다.

장점:

- 컨텐츠 생산 처리가 간편하고 빠르다

단점:

- 컨텐츠 소비 처리가 어렵고 느리다.
    
    → but, 피드의 내용과 순서를 변경할 수 있어서 최근에는 더 널리 사용된다.
    

<img width="1280" height="722" alt="image" src="https://github.com/user-attachments/assets/70ff7a48-9edd-4485-b76d-58650b2ad77e" />


가능하다면 풀 방식이 좋겠지만, 서비스의 특성에 따라 푸시 방식이 불가피한 경우도 있다.

실제 서비스에서는 풀과 푸시, 둘 다 필요하다.

# What is S2Graph

→ Storage-as-a-Service + GraphAPI = Realtime Breadth First Search

즉, 실시간 너비 우선 탐색을 위한 그래프 데이터 저장소와 그래프 API 서버이다 .

<img width="743" height="410" alt="image" src="https://github.com/user-attachments/assets/6d5637ee-373c-4da0-9a0c-57a1a0a53f8d" />


HBase를 그래프 데이터 저장소로 사용한다.

HBase/HDFS/Hadoop이 가진 성능, 확장성, 가용성을 그대로 흡수하면서, 실시간 너비 우선 탐색에 최적화된 형태로 그래프 데이터를 저장한다.

저장소 레어어는 물리적 구현체에 독립적으로 설계되어 MySQL을 저장소로 사용할 수도 있다.

그 밖에도 [**Kafka**](http://kafka.apache.org/)와 [**Spark**](http://spark.apache.org/) 기반의 배치 처리, Top카운터, A/B 테스트 등을 포함한 소셜 그래프 기반 서비스에 유용한 여러가지 기능을 포함하고 있다.
