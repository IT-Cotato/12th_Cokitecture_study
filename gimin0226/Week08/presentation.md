### 1. 문제의 시작: LLM과 LLM Search

기존의 LLM은 학습된 데이터 내에서만 답변할 수 있어, 최신 정보나 특정 웹사이트의 실시간 정보에 접근할 수 없다. 이를 보완한 것이 바로 **LLM Search**다.

- 위의 사진은 동일하게 GPT-40-mini를 기반으로 하는데 품질 차이가 발생했다.

LLM Search는 사용자의 질문에 답변하기 위해 LLM을 실시간 웹 검색 기능과 결합한다.

 즉, LLM이 웹을 검색해 찾은 여러 HTML 문서를 참고 자료로 삼아 답변을 생성하는 방식(RAG의 일종)이다.

여기서 새로운 문제가 발생한다. LLM에게 제공할 "참고 자료"인 HTML 문서는 광고, 메뉴 바, 헤더, 푸터 등 답변에 불필요한 "노이즈(Noise)"로 가득 차 있다.

LLM의 성능을 높이고 토큰 사용을 최적화하려면, 이 노이즈를 제거하고 순수한 **"본문"**만 정확하게 추출하는 기술이 필수적이다. 이 글은 SKT가 에이닷(A.)의 PAAS 프로젝트를 위해 개발한 HTML 본문 추출기의 핵심 원리를 다룬다.

---

### 2. HTML 본문 추출기 요구 조건

실시간 LLM Search에 사용될 추출기는 다음 조건을 만족해야 한다.

1. **속도 (1초 이내):** 사용자는 실시간 답변을 원한다. HTML 문서를 가져오는 데 3초를 쓴다면, 본문 추출은 1초 이내에 완료되어야 한다.
2. **정확성 (No Data Loss):** 가장 중요한 조건이다. 노이즈(광고 등)가 조금 섞이는 것(낮은 **Precision**)은 LLM이 어느 정도 감내할 수 있다. 하지만 본문의 중요 내용이 일부라도 유실되는 것(낮은 **Recall**)은 LLM에게 잘못된 정보를 주입하는 치명적인 결함이다.
3. **성능 (Benchmark):** `trafilatura`와 같은 기존의 유명한 본문 추출 라이브러리보다 안정적이거나 뛰어나야 한다.

### 3. 기존 방식들은 왜 실패했는가?

범용적인 본문 추출기를 만들기 위해 여러 아이디어를 시도했지만, 모두 명확한 한계가 있었다.

- **CSS 선택자 방식**
    - `article`, `main`, `.content`처럼 본문에 자주 쓰이는 HTML 태그나 CSS 클래스 목록을 만들어 필터링하는 방식이다. `trafilatura`가 이 방식을 주로 사용한다.
    - **한계:** 사이트마다 HTML 구조가 천차만별이다.
        
         나무위키처럼 독자적인 구조를 쓰는 사이트에서는 본문을 통째로 유실할 수 있다. 
        
        새로운 사이트가 생길 때마다 수동으로 CSS 목록을 관리해야 하므로 범용성이 없다.
        
- **키워드 활용 방식**
    - 사용자 쿼리나 HTML의 `og:tag` (제목, 설명)와 텍스트 유사도가 높은 영역을 본문으로 간주하는 아이디어다.
    - **한계:** 유사한 텍스트 "영역"의 범위를 정하기가 불가능하다. 
    키워드를 포함한 `<div>` 태그 하나만 추출하면 문장 하나만 남을 수 있고, 그 부모 태그를 선택하면 문서 전체가 선택될 수 있다.
- **Longest Child 방식**
    - HTML을 트리 구조로 봤을 때, 특정 깊이(depth)에서 가장 텍스트가 긴 노드를 본문으로 선택하는 방식이다.
    - **한계:** 본문의 최적 깊이가 사이트마다 다르다. 
    어떤 사이트는 `body` 바로 아래에 본문이 있지만, 어떤 사이트는 10단계(depth)나 들어가야 본문이 있다.

### 4. 엔트로피(Entropy)로 노이즈 식별하기

모든 웹사이트에 범용적으로 적용할 규칙을 찾기 위해, HTML의 DOM 트리 구조 자체에 집중했다. 

그 결과 본문과 노이즈는 자식 노드들의 텍스트 길이 비율에서 명확한 차이를 보인다는 것을 발견했다.

- **노이즈의 특징 (예: 메뉴 바)**
    - 내비게이션 바(`<nav>`)는 `<li>메뉴1</li>`, `<li>메뉴2</li>`, `<li>고객센터</li>` 같은 자식 노드들로 구성된다.
    - 이 자식들의 텍스트 길이는 `[4자, 4자, 4자]`처럼 **비율이 비슷비슷하다.** (1:1:1)
- **본문의 특징 (예: 기사 페이지)**
    - `<body>` 태그는 `[<header>(100자)], [<main>(5000자)], [<footer>(100자)]` 같은 자식들로 구성된다.
    - 텍스트 길이가 특정 자식(`<main>`)에게 **극도로 치우쳐 있다.** (1:50:1)

이 "비율이 치우친 정도"를 측정하기 위해 정보 이론의 **엔트로피** 개념을 도입했다. 엔트로피는 불확실성 또는 무질서도를 나타내는 값이다.

- **엔트로피가 높다:** 비율이 `1:1:1`처럼 비슷하다. 누가 중요한지 불확실하므로 무질서하다. **(노이즈)**
- **엔트로피가 낮다:** 비율이 `1:50:1`처럼 한쪽에 치우쳤다. `50`이 중요하다는 것이 확실하므로 질서가 잡혀있다. **(본문)**

즉, HTML 트리를 탐색하며 **엔트로피가 낮은(자식 노드 비율이 한쪽으로 치우친) 노드를 재귀적으로 찾아 내려가면** 핵심 본문에 도달할 수 있다.

### 5. 최종 구현

엔트로피만으로는 완벽하지 않았다. 트리의 맨 끝(leaf) 노드는 텍스트가 `[1자, 10자]`만 있어도 엔트로피가 낮게 계산되지만, 이는 본문이 아니다.

### 1. `rankScore`로 피처(Feature) 조합

단순 엔트로피 대신, 여러 피처를 조합한 `rankScore`라는 새로운 점수를 만들었다. `rankScore`는 낮을수록 본문일 확률이 높다.

- **텍스트 밀도(Density):** `(총 텍스트 길이) / (총 자식 노드 수)`. 태그는 적고 텍스트만 빽빽할수록(밀도가 높을수록) 본문일 확률이 높다.
- **`rankScore` 공식:** `(엔트로피 * 깊이) / (텍스트 밀도)`의 변형된 형태를 사용한다.
    - 이 공식은 **엔트로피가 낮고**, **텍스트 밀도가 높으며**, **너무 깊지 않은** 노드에게 가장 좋은 점수(낮은 `rankScore`)를 준다.

### 2. 방어 로직 (앙상블)

통계적으로 `rankScore`(엔트로피)만으로 본문을 찾는 경우는 5%에 불과했다. 95%는 방어 로직이 필요했다. 이는 단일 모델이 아닌 **앙상블(Ensemble)** 방식을 사용했다는 의미다.

1. 일단 `rankScore`가 가장 낮은 노드를 '후보 본문'으로 선택한다.
2. **검증:** 이 후보 본문이 (A) 너무 짧게 잘렸거나(NotEnoughText), (B) 기존에 알려진 '안전한' CSS 선택자(예: `#article-body`)를 포함하지 않으면(NotContainsCSS) 이 결과를 **폐기**한다.
3. **대체(Fallback):** 폐기된 경우, `trafilatura`가 사용하는 CSS 목록을 포함한 '안전한 CSS 필터링' 로직을 차선책으로 사용한다.

즉, **'엔트로피 모델'**을 먼저 시도하되, 실패하면 안정적인 **'CSS 모델'**로 전환하는 2중 안전장치를 둔 것이다.

### 3. 속도 개선 (Memorization)

1초 안에 이 모든 계산을 끝내기 위해 탐색 방식을 최적화했다.

- **기존 방식:** `rankScore`로 '베스트 노드'를 찾은 *뒤에*, 다시 DOM을 순회하며 해당 노드의 텍스트를 조합했다. (느림)
- **개선 방식:** DOM 트리를 **단 한 번만 깊이우선탐색(DFS)**한다.
    1. 탐색하면서 모든 노드의 텍스트를 순서대로 하나의 거대한 문자열(`textBuilder`)에 저장한다.
    2. 동시에 각 노드의 `rankScore`와, `textBuilder`에서의 `startIndex`, `endIndex`를 모두 계산해 저장한다.
    3. 탐색이 끝나면, `rankScore`가 가장 좋은 노드를 찾는다.
    4. DOM을 다시 볼 필요 없이, `textBuilder.substring(startIndex, endIndex)`로 **즉시 텍스트를 추출**한다. (약 30배 속도 향상)

---

### 6. 결론: Recall 100%의 가치

이렇게 완성된 PAAS 추출기는 벤치마크인 `trafilatura`와 비교 테스트를 진행했다.

- **`trafilatura`:** 노이즈 제거율은 55%로 높지만(높은 Precision), 약 20~30% 확률로 **본문을 유실**했다(낮은 Recall).
- **PAAS 추출기:** 노이즈 제거율은 10%로 낮지만(낮은 Precision), 실험 데이터셋에서 **본문 유실이 0%**였다(Recall 100%).

LLM Search의 목적은 **"정확한 답변"**이다. LLM에게 노이즈(광고)가 좀 섞인 참고 자료를 주는 것은 괜찮지만, 본문의 핵심 내용이 잘린 자료를 주는 것은 치명적이다.

따라서 Precision(노이즈 제거)을 일부 희생하더라도 **Recall(본문 보존)을 100%로 확보**한 이 추출기 방식이 실제 서비스에 적용되었다.
