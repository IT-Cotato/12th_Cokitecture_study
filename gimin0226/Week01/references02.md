# NoSQL을 사용하는 일반적인 이유

## 1. 규모

- NoSQL은 빅데이터, 대규모 사용자, 대규모 컴퓨터, 대규모 공급망, 대규모 과학 등을 지원하는 새로운 데이터 스택의 핵심 요소로 여겨진다.
- 무언가가 너무 방대해져 대규모로 분산되어야 할 때 NoSQL 활용 가능
    - 모든 NoSQL 시스템이 대규모 데이터를 대상으로 하는 것은 아니다.
- 규모는 단순히 많은 디스크 공간은 사용하는 것뿐만 아니라 다양한 차원에 걸쳐 나타날 수 있다.

## 2. 엄청난 쓰기 성능

- 대규모 서비스에서는 쓰기 작업의 양이 매우 많다.
    - 페이스북은 매달 1350억 개의 메시지를 저장해야 한다.
    - 트위터는 하루에 7TB의 데이터를 저장해야 하는 문제를 안고 있으며, 이 요구량은 매년 여러 번 두 배로 증가할 것으로 예상된다.
- 이는 데이터가 너무 커서 단일 노드에 저장할 수 없다는 문제이다.
- 대용량의 데이터에 대한 쓰기 작업은 오래 걸리므로 쓰기 작업을 클러스터에 분산해야 한다.
- 이는 키-값 접근, 맵리듀스, 복제, 내결함성, 일관성 문제 등을 의미한다.
    
    # 1) Key-Value 접근 (키로 바로 찾기)
    
    **개념**
    
    키 하나로 바로 값을 읽고/쓰는 단순 인터페이스. 해시 테이블에 가까워서 지연이 매우 낮고 스케일아웃이 쉽습니다.
    
    **대표 NoSQL과 특징**
    
    - **Redis**: 인메모리 기반. 문자열·해시·리스트·셋·Sorted Set 등 다양한 구조를 “키” 밑에 저장. 초저지연. 영속성(AOF/RDB) 선택 가능.
    - **Amazon DynamoDB**: 매니지드 키-값/문서 DB. 파티션 키(+정렬 키) 설계가 성능의 핵심. 자동 파티셔닝·자동 확장.
    - **Riak KV**: Dynamo 계열 키-값 저장소. R/W/노드 수를 조절하는 튜너블 일관성 모델로 유명.
    - **Cassandra**: 엄밀히는 와이드-컬럼이지만, “파티션 키 + 클러스터링 키”로 키-값처럼 빠르게 접근하는 패턴이 흔함.
    - **Couchbase**: 키-값 API 제공 + 문서(JSON) 저장, N1QL로 질의.
    
    언제 쓰나 → 세션/토큰/카운터/순위표/짧은 TTL 캐시 등 **초저지연 랜덤 액세스**가 중요한 곳.
    
    ---
    
    # 2) MapReduce (대량 분산 처리 패턴)
    
    **개념**
    
    Map 단계에서 레코드를 키/값으로 매핑해 방대한 데이터를 병렬로 쪼개 처리하고, Reduce 단계에서 집계/결합. 요즘은 Spark/Flink 같은 프레임워크가 더 흔하지만, 일부 NoSQL은 자체 MapReduce/대체 기능을 제공합니다.
    
    **대표 NoSQL과 특징**
    
    - **CouchDB / Couchbase**: 디자인 문서에 JS 기반 **Map/Reduce 뷰**. 인덱스처럼 쓰이며 집계/조회에 활용.
    - **MongoDB**: 예전 **mapReduce**가 있었지만 지금은 **Aggregation Pipeline** 권장(성능·표현력 우수).
    - **Riak**: 자바스크립트/에रल랑 MapReduce 지원(역사적으로 유명).
    - **HBase**: 자체 MR이라기보다 **Hadoop MapReduce/Spark**와 결합해 대량 배치 처리.
    - **Elasticsearch**: MR은 아니지만 **Aggregations**로 분산 집계(히스토그램/버킷/파이프라인 집계 등) 수행.
    
    요약 → “원조” MR은 쇠퇴, **Aggregation 파이프라인/분산 집계/스트리밍**으로 대체되는 추세.
    
    ---
    
    # 3) Replication (복제)
    
    **개념**
    
    데이터를 여러 노드/가용영역/리전에 **다중 사본**으로 유지. 가용성·읽기 확장·장애 대비가 목적. “어떻게 복제하고, 충돌을 어떻게 해소하느냐”가 포인트.
    
    **대표 NoSQL과 복제 모델**
    
    - **Cassandra**: **리더리스** + **Replication Factor(RF)** 설정. **Gossip**, **Hinted Handoff**, **Read Repair**, **Anti-Entropy Repair**(Merkle Tree)로 사본 동기화. 멀티-DC 복제 일반적.
    - **MongoDB**: **Replica Set(리더 기반)**. 자동 선출(election)로 장애 조치. **Write Concern/Read Concern**으로 내구성·일관성 제어.
    - **DynamoDB**: 리전 내 **다중 AZ** 복제 기본. **Global Tables**로 멀티-리전(비동기 복제).
    - **CouchDB**: **멀티-마스터** 레플리케이션. 충돌(MVCC)을 문서 히스토리로 보존하고 앱/툴로 해소.
    - **Couchbase**: **XDCR**로 클러스터 간 복제.
    - **Elasticsearch**: **Primary/Replica Shards**. 샤드 재배치·자동 복구.
    - **Redis**: **Primary-Replica** 복제 + **Sentinel**(자동 장애조치) + **Cluster**(샤딩+복제).
    
    ---
    
    # 4) Fault Tolerance (내결함성)
    
    **개념**
    
    일부 노드나 네트워크 장애가 나도 **서비스가 계속 동작**하도록 설계. 복제·샤딩·오토힐링·재선출·재배치 등이 핵심 기법.
    
    **대표 NoSQL의 내결함성 메커니즘**
    
    - **Cassandra**: 리더리스 + 다중 사본 + **튜너블 합의(R/W 쿼럼)** → 노드 장애에도 읽기/쓰기 지속. 정기 **Repair** 중요.
    - **MongoDB**: 리더 장애 시 **자동 선출**로 리더 교체. 저널링·oplog 재적용.
    - **Elasticsearch**: 노드 다운 시 샤드를 다른 노드로 **자동 재할당**.
    - **Redis**: **Sentinel**로 자동 failover, **Cluster**로 슬롯 재배치.
    - **DynamoDB**: 매니지드로 내부적으로 다중 AZ/오토힐링.
    - **Riak**: 가상 노드(vnode)·일관성 해시로 균형 분산, 노드 장애 시에도 계속 동작.
    
    ---
    
    # 5) Consistency (일관성)
    
    **개념**
    
    복제본들이 서로 **얼마나 빨리/어떻게 일치**하느냐의 규칙. CAP 제약 하에서 **Strong / Eventual / Causal / Read-Your-Writes / Monotonic** 등 다양한 모델이 있습니다. 일부 시스템은 **튜너블 일관성**을 제공해 요청별로 강도를 조정할 수 있게 합니다.
    
    **대표 NoSQL의 일관성 모델**
    
    - **Cassandra / Riak**: **튜너블 일관성**. 예: `QUORUM`, `LOCAL_QUORUM`, `ALL` 등으로 요청별 강도 선택. 기본은 결국 일관(Eventual).
    - **MongoDB**: 프라이머리 읽기는 **강한 일관성(단일 문서 수준)**. 세컨더리 읽기는 지연/과거값 가능. `readConcern`(local/majority/linearizable)과 `writeConcern`으로 제어. 트랜잭션(멀티문서)도 지원.
    - **DynamoDB**: 기본 **Eventual Read**, 옵션으로 **Strongly Consistent Read**(동일 리전) 제공. 글로벌 테이블은 결국 일관.
    - **CouchDB**: **멀티-마스터 + MVCC** → 기본은 결국 일관, 충돌은 애플리케이션 레벨/툴로 해소.
    - **Elasticsearch/Redis**: 쓰기 후 즉시 모든 복제본이 동일하다고 보장되진 않음(결국 일관에 가깝고, 설정/모드에 따라 다름).
    - **HBase**: 단일 클러스터 내에서 **Row 단위 강한 일관성**(RegionServer 리더 중심). 멀티-클러스터 복제는 비동기.
- 더 빠른 쓰기를 위해서는 인메모리 시스템을 사용할 수 있다.
    
    # 6) “더 빠른 쓰기”를 위한 인메모리 시스템
    
    **개념**
    
    디스크보다 훨씬 빠른 **RAM**에 쓰고(또는 먼저 쓰고), 필요하면 백그라운드로 영속화해 **지연을 극도로 낮추는** 전략.
    
    **주요 선택지와 패턴**
    
    - **Redis**: 사실상의 표준 인메모리 저장소. **AOF**(Append-Only File), **RDB 스냅샷**으로 내구성 선택. Pub/Sub, Streams, Lua 스크립트 등 고성능 패턴 다수.
    - **Memcached**: 초간단 인메모리 캐시. 영속성 없음(순수 캐시).
    - **Aerospike**: 인메모리 인덱스 + 메모리/SSD 하이브리드 저장. 매우 빠른 쓰기·복제, **강한 일관성 모드** 선택 가능.
    - **Apache Ignite / Hazelcast**: 인메모리 데이터 그리드(IMDG). 키-값 + 컴퓨팅(분산 실행)까지.
    - **“Write-through / Write-behind” 캐싱**: 캐시에 먼저 쓰고 원본 저장소(DB)에 동기/비동기로 반영. 쓰기 지연·스파이크 흡수에 유리.
    
    현실 팁 → 인메모리만 믿지 말고, **복제 + 영속화 전략(AOF, 스냅샷, 동기/비동기 복제)**을 비즈니스 RPO/RTO에 맞춰 조합하세요.
    

## 3. 빠른 키-값 접근

- 지연 시간이 중요한 경우, 키를 해싱하고 메모리에서 직접 또는 단 한 번의 디스크 탐색으로 값을 읽는 것보다 더 나은 방법은 없다.

## 4. 유연한 스키마 와 유연한 데이터 유형

- 다양한 새로운 데이터 유형을 지원한다.
- 열 지향, 그래프, 고급 데이터 구조, 문서 지향, 키-값 데이터 구조를 제공한다.
    
    ## 열 지향(Column-oriented)
    
    - **개념**: 행(row) 단위가 아니라 **열(column) 단위**로 데이터를 저장. 분석/집계에 강함.
    - **예시**
        - **Cassandra / HBase**: Wide-column 저장소. 테이블처럼 보이지만, 각 행마다 열이 동적으로 달라질 수 있고, 대규모 분산에 강함.
        - **ClickHouse**(OLAP): 컬럼 기반 저장 + 압축 + 고속 분석.
    
    ---
    
    ## 그래프(Graph)
    
    - **개념**: 노드(객체)와 엣지(관계)를 저장/탐색. `친구 관계`, `추천 그래프`, `네트워크 경로 탐색`에 최적.
    - **예시**
        - **Neo4j**: 가장 유명한 그래프 DB. Cypher 쿼리 언어.
        - **Amazon Neptune**: AWS 관리형 그래프 DB. Gremlin·SPARQL 지원.
        - **ArangoDB**: 멀티 모델(문서 + 키-값 + 그래프).
    
    ---
    
    ## 고급 데이터 구조(Advanced Data Structures)
    
    - **개념**: 단순 문자열/숫자뿐 아니라 리스트, 집합, 정렬집합, 스트림 같은 구조를 DB 자체가 제공.
    - **예시**
        - **Redis/Valkey**: String, Hash, List, Set, Sorted Set, HyperLogLog, Bitmap, Stream 등 다양한 자료구조 지원.
        - **Aerospike**: List, Map, GeoJSON 지원.
        - 이런 구조 덕분에 어플리케이션 레벨에서 따로 매핑/직렬화 작업을 덜 해도 됨.
    
    ---
    
    ## 문서 지향(Document-oriented)
    
    - **개념**: JSON/BSON/XML 같은 **반정형 문서**를 그대로 저장. 중첩 구조, 배열을 지원해 복잡한 객체 표현에 강함.
    - **예시**
        - **MongoDB**: BSON 문서 저장, 중첩 쿼리·Aggregation Pipeline 제공.
        - **CouchDB**: JSON 문서 저장, MVCC 기반 복제.
        - **Couchbase**: 키-값 + 문서 + N1QL(SQL 유사 쿼리).
    
    ---
    
    ## 키-값(Key-Value)
    
    - **개념**: 단순한 `key → value` 매핑. 초고속 접근 가능.
    - **예시**
        - **Redis/Valkey, Memcached**: 인메모리 KV 스토어.
        - **Riak KV**: 튜너블 일관성을 가진 분산 KV.
        - **Amazon DynamoDB**: 확장성 좋은 관리형 KV/문서 DB.
- 복잡한 객체는 많은 매핑 없이도 쉽게 저장할 수 있다.
    - RDBMS에서는 객체 → 테이블(정규화) → JOIN으로 분해 저장. → ORM/DAO 계층 필요.
    - 문서 DB에서는 애플리케이션 객체(JSON)를 **그대로 저장**. 즉시 읽기/쓰기 가능, 매핑 코드 최소화.
    - 그래프 DB에서는 객체 관계를 **엣지**로 직접 모델링. 별도 매핑 불필요.
    - Redis 같은 경우도 Hash/Set/List 자료구조를 그대로 쓰면 객체 매핑 로직이 단순화.
- 개발자는 복잡한 스키마와 ORM 프레임워크를 피하는 것을 선호한다.
    - 구조가 없기 때문에 훨씬 더 유연하게 사용할 수 있다.

## 5. 스키마 마이그레이션

- 데이터베이스 구조(컬렉션, 테이블, 인덱스, 필드 정의)가 바뀔 때, 기존 데이터를 새 구조에 맞게 변환/이동하는 것
    
    ### 스키마리스 vs 스키마풀
    
    - 완전 스키마리스: 문서마다 필드가 달라도 DB가 불만 없지만, 코드가 필드를 가정하므로 사실상 스키마 존재
    - 스키마풀(강타입): 열/컬럼 정의가 DB 자체에 강제됨 → 변경 시 반드시 마이그레이션 필요
- 스키마리스를 사용하면 스키마 마이그레이션을 크게 걱정하지 않고 더 쉽게 처리
- 스키마는 런타임에 애플리케이션에 의해 적용되므로 동적이라고 할 수 있다.

## 6. 분산 시스템 지원

- 장애 밸생 시에도 문제없이 데이터센터를 확장 할 수 있는 분산 시스템이 필요
- NoSQL 시스템은 확장성에 중점을 두었기 때문에 파티션을 활용하는 경향이 있음
- 엄격한 일관성 프로토콜을 사용하지 않는 경향이 있어 분산 환경에서 운영하기에 유리

## 7. 쓰기 가용성

- 전통 RDBMS는 보통 **강한 일관성**을 우선하기 때문에, 네트워크 파티션이나 장애가 있으면 쓰기 실패가 발생
- 하지만 NoSQL은 **가용성**을 더 우선시하는 경우가 많다
    - 일부 노드가 죽더라도 쓰기를 받아주고, 나중에 최종적으로 동기화 한다.
- **예시**: Cassandra, DynamoDB는 “쓰기를 우선”하는 구조로, 파티션 장애 중에도 데이터를 받아들일 수 있다.

## 8. 단일 장애 지점 없음(No SPOF)

- **고가용성(HA)**을 지향 → 노드가 죽어도 서비스 전체는 계속 동작.
- **방법**: 자동 로드 밸런싱, 복제(replication), 클러스터 크기 자동 조정.
- **예시**: Cassandra는 “masterless 구조”라 어느 노드가 죽어도 나머지가 처리. Redis Sentinel/Cluster도 장애조치를 자동화.

## 9. 문제별 최적 모델 선택

- 모든 문제를 “관계형 모델” 하나로 해결하려던 과거 → **이제는 문제에 맞는 DB를 고르자**는 철학.
- 예시:
    - **그래프 문제** → Neo4j, Amazon Neptune
    - **키-값 초저지연 캐시** → Redis, Memcached
    - **문서 저장/검색** → MongoDB, Couchbase, Elasticsearch
    - **분산 시계열** → InfluxDB, TimescaleDB
- 즉, “한 가지 DB로 다 해결하려고 하지 말고, **적재적소에 최적 DB 모델**을 써라”는 메시지.

## 10) CAP 이론과 트레이드오프 조정 가능

- **CAP 정리**:
    - Consistency(일관성)
    - Availability(가용성)
    - Partition tolerance(분할 허용성)
        
        → 이 세 가지를 동시에 완벽히 만족할 수는 없음.
        
- **RDBMS**는 보통 C + P를 택해 A를 희생(네트워크 장애 시 가용성 저하).
- **NoSQL**은 **“조정 가능한 슬라이더”**를 제공 → 상황별로 일관성과 가용성의 균형을 맞출 수 있음.
- **예시**:
    - Cassandra: `ConsistencyLevel`(ONE, QUORUM, ALL 등) 지정 가능.
    - DynamoDB: Strong read / Eventually consistent read 선택 가능.
    - Riak: R/W/N 튜너블 파라미터.

즉, “내 서비스에서 다운타임이 더 치명적인가, 아니면 데이터 불일치가 더 치명적인가?”에 따라 **비즈니스적 의사결정**을 할 수 있게 된 것.

# 더욱 구체적인 사용 사례

### 1) 비(非)트랜잭션 대량 스트림(로그/클릭스트림)

- **상황**: 초당 수만~수백만 건이 연속으로 들어오는 Append-only 데이터.
- **왜 NoSQL**: 스키마 유연 + 수평 확장 + 높은 쓰기 처리량.
- **기본 설계**: 수집(Kafka) → 저장(Cassandra/HBase/ClickHouse/Elasticsearch) → TTL·시간 파티션.
- **주의**: 핫파티션 방지(키에 시간 버킷 포함), 백프레셔, 보존기간/스토리지 비용 관리.

### 2) 온라인↔오프라인 동기화(CouchDB 니치)

- **상황**: 네트워크 불안정/오프라인에서도 쓰고 나중에 서버와 자동 동기.
- **왜 NoSQL**: 멀티-마스터 복제·충돌해결(MVCC).
- **기본 설계**: PouchDB/모바일 ↔ CouchDB/서버, 문서 버전/리비전 관리.
- **주의**: 충돌 정책 명확화, 동시 편집 UX, 대량 동기 시 리밸런싱.

### 3) 모든 부하에서 낮은 지연(일관된 응답시간)

- **상황**: P99, P99.9도 엄격(결제/매칭/광고 입찰).
- **왜 NoSQL**: 인메모리/샤딩/복제 기반 초저지연.
- **기본 설계**: 앞단 캐시(Redis) + 백엔드 KV(DynamoDB/Aerospike), 읽기경로 최소화.
- **주의**: 스탬피드 방지, 핫키 분산, 멀티AZ, 장애모드(디그레이드) 정의.

### 4) 무거운 조인 회피

- **상황**: RDB 조인으로 CPU/IO 폭발, 지연 급증.
- **왜 NoSQL**: 문서 단위 **디노말라이즈**로 1-hop 조회.
- **기본 설계**: MongoDB/Couchbase에 ‘조회 단위’로 문서 설계(+검색은 ES).
- **주의**: 쓰기 증폭/중복 업데이트 보상 흐름(사가/이벤트).

### 5) 소프트 실시간(게임 등)

- **상황**: ms 단위 반응 필요(매칭/순위/세션).
- **왜 NoSQL**: 인메모리 카운터·큐·세션에 적합.
- **기본 설계**: Redis(세션·SortedSet 랭킹·Pub/Sub), 이벤트 스트림(Kafka).
- **주의**: 지역성(리전별 분리), 레이트리밋, 치터/치트 내성.

### 6) 다양한 R/W 패턴(50/50, 95/5 등)

- **상황**: 읽기/쓰기 편향이 뚜렷.
- **왜 NoSQL**: 워크로드 특화(쓰기-헤비: Cassandra, 읽기-헤비: ES/Pinot).
- **기본 설계**: 파티션키 설계, 서빙·인덱스 테이블 분리.
- **주의**: 보조 인덱스 남발 금지, 핫파티션 모니터링.

### 7) 부하 분산으로 CPU 가동률 유지

- **상황**: 워커 풀로 균등 처리 필요(ETL, 썸네일, 메일 발송).
- **왜 NoSQL**: 파티션/큐/락으로 병렬 처리 안정화.
- **기본 설계**: Kafka/SQS 큐 + 상태 저장(Redis/Cassandra).
- **주의**: 중복 처리/멱등성 키, 재시도/역압(backpressure).

### 8) 실시간 삽입·업데이트·쿼리 동시 요구

- **상황**: ingest와 조회가 모두 즉시성.
- **왜 NoSQL**: LSM-tree/컬럼/인메모리로 동시 성능 확보.
- **기본 설계**: 핫셋 캐시(Redis) + 원천(Cassandra/Bigtable), OLAP은 ClickHouse/Pinot.
- **주의**: 쓰기경로와 조회경로 분리, 인덱스 최소화.

### 9) 계층적 데이터(스레드, 부품 BOM)

- **상황**: 트리/그래프 탐색.
- **왜 NoSQL**: 그래프/문서가 자연스러움.
- **기본 설계**: Neo4j(깊은 탐색) 또는 MongoDB(머티리얼라이즈드 패스).
- **주의**: 깊은 트리는 그래프 DB, 문서형은 깊이 제한·경로 캐싱.

### 10) 동적 테이블/컬렉션 증가

- **상황**: 테넌트/프로젝트 단위로 스키마가 수시로 생김.
- **왜 NoSQL**: 스키마 유연·네임스페이스 격리.
- **기본 설계**: “단일 컬렉션 + tenant_id 파티션” 우선, 정말 필요할 때만 분리.
- **주의**: 테이블 폭발 방지, 인덱스 관리 비용.

### 11) 2계층 앱(초저지연 서빙 + 오프라인 계산)

- **상황**: 결과는 즉시 제공, 계산은 배치/스트리밍.
- **왜 NoSQL**: 사전 계산 결과를 빠르게 서빙.
- **기본 설계**: Spark/Flink → 머티리얼라이즈드 뷰(Redis/Mongo/Pinot)로 업서트.
- **주의**: 지연/신선도 SLO 정의, 재계산·백필 전략.

### 12) 순차적 데이터 읽기(대범위 스캔)

- **상황**: 범위 조회/집계가 많음.
- **왜 NoSQL**: 컬럼식 저장·타임 파티션 유리.
- **기본 설계**: ClickHouse/Druid/Pinot, HBase/Bigtable(시간 키).
- **주의**: 파일/세그먼트 크기, 압축·인덱싱 튜닝.

### 13) 고성능이 필요한 부분 분리(예: 로그인)

- **상황**: 특정 경로만 초저지연/고QPS.
- **왜 NoSQL**: 세션·토큰·레이트리밋 캐시화.
- **기본 설계**: Redis + 백엔드 사용자 메타(DynamoDB).
- **주의**: 독립 배포/격리, 장애시 디그레이드 플랜.

### 14) 캐싱 계층

- **상황**: 읽기 과부하·원천 DB 보호.
- **왜 NoSQL**: 인메모리·분산·TTL.
- **기본 설계**: Redis/Valkey, 키 설계·TTL·지터, negative cache.
- **주의**: 스탬피드·일관성 정책, 캐시 미스 폭주.

### 15) 투표/좋아요 카운트

- **상황**: 원자적 증가·중복 방지 필요.
- **왜 NoSQL**: 원자 연산·조건부 쓰기.
- **기본 설계**: Redis INCR + Set(사용자 중복 방지) 또는 DynamoDB 조건부 업데이트.
- **주의**: 멱등 키, 스팸/봇 방지.

### 16) 실시간 페이지뷰 카운터

- **상황**: 초단위/분단위 카운팅.
- **왜 NoSQL**: INCR·롤업 쉬움.
- **기본 설계**: Redis 버킷(분/시) + 배치 롤업(ClickHouse).
- **주의**: 메모리 사용량, 롤업/보존 정책.

### 17) 사용자 등록·프로필·세션

- **상황**: CRUD + 세션 TTL.
- **왜 NoSQL**: 문서형(프로필) + 인메모리(세션).
- **기본 설계**: MongoDB(프로필) + Redis(세션), 유니크 제약은 애플리케이션/인덱스로.
- **주의**: PII 암호화, GDPR, 인덱스·샤딩.

### 18) 문서/카탈로그/CMS

- **상황**: 복잡 문서를 “한 덩어리”로 관리·검색.
- **왜 NoSQL**: 문서형과 검색의 결합.
- **기본 설계**: MongoDB/Couchbase + Elasticsearch 동기화.
- **주의**: 버저닝, 퍼블리시 워크플로, 역색인 일관성.

### 19) 아카이빙(온라인 접근 가능한 장기 보관)

- **상황**: 대용량 로그/이벤트를 오래 보관·조회.
- **왜 NoSQL**: 유연 스키마·티어링.
- **기본 설계**: S3(콜드) + ES/ClickHouse(핫/워ーム), 수명주기 정책.
- **주의**: 비용/성능 균형, 인덱스 축소.

### 20) 분석(MapReduce/버치 OLAP)

- **상황**: 대량 데이터 배치 분석.
- **왜 NoSQL**: 분산 저장 + 분산 집계.
- **기본 설계**: Data Lake + Spark/Hive → 서빙용 Druid/Pinot/ClickHouse.
- **주의**: 스키마 진화, 카디널리티, 파티션 프루닝.

### 21) 다양한 데이터 타입(메타+대용량 바이너리)

- **상황**: 이미지/영상/오디오 + 메타데이터.
- **왜 NoSQL**: 메타는 문서/KV, 본문은 오브젝트 스토리지.
- **기본 설계**: DynamoDB/MongoDB(메타) + S3(GCS) Blob, CDN.
- **주의**: 참조 무결성, 서명 URL, 수명주기.

### 22) 임베디드 시스템(경량 저장)

- **상황**: 디바이스/엣지에서 서버 없이 저장.
- **왜 NoSQL**: 경량 K/V, 낮은 오버헤드.
- **기본 설계**: RocksDB/LevelDB/LMDB(로컬), 필요 시 동기화 채널.
- **주의**: 전원 장애 내구성, 플래시 수명.

### 23) “마켓 게임” 소유자 기준 파티션

- **상황**: 소유자별 목록을 한 번에 빠르게.
- **왜 NoSQL**: 파티션 키=소유자 → 단일 파티션 스캔.
- **기본 설계**: DynamoDB(PK=ownerId, SK=timestamp/score) 또는 Cassandra(파티션=owner).
- **주의**: 소유권 이전 트랜잭션은 사가/이벤트로 조율.

### 24) 메타는 KV/문서, 본문은 Blob(S3) 패턴

- **상황**: 대용량 본문은 외부 저장, 메타로 인덱싱.
- **왜 NoSQL**: 단순/유연한 메타 조회.
- **기본 설계**: KV/문서에 S3 키 보관, 비동기 처리 파이프라인.
- **주의**: 일관된 쓰기 순서, 실패시 보상 처리.

### 25) 대규모 이벤트 상관(감시/모니터링/조사)

- **상황**: 다양한 소스 이벤트를 연결해 패턴 탐색.
- **왜 NoSQL**: 스트리밍 + 그래프/검색의 결합.
- **기본 설계**: Kafka/Flink(전처리) + Elasticsearch(검색) + 그래프DB(관계).
- **주의**: 개인정보·규제 준수, 접근 통제·감사 로그.

### 26) 실시간 사기 탐지

- **상황**: 거래를 즉시 스코어링/차단.
- **왜 NoSQL**: 피처 캐시·세션 상태·CEP.
- **기본 설계**: Redis(피처/레이트리밋) + Flink CEP + 영속 스토어.
- **주의**: 지연·정확도 트레이드오프, 온라인 학습 피드백 루프.

### 27) 환자 병력 통합(진단 보조)

- **상황**: 이질 소스 기록을 한 환자 타임라인/관계로 통합.
- **왜 NoSQL**: 문서/그래프 유연 모델.
- **기본 설계**: FHIR 문서(MongoDB) + 관계(Neo4j) + 검색(ES).
- **주의**: PHI 보호/암호화/감사, 지연된 동기화의 임상적 영향.

### 28) “마지막 활동” 고빈도 업데이트

- **상황**: 초단위/수초 단위로 상태 갱신(채팅 ‘활동 중’).
- **왜 NoSQL**: 인메모리 TTL/원자 연산.
- **기본 설계**: Redis `SETEX`/SortedSet(lastSeen), 만료 기반 청소.
- **주의**: 쓰기 폭주 대비 샤딩, 네트워크 스파이크.

### 29) 고주파 스트림 + 저주파 다중 파티션 질의(머티리얼라이즈드 뷰)

- **상황**: 지속 스트림 처리 + 가끔 무거운 분석 조회.
- **왜 NoSQL**: 사전 집계 결과를 서빙 테이블로.
- **기본 설계**: Flink/Spark → Pinot/Druid/ClickHouse/Cassandra 뷰 적재.
- **주의**: 뷰 신선도/정합 검증, 백필/리빌드 계획.

### 30) 우선 대기열

- **상황**: 작업 중요도·마감에 따른 스케줄링.
- **왜 NoSQL**: 정렬 가능한 구조.
- **기본 설계**: Redis SortedSet(스코어=우선순위/만료), 소비자 워커 풀.
- **주의**: 재큐잉, 중복 방지, 가시성 타임아웃.

### 31) ORM 우회·캐시 위 계산

- **상황**: 캐시 데이터에 서버 사이드 계산(왕복 최소화).
- **왜 NoSQL**: 내장 스크립트/파이프라인.
- **기본 설계**: Redis Lua/Functions, Mongo Aggregation.
- **주의**: 짧은 실행시간, 타임아웃/격리.

### 32) 단순 KV로 대규모 유니크 보장

- **상황**: 방대한 키 공간에서 유일성 보장.
- **왜 NoSQL**: 조건부 put/원자 연산.
- **기본 설계**: DynamoDB 조건부 쓰기, Redis `SET NX`, Mongo unique 인덱스.
- **주의**: 멱등 키, 재시도·충돌 처리.

### 33) 시간 롤업으로 빠른 질의 유지

- **상황**: 초→분→시→일 사전 집계로 조회 가속.
- **왜 NoSQL**: 프리컴퓨트/물리화 뷰.
- **기본 설계**: ClickHouse Materialized View, Redis/카운터 버킷, Cassandra 타임버킷.
- **주의**: 지연 허용치, 백필·재계산.

### 34) 거대한 집합의 교집합(조인 대체)

- **상황**: 전통 조인이 너무 느림(초대형 세트 연산).
- **왜 NoSQL**: 비트맵/집합 연산이 고속.
- **기본 설계**: Redis SINTER/BitOp, Roaring Bitmap, 역색인(ES).
- **주의**: 정확/근사 선택(Bloom), 메모리 소비.

### 35) 트위터형 타임라인

- **상황**: 팔로우 관계로 개인 피드 생성·조회.
- **왜 NoSQL**: 대규모 분산·fan-out 패턴 적합.
- **기본 설계**: Cassandra(타임라인 테이블), 핫피드는 Redis, 검색/추천은 ES/벡터DB.
- **주의**: 거대 계정 처리(fan-out on read), 정합성 vs 지연 트레이드오프.
