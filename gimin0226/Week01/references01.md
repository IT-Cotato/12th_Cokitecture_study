# Active-Active

- 두 리전에 같은 서비스를 모두 띄워두고, 사용자는 가까운 리전으로 들어와 동시에 트래픽을 처리하는 방식

### 요구사항

- 서비스는 상태 비저장
    - 모든 데이터/상태 복제는 데이터 계층에서 처리한다
- 로컬 리전 내 모든 리소스에 액세스해야 한다
    - 사용자의 요청을 처리하는 동안에는 같은 리전 안의 자원(S3, SQS, DB 등)만 사용하라는 의미
    - 이유: 지역 간 호출은 지연이 커지고, 네트워크 이슈가 곧 서비스 장애로 번지기 쉽기 때문
- 사용자 통화 경로에는 지역 간 통화가 없어야 한다
    - 사용자 요청을 처리하는 동기 경로에서는 절대 다른 리전을 호출하지 말라는 의미
    - 지역 간 데이터 동기화가 필요하면 요청이 끝난 뒤, 백그라운드(비동기)에서 하라는 의미

정상적인 운영 상태에서는 사용자가 가장 가까운 AWS 리전으로 지리적 DNS 라우팅되며 ,대략 50/50% 비율로 라우팅된다.

리전 전체에 걸쳐 심각한 장애가 발생 → 지리적 DNS 재정의 → 모든 사용자 트래픽을 정상 리전으로 라우팅

# DNS - Denominator를 사용하여 사용자 트래픽 제어

---

## 개념 설명

DNS (Domain Name System)

- 사용자가 어디로 가야 할지 알려주는 내비게이션
- 웹사이트 주소를 물어보면, 최적의 IP 주소를 알려주는 역할

UltraDNS 

- 정교한 방향성 라우팅을 해줌 ( 지리·네트워크 기반의 세밀한 매핑)
- 사용자의 지리적 위치에 따라 접속할 서버 그룹을 안정적으로 나누기 위해 사용

Route53 

- UltraDNS가 골라준 “지역별 엔드포인트”를 더 유연하게 전환하는 컨트롤 레이어로 씀
- 실제 트래픽을 받는 서버를 변경해야 할 때, API를 통해 매우 빠르고 안정적으로 전환하기 위해 사용

Denominator

- UltraDNS와 Route53이라는 서로 다른 DNS 서비스를 한 곳에서 편리하게 관리할 수 있게 해줌

---

## UltraDNS와 Route53을 같이 쓰는 이유

### **1. 예측 불가능한 트래픽 이동 효과를 유발할 수 있는 ‘지연 시간 기반 라우팅’ 방식을 사용하고 싶지 않았음**

- **지연 시간 기반 라우팅**:
    - 사용자에게 가장 응답 속도가 빠른 서버로 자동으로 연결해주는 기능
- 이 방식은 인터넷 상태에 따라 최적의 경로가 수시로 바뀜
    - 서부 사용자가 갑자기 동부 서버로 연결되었다가 다시 서부 서버로 바뀌는 등 트래픽이 예기치 않게 왔다 갔다 하는 현상이 발생할 수 있음
    - 이런 불안정성을 피하고 싶었음

### **2. UltraDNS는  ‘지향성 라우팅’ 기능을 제공 → 하지만 Route53은 제공 x**

- **지향성 라우팅**:
    - 사용자의 지리적 위치를 기준으로 접속할 서버를 정해주는 기능
    - 북미의 여러 다른 지역에서 접속하는 고객들을 각기 다른 지역의 엔드포인트(서버)로 보낼 수 있음
- 예를 들어, 미국 서부(캘리포니아)에서 접속한 사용자는 물리적으로 가까운 서부 데이터센터 서버로 보내고, 동부(뉴욕)에서 접속한 사용자는 동부 데이터센터 서버로 보내는 방식
- ‘지향성 라우팅’ 기능이 아마존의 Route53에는 없기 때문에, 이 기능을 사용하기 위해 UltraDNS를 써야함
    - 추가: (현재는 Route53에 지향성 라우팅 기능 존재)

### **3. UltraDNS와 ELB(로드밸런서) 사이에 Route53 계층을 둠으로써, 사용자 트래픽을 전환할 추가적인 수단을 갖게 됨**

- Rout53의 API가 매우 빠르고 안정적이기 때문
- 예를 들어, 서버에 문제가 생겨 트래픽을 다른 곳으로 급하게 돌려야 할 때, 다른 DNS 서비스(UltraDNS)는 변경 사항이 적용되기까지 시간이 오래 걸릴 수 있지만, Route53은 거의 즉각적으로 변경이 가능하다.
- 전체적인 트래픽 흐름: `사용자 -> UltraDNS -> Route53 -> ELB(서버 그룹)`

### **4. 별도의 Route53 계층을 사용하여 트래픽을 전환하면, 변경 작업이 간단해진다.**

- 복잡한 UltraDNS의 설정을 직접 건드리는 것보다, 중간에 있는 Route53 설정만 살짝 바꿔주는 것이 훨씬 간편하고 빠르다는 의미
- 즉, UltraDNS의 복잡한 지향성 그룹의 지역 설정을 바꾸는 대신, Route53의 CNAME만 바꾼다는 의미
    - **어려운 방법:**
        
        UltraDNS에서 “이제부터 미국 서부 지역은 B서버 그룹으로 보내라” 와 같이 복잡하고 큰 지리적 규칙을 변경하는 것
        
    - **쉬운 방법:**
        
         UltraDNS에는 “미국 서부 지역은 무조건 Rout53 주소로 보내라” 고 고정해둠
        
        평소에는 Route53에서 A서버 그룹으로 연결해 둠
        
        만약 서버를 바꿔야 한다면, Route53 설정만 바꿔서 B 서버 그룹으로 연결이 바뀌도록 변경
        
        **CNAME**는 이처럼 하나의 도메인 이름을 다른 도메인 이름으로 연결해주는 DNS 설정
        

---

## 총정리

- **UltraDNS 사용 이유:** 사용자의 **지리적 위치**(동부/서부)에 따라 접속할 서버 그룹을 안정적으로 나누기 위해. (Route53에는 이 기능이 없기 때문)
- **Route53 사용 이유:** 실제 트래픽을 받는 서버를 변경해야 할 때, **API를 통해 매우 빠르고 안정적으로 전환**하기 위해.

결론적으로, 큰 단위의 지역별 트래픽 분배는 UltraDNS가 담당하고, 그 안에서 실제 서버로 연결되는 세부적인 트래픽 제어는 Route53이 담당하는 이중 구조를 통해 안정성과 신속성을 모두 확보한 것이다.

# Zuul - 트래픽 세이핑

---

## 트래픽 셰이핑(Traffic SHaping)이란?

- 단순히 트래픽을 전달하는 것을 넘어, 트래픽의 흐름을 원하는 대로 제어하고, 양을 조절하며, 우선순위를 정하는 등 적극적으로 관리하는 기술을 의미한다.
- 넷플릭스의 모든 ‘엣지 서비스’는 Zuul 계층의 맨 앞에 위치한다.
    - **엣지 서비스**: 외부 사용자의 요청이 넷플릭스 내부 시스템으로 들어오는 가장 첫 관문을 의미

---

## Zuul이 하는 역할

- **길안내 (Direct Traffic)**
    - 요청의 종류에 따라 올바른 내부 서비스(서비스 클러스터)로 안내한다.
    - “이 요청은 영화 추천 서비스로”, “저 요청은 사용자 계정 서비스로”
- **실시간 경로 변경 (Change routing at runtime)**
    - 특정 서비스에 문제가 생기면, 운영을 중단하지 않고도 경로를 바꿈
- **과부하 차단 (Shed load)**
    - 특정 서비스에 요청이 너무 많이 몰려서 마비될 것 같으면, 일부 요청을 미리 차단하여 내부 시스템 전체가 다운되는 것을 막는다.

---

## Active - Active를 도입하기 위해 Zuul의 성능 개선

### 1. 잘못 라우팅된 요청을 식별하고 처리하는 능력

- **잘못 라우팅된 요청이란?**
    - 아시아 사용자는 당연히 아시아 데이터센터로 접속해야 하는데, 어떤 이유로 미국 데이터센터로 요청이 잘못 도착한 경우
- **왜 문제인가?**
    - 한 사용자의 요청이 여러 지역을 오가며 서비스가 느려지고 데이터가 꼬일 수 있다.
- **해결책**
    1. **lsthmus 모드**
    - Zuul이 잘못 온 요청을 내부의 빠른 전용선(isthmus, 지협)을 통해 올바른 지역의 Zuul로 몰래 전달해 준다. 사용자는 아무것도 눈치채지 못한다.
    1. **리다이렉트 응답**
    - Zuul이 사용자에게 주소 잘못 찾아왔다고 올바른 지역 주소를 넣은 응답을 보낸다.
    - 사용자의 브라우저나 앱이 그 주소로 다시 요청하게 된다.

### 2. 페일오버 모드로 선언하는 기능

- **페일오버(Failover) 모드란?**
    - 특정 데이터센터 지역에 장애가 발생하거나 불안정한 비상사태를 의미
- 이 모드가 되면, 해당 지역의 Zuul은 다른 지역으로 요청을 전달해 주는 행위(라우팅)을 시도하지 않는다.
    - 자기 리전이 불안정하니 다른 리전 요청은 라우팅하지 않음
    - 자기 리전 요청만 처리하면서, 시스템이 완전히 붕괴하지 않도록 안전성을 우선한다.

### 3. 어느 시점에서든 최대 트래픽 수준을 정의하는 능력

추가 요청이 자동으로 차단되도록(응답=오류) 설정하여 다운스트림 서비스를 폭주하는 요청으로부터 보호할 수 있다.

- **다운스트림 서비스:**
    - 요청이나 데이터의 흐름에서 아래쪽에 있는 서비스
    - 예를 들면, `사용자 요청(업스트림)` → `Zuul (게이트웨이)` → `넷플릭스 내부 서비스들 (다운스트림)`
    - 여기서 다운스트림 서비스는 Zuul 을 통과한 요청을 실제로 처리하는 `넷플릭스의 모든 내부 전문 서비스들` 을 말한다.
    - 로그인 및 인증 서비스, 영화 목록을 보여주는 서비스, 개인별 영화 추천 서비스 등
- **왜 필요한가?**
    1. Thundering Herd (몰려드는 소 떼):
        - 장애 복구 직후처럼 모든 사용자가 한꺼번에 접속을 시도하면, 마치 소 떼가 몰려오듯 엄청난 트래픽이 발생해 서버를 다시 다운시킬 수 있다.
    2. 서버 증설 중(Scaling up): 
        - 트래픽이 늘어 서버를 추가하는 중일 때, 아직 준비가 덜 된 서버들을 보호한다.
    3. 캐시가 비어있을 때(Cold Cache):
        - 서버 재시작 직후에는 캐시가 비어있어 모든 요청을 느린 DB에서 처리해야 한다.
        - 이때 요청이 몰리면 데이터베이스가 마비되므로, 트래픽 양을 조절해줘야 한다.

이러한 모든 기능은 ‘안정적인 상태’와 ‘장애 조치 상황에서 사용자 트래픽을 처리하는 방법을 관리하는 강력하고 유연한 툴셋’을 제공한다.

# 데이터 복제 - Cassandra 및 EvCache

---

## Acitve-Active의 데이터 복제란?

두 개 이상의 데이터센터를 동시에 운영할 때, 한쪽에서 변경된 사용자 데이터를 다른 쪽 데이터센터에서도 빠르고 정확하게 똑같이 복사하여 데이터 일관성을 유지하는 기술을 의미한다.

---

## **1. 아파치 카산드라**

카산드라는 즉 원본 데이터를 저장하는 데이터베이스 역할을 한다.

### **핵심 기능: 다중 데이터센터 복제**

- 카산드라는 데이터베이스 자체에 여러 지역의 데이터센터 간에 데이터를 자동으로 복제하는 기능이 내장되어 있다.
- 마치 한쪽 은행 지점에 입금하면 다른 모든 지점의 계좌 정보가 자동으로 업데이트되는 것과 같다.

### **Active-Active 도입 후의 과제**

- 과거에는 데이터 복제 속도가 조금 느려도 괜찮았지만, Active-Active 환경에서는 사용자가 어느 지역에 접속하든 항상 최신 데이터를 봐야 하므로 **복제 속도가 매우 중요**해졌다.

### **해결책: 카산드라의 속도 검증 테스트**

실제로 빠른지 확인하기 위해 테스트를 진행했다.

- **테스트 내용:** A 지역에 데이터 100만 개를 저장하고, 0.5초 뒤 B 지역에서 해당 데이터가 모두 읽히는지 확인
- **결과:** **성공.** 0.5초 만에 모든 데이터가 완벽하게 복제된 것을 확인하고, 카산드라의 성능에 대한 확신을 얻었다.

---

## **2. 임시 저장소(Cache) - EvCache**

### EvCache란?

넷플릭스가 직접 개발해서 사용하는 초고성능 분산 캐싱 시스템

- **EvCache의 핵심 특징**
    1. **분산 시스템 (Distributed System)**
        - 하나의 거대한 캐시 서버가 아니라, 수천, 수만 대의 작은 캐시 서버(EC2 인스턴스)들이 뭉쳐서 하나의 큰 캐시처럼 동작한다.
        - 이를 통해 엄청난 양의 데이터를 저장하고 처리할 수 있으며(수평적 확장), 일부 서버가 고장 나도 전체 서비스에는 영향이 거의 없다.
    2. **AWS에 최적화**
        - 넷플릭스는 모든 시스템을 AWS(아마존 웹 서비스) 클라우드 위에서 운영한다. EvCache는 처음부터 AWS 환경에 맞춰 설계되어, AWS의 여러 기능(EC2, SQS, Eureka 등)과 아주 매끄럽게 연동된다.
    3. **데이터 복제 및 안정성**
        - 데이터를 여러 복제본으로 만들어 서로 다른 물리적 공간(가용 영역, Availability Zone)에 나누어 저장한다. 그래서 특정 데이터센터에 화재가 나거나 정전이 되어도, 다른 곳에 저장된 복제본을 사용해 중단 없이 서비스를 제공할 수 있다.
    4. **클라이언트 (EvCache Client)**
        - "원격 캐시 무효화" 기능이 바로 이 클라이언트에 구현되어 있다. 단순히 데이터를 저장하고 꺼내는 것을 넘어, 여러 지역의 캐시 데이터가 일관성을 유지하도록 똑똑하게 관리하는 역할을 한다.

### **Active-Active 환경에서의 캐시 문제**

- A 지역의 캐시와 B 지역의 캐시에 저장된 데이터가 서로 달라, 사용자에게 **오래되거나 일치하지 않는 정보**를 보여줄 수 있는 문제가 발생한다.

### **해결책: 원격 캐시 무효화 (Remote Cache Invalidation)**

두 지역의 캐시 데이터를 실시간으로 똑같이 맞추는 복잡한 방식 대신, 한쪽 데이터가 바뀌면 다른 쪽 캐시를 그냥 **"버리게 만드는"** 방식을 사용한다.

- **1단계: 데이터 변경 발생 (A 지역)**
A 지역에서 사용자가 '내가 찜한 콘텐츠' 목록을 변경하면, 원본 데이터(카산드라)가 업데이트된다.
- **2단계: 캐시 무효화 메시지 전송 (A → B)**
A 지역의 EvCache 클라이언트는 B 지역으로 "이제 '내가 찜한 콘텐츠' 캐시는 낡았으니 버려!"라는 메시지를 보낸다. B 지역은 해당 캐시 데이터를 삭제한다.
- **3단계: 데이터 요청 발생 (B 지역)**
B 지역의 사용자가 '내가 찜한 콘텐츠'를 요청하면, 서버는 캐시가 비어있는 것(방금 버렸으니까)을 확인한다.
- **4단계: 최신 데이터 로드 (카산드라 → B 캐시)**
서버는 원본 데이터가 있는 카산드라에 가서 가장 최신 목록을 가져와 사용자에게 보여주고, 그 최신 데이터를 다시 B 지역 캐시에 저장한다.

# 여러 환경 및 지역에 걸친 배포 자동화

## 배포 자동화가 필요해진 배경

넷플릭스가 미국을 넘어 유럽으로 서비스를 확장하고, 미국 내에서 Active-Active(다중 데이터센터 동시 운영)를 도입하면서 소프트웨어를 배포하고 관리해야 할 환경의 수가 폭발적으로 증가했다.

- **초기:** 미국 테스트(Test) 환경, 미국 운영(Production) 환경 (총 2개)
- **유럽 확장:** 유럽 테스트, 유럽 운영 환경 추가 (총 4개)
- **미국 Active-Active 도입:** 미국 서부 테스트, 미국 서부 운영 환경 추가 (총 6개 이상)

## 문제점

개발자가 새로운 버전의 소프트웨어를 출시할 때마다, **아스가르드(Asgard)**라는 배포 도구를 사용해 이 6개가 넘는 환경에 **일일이 수동으로 배포**해야 했다. 이 과정은 매우 번거롭고 시간이 많이 걸리며, 사람이 하기에 실수가 발생할 위험이 컸다.

---

## **해결책: 워크플로우 자동화 도구 ‘미미르(Mimir)’**

이러한 반복적이고 복잡한 배포 과정을 자동화하기 위해, 넷플릭스 내부 도구팀이 **미미르(Mimir)**라는 워크플로우 도구를 개발했다. 미미르는 개별 배포를 담당하는 아스가르드(Asgard)를 지휘하는 **'마스터 컨트롤러'** 역할을 한다.

- **미미르(Mimir)의 핵심 기능**
    1. **다중 리전 배포 계획 정의**
    개발자는 미미르에게 "이 서비스를 A 지역 테스트 환경에 배포하고, 성공하면 A 지역 운영 환경에 배포해. 그리고 8시간 기다렸다가 문제가 없으면 B 지역에도 똑같이 배포해줘." 와 같이 **전체 배포 시나리오를 정의**할 수 있다.
    2. **단계적 순차 배포 (Staged Sequence)**
    미미르는 개발자가 정의한 시나리오에 따라, 사람의 개입 없이도 여러 지역에 걸친 배포 과정을 **단계별로 알아서 순차적으로 실행**한다.
    3. **자동 카나리 분석 및 롤백 연동**
    미미르는 배포 과정에 다음과 같은 안전장치를 통합했다.
        - **카나리 분석 (Canary Analysis):** 새로운 버전을 일부 소수의 사용자에게만 먼저 공개하여 문제가 없는지 자동으로 테스트하는 기법.
        - **자동 롤백 (Automated Rollback):** 카나리 분석 단계에서 문제가 감지되면, 시스템이 자동으로 배포를 중단하고 이전의 안정적인 버전으로 되돌리는 기능.

---

## 넷플릭스의 배포 전략

### **점진적 전 세계 배포 (Gradual Worldwide Deployment)**

넷플릭스는 미미르를 이용해 새로운 버전을 한꺼번에 전 세계에 배포하지 않는다.

- **1단계:** 첫 번째 지역에 새로운 버전을 배포한다.
- **2단계:** 몇 시간 동안 시스템을 면밀히 모니터링하며 예상치 못한 문제가 발생하는지 지켜본다.
- **3단계:** 문제가 없다고 판단되면, 다음 지역으로 배포를 진행한다.

이러한 전략은 **문제가 발생하더라도 그 영향을 하나의 지역으로 국한**시켜, 전 세계적인 서비스 장애로 번지는 것을 막는 매우 안전한 배포 방식이다.

이러한 자동화된 시스템을 통해, 넷플릭스 개발자들은 수동적이고 반복적인 배포 작업에서 벗어나, 전 세계의 여러 환경에 걸쳐 새 버전을 **빠르고 안전하게** 출시할 수 있게 되었다.

# 원숭이 - 고릴라, 콩, 스플릿 브레인

---

## Simian Army(삼인 아미)란? - 고릴라, 콩, 스필릿 브레인

- “원숭이 군단”이라는 뜻으로, 넷플릭스가 시스템을 의도적으로 망가뜨리기 위해 사용하는 여러 도구(원숭이)들의 모음이다.
- 이를 통해 다양한 유형의 장애에 대한 서비스의 복원력을 검증하고, 장애를 겪을수록 더 강해지는 ‘반취약성’ 시스템을 만드는 방법을 배운다.

---

## **1. 카오스 고릴라 (Chaos Gorilla) - 데이터센터(AZ) 장애 시뮬레이션**

- **시뮬레이션 하는 장애가용 영역(Availability Zone, AZ) 하나를 통째로** 다운시킨다. AZ는 물리적으로 분리된 하나 이상의 데이터센터 묶음으로, 서버 한두 대가 죽는 것과는 비교할 수 없는 큰 규모의 장애다.
- **테스트 목표**
하나의 AZ가 마비되더라도, 나머지 AZ에 있는 서비스들이 서비스 품질 저하 없이 모든 사용자 요청을 계속해서 처리할 수 있는지 검증하는 것이 목표다.

## **2. 스플릿 브레인 (Split-brain) - 리전(지역) 간 네트워크 단절 시뮬레이션**

- **시뮬레이션 하는 장애리전(Region) 간의 네트워크 연결을 의도적으로 차단**한다. 예를 들어, 미국 동부와 서부 데이터센터 그룹 간의 통신을 완전히 끊어버린다.
- **테스트 목표**
각 리전이 서로 통신이 안 되는 **'고립된 섬' 상태**에서도 독립적으로 정상 운영되는지 검증하는 것이 목표다. 이 상태에서는 리전 간 데이터 복제는 잠시 중단되고 쌓이지만, 각 리전 내의 핵심 서비스는 문제없이 동작해야 한다.

## **3. 카오스 콩 (Chaos Kong) - 리전 전체 장애 시뮬레이션**

- **시뮬레이션 하는 장애**
넷플릭스가 수행하는 **가장 큰 규모의 장애 시뮬레이션**으로, **리전 하나(예: 미국 동부 전체)가 완전히 다운**되는 최악의 재난 시나리오를 가정한다.
- **테스트 목표**
장애가 발생한 리전의 모든 사용자 트래픽을 다른 정상 리전으로 성공적으로 이전시킬 수 있는지 검증하는 것이다.
- **실제 테스트 시 특별 조치 (사용자 경험 보호)**
실제 사용자의 서비스 이용에 피해를 주지 않기 위해, 실제 비상사태와는 다르게 몇 가지 조치를 추가했다.
    1. **사용자 리디렉션:** 실제 장애였다면 오류를 봤을 사용자도, 테스트 중에는 "실패한" 리전으로 향하는 요청이 정상 리전으로 자동 연결되도록 처리했다.
    2. **점진적인 트래픽 전환:** 비상사태처럼 트래픽을 한 번에 옮기지 않고, **점진적으로 천천히** 옮겼다.
        - **이유 1 (서버 증설 시간 확보):** 트래픽을 받을 정상 리전의 서비스들이 늘어난 부하를 감당하기 위해 서버 수를 늘릴(scale up) 시간을 주기 위함이다.
        - **이유 2 (캐시 워밍업):** 캐시(임시 저장소)에 데이터를 미리 채워넣어(warm up), 갑작스러운 요청 폭증에 대비할 시간을 주기 위함이다.

---

## **테스트 결과**

미국 전체 트래픽의 대부분을 성공적으로 **서부 리전으로 옮겨 24시간 이상 완벽하게 운영**했고, 이를 통해 서부 리전이 미국 전체 부하를 감당할 수 있음을 증명했다.

# 실제 장애 조치

카오스 콩을 통한 최종 검증이 끝나기도 전에, 실제로 Active-Active 아키텍처의 힘을 증명할 기회가 찾아왔다.

### **상황 발생**

한 지역의 핵심 시스템 중 하나에 심각한 성능 저하가 발생하여, 해당 시스템을 사용하는 대부분의 서비스가 응답하지 않는 상태에 빠졌다.

### **과거였다면**

수많은 사용자가 장시간 동안 서비스를 이용하지 못하는 심각한 장애로 이어졌을 것이다.

### **Active-Active 도입 후**

1. **실제 장애 조치 실행:** 개발된 도구들을 사용해 문제가 생긴 지역의 모든 사용자 트래픽을 정상 지역으로 즉시 이전시켰다.
2. **빠른 서비스 복구:** 매우 짧은 시간 안에 모든 사용자의 서비스 품질이 정상적으로 복구되었다.
3. **안정적인 후속 조치:** 서비스가 안정된 상태에서, 서두르지 않고 근본 원인을 분석하고 해결책을 배포한 뒤, 트래픽을 원래대로 되돌릴 충분한 시간을 확보할 수 있었다.

# 가용성의 다음 단계: 완전 자동화를 향하여

지금까지 설명한 모든 작업은 시작에 불과하며, 넷플릭스는 프로젝트 2단계를 진행하며 시스템을 더욱 발전시키고 있다.

## **프로젝트 2단계 목표**

- **운영 자동화:** 현재 사람이 수동으로 개입하는 단계들을 최대한 자동화한다.
- **의사결정 시간 단축:** 장애 조치를 실행할지 '결정'하는 데 걸리는 시간을 최소화한다.
- **실행 시간 단축:** 실제로 트래픽을 이전하는 '실행'에 걸리는 시간을 최소화한다.

## **새로운 과제: 간헐적 장애 대응**

시스템이 완전히 다운된 상황보다, **'느리게 응답하거나', '가끔씩만 실패하는'** 애매한 상황에 대처하는 것이 훨씬 더 어렵다.

## **대응 도구: 레이턴시 몽키 (Latency Monkey)**

- **역할:** 시스템에 의도적으로 **지연(Latency)과 오류(Error)**를 주입하는 원숭이.
- **테스트 목표:** 시스템이 예측 불가능한 '느림'이나 '간헐적 실패'에 얼마나 잘 견디는지 확인하고 개선점을 찾는다.

---

# **요약 및 성공의 비결**

## **핵심 원칙**

서비스의 규모가 커지고 속도가 빨라질수록 장애는 피할 수 없다. 따라서 **격리(Isolation)**와 **중복성(Redundancy)** 원칙을 통해 장애가 발생해도 무너지지 않는 견고한 시스템을 구축해야 한다.

## **성공의 비결**

1. **훌륭한 팀과 엔지니어:** 넷플릭스 내부의 뛰어난 인적 자원.
2. **AWS의 신속한 지원:** 필요할 때 새로운 지역에 대규모 서버(용량)를 즉시 준비해준 AWS의 능력.
3. **강력한 팀워크:** 이 모든 것을 단 몇 달 만에 가능하게 한 모든 팀의 유기적인 협력.
