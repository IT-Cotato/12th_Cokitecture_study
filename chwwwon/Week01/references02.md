### 2. 알아야 할 기본 숫자들

- 일반적인 작업에 소요되는 시간
    
    
    | 작업 종류 | 예상 시간 (Latency) |
    | --- | --- |
    | L1 캐시 참조 | 0.5 ns |
    | 분기 예측 실패 | 5 ns |
    | L2 캐시 참조 | 7 ns |
    | 뮤텍스 잠금/해제 | 100 ns |
    | 주 메모리 참조 | 100 ns |
    | Zippy로 1KB 압축 | 10,000 ns |
    | 1Gbps 네트워크로 2KB 전송 | 20,000 ns |
    | 메모리에서 1MB 순차 읽기 | 250,000 ns |
    | 같은 데이터센터 내 왕복 지연 | 500,000 ns |
    | 디스크 탐색 | 10,000,000 ns |
    | 네트워크에서 1MB 순차 읽기 | 10,000,000 ns |
    | 디스크에서 1MB 순차 읽기 | 30,000,000 ns |
    | CA→네덜란드→CA 패킷 전송 | 150,000,000 ns |
- 고려할 점
    - 옵션별 성능 차이가 매우 크다는 점에 주목.
    - 데이터센터 간 거리가 멀어 전송 시간이 오래 걸림.
    - 메모리는 빠르고 디스크는 느림.
    - 간단한 압축 알고리즘을 사용하면 네트워크 대역폭을 절반 이상 절약할 수 있음.
    - 쓰기 작업은 읽기보다 40배 더 비용이 듦.
    - 전역 공유 데이터는 비쌈(근본적 한계). 공유 객체에 대한 잠금 경합으로 인해 성능이 급격히 저하되고 트랜잭션이 직렬화되어 느려짐.
    - 쓰기 확장을 고려하여 아키텍처 설계
    - 쓰기 경합을 최소화하도록 최적화
    - 쓰기 작업을 가능한 한 병렬화하여 최적화

### 3. 예제: 썸네일 이미지 30개를 보여주는 페이지 설계

- 조건
    - 이미지 용량: 256K
    - 디스크 탐색 속도: 10 ms
    - 디스크 전송 속도: 30 MB/s
1. **Serial**
    - 이미지 읽고 디스크 탐색 수행
    - 성능
        - 디스크 탐색 = 30 * 10 ms/seek = 300
        - 실제 데이터 전송 = 30회 * 256K / 30 MB/s = 256
        
        → 약 560 ms
        
2. **Parallel**
    - read 작업 병렬 처리
    - 성능
        - 디스크 탐색 = 10 ms
        - 실제 데이터 전송 = 256K / 30 MB/s ≈ 8 ms
        
        → 약 18 ms (실제로는 30-60 ms 정도)
        

**추가 고려 사항**

Q1. 단일 썸네일 이미지를 캐싱하는 것이 합리적인가?

- A1.
    
    **개별 썸네일 이미지를 캐시에 저장하고 불러오는 데 걸리는 시간 계산하기**
    
    - 캐시 쓰기 속도 = **100 MB/s**
    - 캐시 읽기 latency = **1 ms**
    - 256K / 100 MB/s + 1 ≈ 3.5
    
    자주 조회되는 이미지를 캐싱하면 응답 지연↓
    
    But, 캐시 크기 관리 어려움 - 개별 이미지 수가 많으면 캐시 메모리 소비↑
    

Q2. 이미지 전체 세트를 하나의 항목으로 캐싱해야 하는가?

- A2.
    
    **전체 세트를 한 번에 캐시에 저장하고 다시 가져올 때의 성능을 측정**
    
    - 30 * 256K / 100 MB/s + 1 ≈ 76
    
    한 번의 요청에서 항상 여러 이미지를 세트로 보여줘야 하는 경우 유용. 캐시 접근 지연시간↓
    
    일부만 필요할 때는 비효율. 세트 내 일부만 변경되는 경우 전체 세트를 무효화해야 함.
    

Q3. 썸네일을 미리 계산하는 것이 합리적인가?

- A3.
    
    **요청 시점에 썸네일을 동적으로 만드는 데 걸리는 시간과 미리 생성해 두었을 때의 조회 시간을 비교**
    
    - 원본 이미지 = **5 MB**
    - 썸네일 변환 CPU 비용 = **50 ms**
    - 실시간 생성 = 5 MB / 30 MB/s + 50 ms ≈ 217 ms
    
    미리 생성해 저장해두면 단순 읽기만 수행하므로 응답 지연↓. 실시간 변환시 CPU + 디스크 IO 비용이 큼
    
    스토리지 공간은 더 필요. (원본 + 썸네일 저장)’
    
    보통 저장 공간 비용보다 사용자 응답속도/CPU 절감 효과가 더 큼.
