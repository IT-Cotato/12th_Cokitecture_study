# 1장. 사용자 수에 따른 규모 확장성

## 단일 서버

- 단일 서버: 모든 컴포넌트가 한 대의 서버에서 실행
- 과정
    1. 도메인 이름을 통해 웹사이트 접속
    2. DNS 조회하여 IP 주소 반환
    3. 해당 IP로 HTTP 요청 전달
    4. 요청 받은 서버가 응답

---

## 데이터베이스

- 서버를 두 개로 분리
    1. 웹/모바일 트래픽 처리 담당
    2. DB 담당
- 종류
    - RDBMS: 관계형 데이터 베이스
        
        → 자료를 테이블, 열, 칼럼으로 표현. 관계에 따라 조인하여 사용 가능
        
    - NoSQL: 비 관계형 데이터베이스
        
        → 키-값 저장소, 그래프 저장소, 칼럼 저장소, 문서 저장소로 구성. 조인 연산 불가능
        
- NoSQL 사용 사례
    - 낮은 응답 지연시간 필요시
    - 비정형 데이터 관리
    - 데이터의 직렬/역직렬화 가능시
    - 많은 양의 데이터 관리시

---

## 수직적 규모 확장 vs 수평적 규모 확장

- 수직적 규모 확장(scale up)
    - 서버를 고사양으로 바꾸는 것
    - 유입 트래픽 양이 적을 때 유리
    - 장점: 단순함
    - 단점: 무한 증설 불가. 장애 자동복구 X. 다중화 방안 제시 X
- 수평적 규모 확장(scale out)
    - 서버 수를 추가

### 로드밸런서

- 로드밸런서: 부하 분산 집합에 속한 웹 서버의 트래픽을 고르게 분산
- 방식: 사설 IP 주소를 이용하여 웹 서버에 접속
    
    Ex) 클라이언트는 공개 IP로 접속 → 로드밸런서가 사설 IP로 웹 서버와 **직접** 통신
    

### 데이터베이스 다중화

- 데이터베이스 다중화: 주-부 서버를 두고, 원본 데이터는 주 서버, 사본은 부 서버에 저장하는 방식
- 주 서버: 쓰기 연산 실행
    
    Ex) insert, delete, update 등
    
- 부 서버: 읽기 연산 실행
- 장점
    - **성능 향상**: 병렬 처리 쿼리 수의 증가로 연산 속도 향상
    - **안정성**: 일부 DB가 멈춰도 여러 장소에 데이터 다중화 가능
    - **가용성**: 여러 지역에 데이터 복제 가능. 한 서버 중단시 다른 서버로 대체 가능

## 캐시

- 캐시: 자주 호출되는 연산을 보관하는 저장소
    - DB 호출 없이 연산 가능 → 속도 향상
- **캐시 계층**: 데이터가 잠시 보관되는 곳
- 유의점
    - **데이터 갱신** 없는지 / **참조 빈번**한지 확인
    - **영속적 데이터** 저장 X (지속적 저장소에 저장)
    - **데이터 만료 처리**: 만료된 데이터는 캐시에서도 삭제
    - **일관성 유지** (원본 저장소 - 사본 저장소 동일하게 유지)
    - **단일 장애 지점(SPOF)**: 한 지점의 장애가 전체 시스템을 중단시킴
    - 캐시 **메모리 크기**
    - **데이터 방출 정책**: 캐시 다 차면 어떤 데이터 버릴지

## 콘텐츠 전송 네트워크(CDN)

- CDN: 정적 콘텐츠 전송에 사용되는 지리적 분산 네트워크
    - 사용자와 **가장 가까운** CDN 서버가 정적 콘텐츠 전달 (가까울수록 빠름)
    - 정적 콘텐츠: 이미지, 비디오, CSS, JavaScript 등
- 동작 과정
    1. 클라이언트가 이미지에 접근: CDN이 제공한 URL 이용
    2. CDN 서버에서 이미지 검색, 없으면 원본 서버에서 검색
    3. 서버가 CDN 서버에 전달
    4. CDN에서 파일 캐시 후 클라이언트에 전달
    5. 이후 재요청시 캐시로 처리
- 고려사항
    - **비용**: 자주 사용X면 삭제 (비용↑)
    - 적절한 **만료 시한** 설정
    - **CDN 장애 대처 방안**: CDN 서버 오류시 처리 방안 고려 (직접 원본 서버 접속)
    - **콘텐츠 무효화**: 강제로 콘텐츠 제거

## 무상태(stateless) 웹 계층

- 무상태 웹 계층: 상태 정보가 제거된 웹
    - 상태 정보: 사용자 세션 데이터 등
    - 상태 정보는 DB에 저장하고 필요시 호출하여 사용
- 상태 의존 아키텍처
    
    → 클라이언트의 요청을 항상 같은 서버로 전송
    
    → 로그밸런서가 고정 세션 기능 제공
    
    → 로드벨런서에 부담↑. 서버 장애 처리 복잡
    
- 장점
    - **단순한 구조**
    - **안정적**
    - **규모 확장 용이**

## 데이터 센터

- 지리적 라우팅: 클라이언트와 가까운 데이터 센터로 안내하는 절차
    - geoDNS: 클라이언트 위치에 따라 도메인 IP 주소를 결정하는 DNS 서비스

### 다중 데이터센터

- 다중 데이터센터 아키텍처: 두 개 이상의 데이터 센터를 이용하는 것
- 고려사항
    - 트래픽 우회: 올바른 데이터 센터로 트래픽 보내는 방법 찾기
    - 데이터 동기화: 데이터 센터간 동일한 데이터 베이스 이용
    - 테스트와 배포: 모든 데이터 센터에 동일한 서비스 설치. 자동화된 배포 도구 이용

## 메시지 큐

- 메시지 큐: 메시지 무손실을 보장하는 비동기 통신 컴포넌트
    - 메시지 무손실: 메시지 큐에 보관된 메시지는 클라이언트 사용시까지 안전 보장
- 장점
    - 느슨한 서버 결합: 규모 확장성 보장. 안정적 애플리케이션 구성
    - 생산자 →  소비자 프로세스 다운시 메시지 발생 가능
    - 소비자 → 생산자 서비스 가용 상태 아니어도 메시지 수신 가능

---

## 로그, 메트릭 그리고 자동화

- 로그: 단일 서비스로 모으는 도구 이용
- 메트릭: 사업에 유용한 정보 수집. 시스템 현상태 파악
    - 호스트 단위 메트릭: CPU, 메모리, 디스크 I/O 관련
    - 종합 메트릭: DB 계층의 성능, 캐시 계층의 성능
    - 핵심 비즈니스 메트릭: 일별 능동 사용지, 수익, 재방문 등
- 자동화: CI/CD 등. 개발 생산성 향상

## 데이터베이스의 규모 확장

### 수직적 확장

- 수직적 확장: 고성능 자원 증설
- 장점
    - 많은 양의 데이터 보관 & 처리 가능
- 단점
    - 무한 증설 불가: 한 대로 감당 불가능
    - SPOF↑
    - 비↑

### 수평적 확장(Sharding)

- 수평적 확장: 서버를 추가
- 샤드(shard): 대규모 데이터베이스를 작은 단위로 분할
    - 모든 샤드는 동일 스키마 사용
    - 샤드가 데이터 중복 X
- 샤딩 키(파티션 키): 데이터 분산을 결정하는 하나 이상의 칼럼
- 고려사항
    - 데이터 재 샤딩
        - 한 샤드로 감당 불가
        - 샤드 간 데이터 분포 불균등 (샤드 소진)
    - celebrity 문제(hotspot key): 특정 샤드에 쿼리가 집중됨
    - 조인과 비정규화: 여러 샤드의 데이터 조인 어려움
        
        → DB를 비정규화하여 하나의 테이블에서 질의 수행
