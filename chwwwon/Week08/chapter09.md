# 9장. 웹 크롤러 설계

### 웹 크롤러란?

---

> 검색 엔진에 널리 쓰는 기술로, 웹에 새로 올라오거나 갱신된 콘텐츠를 찾아낸다.
> 
- 로봇, 스파이더로도 불림
- **콘텐츠**: 웹 페이지, 이미지, 비디오, PDF 파일 등
- **수집 방식**: 몇 개 웹 페이지에서 시작하여 링크를 따라나가며 새로운 콘텐츠 수집

### 크롤러의 이용

---

1. **검색 엔진 인덱싱**: 웹 페이지를 모아 검색 엔진을 위한 로컬 인덱스 생성
    - 가장 보편적인 용례
    
    ex) 구글 검색 엔진이 사용하는 Googlebot
    
2. **웹 아카이빙**: 나중에 사용할 목적으로 장기보관을 위해 웹에서 정보를 모으는 절차
    - 국립 도서관에서 주로 사용
    
    ex) 미국 국회 도서관, EU 웹 아카이브
    
3. **웹 마이닝**: 인터넷에서 유용한 지식 도출
    
    ex) 유명 금융 기업: 주주 총회 자료나 연차 보고서 다운받아 기업의 핵심 사업 방향 결정
    
4. **웹 모니터링**: 인터넷에서 저작권이나 상표권이 침해되는 사례 모니터링
    
    ex) 디지마크: 웹 크롤러를 사용해 해적판 저작물 찾아 보고
    

# 1. 문제 이해 및 설계 범위 확정

---

### 웹 크롤러의 기본 알고리즘

1. URL 집합이 입력으로 주어지면, 해당 URL의 모든 웹 페이지 다운로드
2. 다운받은 웹 페이지에서 URL 추출
3. 추출된 URL들을 다운로드할 URL 목록에 추가하고 위 과정 처음부터 반복

### 웹 크롤러가 만족시켜야 할 속성

- **규모 확장성**: 병행성을 사용하면 거대한 웹을 보다 효과적으로 크롤링 가능
- **안정성(robustness)**: 비정상적 입력이나 환경(잘못 작성된 HTML, 무반응 서버, 장애, 악성코드 등)에 대응
- **예절(politeness)**: 수집 대상 웹 사이트에 짧은 시간동안 너무 많은 요청 보내지 않기
- **확장성(extensibility)**: 새로운 형태의 콘텐츠 지원 가능

### 개략적 규모 추청

- 매달 10억 개의 웹 페이지를 다운로드
- QPS = 10억/30일/24시간/3600초 = 대략 400페이지/초
- 최대 QPS = 2 * QPS = 800
- 웹 페이지 평균 크기는 500k로 가정
- 10억 페이지 * 500k = 500TB/월
- 총 500TB * 12M * 5Y = 30PB

# 2. 개략적 설계안 제시 및 동의 구하기

---

### 시작 URL 집합

> 웹 크롤러가 크롤리을 시작하는 출발점
> 
- 가능한 한 많은 링크를 탐색할 수 있는 URL 결정
    - 일반적으로 전체 URL 공간을 작은 부분집합으로 나누는 전략 사용
    - 주제별로 다른 시작 URL 사용

### 미수집 URL 저장소

> 다운로드할 URL 저장 관리 컴포넌트
> 
- 웹 크롤링 상태는 ‘**다운로드 할 URL’** / ‘다운로드 된 URL’로 구분 가능
- FIFO queue 형식

### HTML 다운로더

> 인터넷에서 웹 페이지를 다운로드하는 컴포넌트
> 
- 미수집 URl 저장소가 제공한 페이지를 다운로드함

### 도메인 이름 변환기

> URL에 대응되는 IP 주소 검색
> 
- 웹 페이지 다운로드를 위해 URL → IP 변환 필요

### 콘텐츠 파서

> 웹 페이지 다운로드시 파싱&검증 과정 수행
> 
- 웹 페이지 문제 및 저장공간 낭비 방지, 크롤링 속도 향상 → **독립된 컴포넌트로 분리**

### 중복 콘텐츠 여부

> 데이터 중복 감소 및 데이터 처리 소요 시간 절감을 이룸
> 
- 약 29% 가량의 웹 페이지 콘텐츠가 중복됨
- 웹 페이지 해시값 비교를 통해 구현

### 콘텐츠 저장소

> HTML 문서 보관 시스템
> 
- 저장소 구현을 위한 데이터 유형, 크기, 저장소 접근 빈도, 데이터 유효 기간 등 고려
- 디스크와 메모리를 통시에 사용하는 저장소 선택
    - 데이터 양이 너무 많으므로 대부분의 콘텐츠를 디스크게 저장
    - 인기 있는 콘텐츠는 메모리에 두어 접근 지연시간 절감

### URL 추출기

> HTML 페이지를 파싱하여 링크 골라냄
> 
- 상대 경로를 절대 경로로 변환

상대경로 → 절대경로 변환 예시

### URL 필터

> 특정 URL을 크롤링 대상에서 배제
> 
- 특정 콘텐츠 타입이나 파일 확장자를 갖는 URL
- 접속시 오류가 발생하는 URL
- 접근 제외 목록에 포함된 URL

### URL 방문 여부

> 이미 방문한 URL을 구분하여 서버 부하 절감 및 시스템의 무한 루프 방지
> 
- 자료구조를 이용하여 이미 방문한 URL, 미수집 URL을 추적
    - 블룸 필터, 해시테이블 주로 사용

### URL 저장소

> 이미 방문한 URL 보관
> 

![image.png](9%EC%9E%A5%20%EC%9B%B9%20%ED%81%AC%EB%A1%A4%EB%9F%AC%20%EC%84%A4%EA%B3%84/image%201.png)

### 전체작업흐름

1. 시작 URL들을 미수집 URL 저장소에 저장
2. HTML 다운로더는 미수집 URL 저장소에서 URL 목록 가져옴
3. HTML 다운로더는 도메인 이름 변환기로 URL의 IP 주소 알아내고, 해당 주소에 접속하여 웹 페이지 다운로드
4. 콘텐츠 파서는 다운된 HTML 페이지를 파싱하여 올바른 형식을 갖춘 페이지인지 검증
5. 콘텐츠 파싱과 검증이 끝나면 중복 콘텐츠인지 확인하는 절차 개시
6. 중복 콘텐츠인지 확인을 위해 해당 페이지가 이미 저장소에 있는지 확인
    1. 이미 저장소에 있는 콘텐츠는 버림
    2. 저장소에 없으면 저장소에 저장 후 URL 추출기로 전달
7. URL 추출기는 해당 HTML 페이지에서 링크 골라냄
8. 골라낸 링크를 URL 필터로 전달
9. 필터링 끝난 후 남은 URL만 중복 URL 판별 단계로 전달
10. URL 저장소에 보관된 URL인지 확인하여 처리 여부 확인. (중복은 버림)
11. 저장소에 없는 URL은 URL 저장소에 저장 후 미수집 URL 저장소에 전달

# 3. 상세 설계

---

### DFS vs BFS

> DFS 방식과 BFS 방식중 어떤 탐색법을 사용할 지 결정
> 

웹 - 유향 그래프

페이지 - 노드

하이퍼링크 - 엣지

- **DFS: 깊이 우선 탐색법**
    - 그래프 크기가 커지면 어느 정도로 깊숙이 가게 될지 가늠 어려움
    
    → 좋은 선택 X
    
- **BFS: 너비 우선 탐색법**
    - FIFO 큐를 이용하는 알고리즘
    
    → 탐색할 URL을 한쪽에 넣고 다른 한쪽으로 꺼냄
    
- BFS 단점
    1. **한 페이지에서 나오는 링크의 상당수는 같은 서버로 되돌아감**
        - 크롤러는 같은 호스트에 속한 많은 링크를 다운받으라 바쁨
        - 링크를 병렬로 처리하면 요청 과다로 서버에 과부하 발생
            
            → “impolite 크롤러”
            
    2. **표준적 BFS 알고리즘은 URL간 우선순위가 없음**
        - 처리 순서에 있어 모든 페이지를 공평하게 대우
        - But, 모든 웹 페이지가 같은 수준의 품질, 중요성을 갖지 않음
        
        → 페이지 순위, 사용자 트래픽 양, 업데이트 빈도 등 여러 척도에 따라 우선순위 구별 필요
        

### 미수집 URL 저장소

> “politeness 크롤러”, URL 사이의 우선순위와 신선도를 구별하는 크롤러 구현 가능
> 

**예의**: 수집 대상 서버로 짧은 시간 안에 너무 많은 요청 보내지 않기

- 안정장치 없는 크롤러는 웹 사이트 마비시킬 수 있음
- **동일 웹 사이트에 대해 한 번에 한 페이지만 요청 & 같은 웹 사이트의 페이지 다운을 위해 시간차를 두고 실행**
    
    → 웹 사이트의 호스트명과 다운로드 수행 작업 스레드 사이의 관계 유지
    
- 각 다운로드 스레드 별 FIFO 큐를 두고, 해당 큐에서 꺼낸 URL만 다운로드 수행

1. **큐 라우터**: 같은 호스트의 URL은 항상 같은 큐로 가도록 보장
2. **매핑 테이블**: 호스트 이름과 큐 사이의 관계 보관
3. **FIFO 큐**: 같은 호스트에 속한 URL은 언제나 같은 큐에 보관
4. **큐 선택기**: 큐를 순회하며 큐에서 URL을 꺼내 해당하는 작업 스레드에 전달
5. **작업 스레드**: 전달된 URL을 다운로드. 순차 처리. 지연 시간 가능

![image.png](9%EC%9E%A5%20%EC%9B%B9%20%ED%81%AC%EB%A1%A4%EB%9F%AC%20%EC%84%A4%EA%B3%84/image%202.png)

**우선순위**: 다양한 척도를이용 하여 URL의 우선순위 분리

- 페이지랭크, 트래픽 양, 갱신 빈도 등의 척도 사용
- 순위결정장치를 이용하여 결정

1. **순위결정장치**: URL을 입력으로 받아 우선순위 계산
2. **큐**: 우선순위별로 하나의 큐 할당. 우선순위와 선택 확률 비례
3. **큐 선택기**: 임의 큐에서 처리할 URL을 꺼내는 역할

![image.png](9%EC%9E%A5%20%EC%9B%B9%20%ED%81%AC%EB%A1%A4%EB%9F%AC%20%EC%84%A4%EA%B3%84/image%203.png)

- 실제 구현: 전/후면 큐를 구분하여 설계

1. 전면 큐: 우선순위 결정 과정 처리
2. 후면 큐: 크롤러가 예의 바르게 동작하도록 보장

![image.png](9%EC%9E%A5%20%EC%9B%B9%20%ED%81%AC%EB%A1%A4%EB%9F%AC%20%EC%84%A4%EA%B3%84/image%204.png)

**신선도**: 주기적으로 페이지 재수집

- 웹 페이지의 변경 이력 활용
- 우선순위 활용. 중요 페이지는 좀 더 자주 재수집

**미수집 URL 저장소를 위한 지속성 저장장치**

→ 버퍼의 데이터를 주기적으로 디스크에 기록하여 I/O 비용 절감

- 전체 메모리 보관 - 안정성/규모 확장성 X
- 전부 디스크에 저장 - 속도 저하로 성능 병목지점

### HTML 다운로더

> HTTP 프로토콜을 통해 웹 페이지를 내려받음
> 

**Robots.txt**

- 로봇 제외 프로토콜이라 칭함
- 웹사이트가 크롤러와 소통하는 표준적 방법
    - 크롤러가 수집해도 되는 페이지 목록 포함
- 해당 파일을 주기적으로 다시 다운로드 받아 캐시에 보관

**성능 최적화 기법**

1. **분산 크롤링**
    - 성능 향상을 위해 여러 서버에 크롤링 작업 분산
    - URL 공간을 작은 단위로 분할하여 각 서버가 일부 다운로드 담당하도록 함
    
    ![분산 크롤링 최적화 예시](9%EC%9E%A5%20%EC%9B%B9%20%ED%81%AC%EB%A1%A4%EB%9F%AC%20%EC%84%A4%EA%B3%84/image%205.png)
    
    분산 크롤링 최적화 예시
    
2. **도메인 이름 변환 결과 캐시**
    - 크롤러 성능 병목 중 하나. (요청 전송-수신 작업의 동기화 때문)
    - 크롤러 스레드 중 하나라도 이 작업 수행시 다른 스레드의 DNS 요청 블록
    - DNS 조회 결과로 얻어진 도메인 이름과 IP 주소 사이의 관계를 캐시에 보관
    - 주기적으로 갱신하여 성능 향상
3. **지역성**
    - 크롤링 작업을 수행하는 서버를 지역별로 분산하는 방법
    - 페이지 다운로드 시간 절감
    - 크롤 서버, 캐시, 큐, 저장소 등 대부분의 컴포넌트에 적용 가능
4. **짧은 타임아웃**
    - 최대 대기 시간 미리 설정
    - 초과시 다운로드 중단 후 다음 페이지로 이동

### 안정성 확보 전략

**안정해시**

- 다운로더 서버들에 부하 분산시 적용 가능한 기술
- 다운로더 서버를 쉽게 추가/삭제 가능

**크롤링 상태 및 수집 데이터 저장**

- 크롤링 상태와 수집 데이터를 지속적 저장장치에 기록
- 장애 발생 복구 용이
- 중단후 재시작 쉬움

**예외처리**

- 전체 시스템 중단 없이 작업 가능

**데이터 검증**

- 시스템 오류 방지 수단

### 확장성 확보 전략

> 새로운 형태의 콘텐츠 지원 가능하도록 구현
>

### 문제 있는 콘텐츠 감지 및 회피 전략

> 중복, 의미 없는 또는 유해한 콘텐츠 감지 및 차단 방안
> 
1. **중복 콘텐츠**
    
    해시, 체크섬을 이용하여 중복 콘텐츠 탐지
    
2. **거미 덫**
    
    : 크롤러를 무한 루프에 빠뜨리도록 설계한 웹 페이지
    
    - URL 최대 길이 제한하여 회피
    - 사람이 수작업으로 확인하여 크롤러 탐색 대상 제외/URL 필터 목록에 걸어둠
3. **데이터 노이즈**
    
    : 가치가 없는 컨텐츠
    
    - 가치없는 콘텐츠를 제거

- 규모 확장성
- 예의
- 확장성
- 안정성

### 추가 논의 사항

- **서버 측 렌더링**: 동적 렌더링으로 동적으로 생성된 URL 처리
- **원치 않는 페이지 필터링**: 크롤링 자원은 유한하므로 스팸성 페이지를 필터링
- **데이터베이스 다중화 및 샤딩**: 데이터 계층의 가용성, 규모 확장성, 안정성 향상
- **수평적 규모 확장성**: 무상태 서버로 구현하여 서버 확장 가능
- **가용성, 일관성. 안정성**
- **데이터 분석 솔루션**
