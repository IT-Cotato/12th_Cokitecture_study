# 13장. 검색어 자동완성 시스템

입력중인 글자에 맞는 검색어 자동 완성

많이 이용된 검색어 k개를 자동완성하여 출력하는 시스템 설계

# 1단계 문제 이해 및 설계 범위 확정

---

### 요구사항

- **빠른 응답 속도**
    - 자동완성 검색어가 충분히 빠르게 표시될 것
- **연관성**
    - 자동완성 검색어는 사용자가 입력한 단어와 연관될 것
- **정렬**
    - 시스템의 계산 결과는 인기도 드으이 순위 모델에 의해 정렬될 것
- **규모 확장성**
    - 시스템은 많은 트래픽을 감당할 수 있도록 확장 가능할 것
- **고가용성**
    - 시스템의 일부에 장애 발생, 느려짐, 네트워크 문제 발생 시 시스템 사용 가능할 것

### 개략적 규모 추정

- 일단 능동 사용자는 천만 명으로 가정
- 평균적으로 한 사용자는 매일 10건의 검색 수행한다고 가정
- 질의마다 평균적으로 20바이트의 데이터 입력
- 검색창에 글자 백엔드에 요청 전송
    - 평균적으로 1회 검색당 20건의 요청이 전달됨
- 대략 초당 24,000건의 질의 발생
- 최대 QPS =QPS*2 = 대략 48,000
- 질의 가운데 20%는 신규 검색어. 대략 0.4GB 정도

## 트라이 자료구조

> 트라이를 이용해 인기있는 질의문 추출
> 

### 트라이 자료구조

![트라이 자료구조 예시](13%EC%9E%A5%20%EA%B2%80%EC%83%89%EC%96%B4%20%EC%9E%90%EB%8F%99%EC%99%84%EC%84%B1%20%EC%8B%9C%EC%8A%A4%ED%85%9C/image%203.png)

트라이 자료구조 예시

- 문자열을 간락하게 저장할 수 있는 자료구조
- 트리 형태의 자료구조
- 트리의 루트 노드는 빈 문자열 나타냄
- 각 노드는 글자 하나를 저장. 26개의 자식 노드를 가짐
- 각 트리 노드는 하나의 단어, 접두어 문자열을 가짐
- 시간 복잡도: $O(p) + O(c) + O(clogc)$
- 결과를 위해 전체 트라이을 다 검색하는 경우가 생길 수 있음
    - 해결1: 접두어 최대 길이를 제한
    - 해결2: 각 노드에 인기 검색어를 캐시

**접두어 최대 길이 제한**

- 검색어 최대 길이 제한 시 “접두어 노드 찾는” 단계 시간 복잡도 절약 가능
- 시간 복잡도: $O(1) + O(c) + O(clogc)$

**노드에 인기 검색어 캐시**

- 각 노드에 k개의 인기 검색어 저장
- 전체 트라이 검색하는 일 방지 가능
- 단점: 각 노드에 질의어를 저장할 공간이 많이 필요함

**최적화 기법 적용시 시간 복잡도**

1. 접두어 노드 찾는 시간 복잡도는 $O(1)$
2. 최고 인기 검색어 5개를 찾는 질의 시간 복잡도가 $O(1)$이 됨

→ 전체 시간 복잡도: $O(1)$

데이터 수집 서비스 예시

### 데이터 분석 서비스 로그

- 검색창에 입력된 질의에 관한 원본 데이터 보관
- 새로운 데이터 추가 가능
- 데이터 추가시 수정X
- 로그 데이터에 인덱스 걸지 않음

### 로그 취합 서버

- 데이버 분석 서비스로부터 나오는 로그는 양이 매우 크다
- 데이터를 잘 취함하여 시스템이 쉽게 소비하도록 해야 함

### 취합된 데이터

- time: 해당 주가 시작한 날짜
- frequency: 해당 질의가 해당 주에 사용된 횟수의 합

### 작업 서버

- 주기적으로 비동기적 작업을 실행하는 서버 집합
- 트라이 자료구조를 만들고 트라이 데이터베이스에 저장하는 역할 담당

### 트라이 캐시

- 트라이 데이터를 메모리에 유지하여 읽기 연산 성능 높임
- 트라이 데이터베이스의 스냅샷을 떠서 갱신

### 트라이 데이터베이스

- 지속성 저장소
- **문서 저장소**
    - 새 트라이를 매주 만듦
    - 주기적으로 트라이를 직렬화하여 데이터베이스에 저장
- **키-값 저장소**
    - 트라이에 보관된 모든 접두어를 해시 테이블 키로 변환
    - 각 트라이 노드에 보관딘 모든 데이터를 해시 테이블 값으로 변환

## 질의 서비스

![image.png](13%EC%9E%A5%20%EA%B2%80%EC%83%89%EC%96%B4%20%EC%9E%90%EB%8F%99%EC%99%84%EC%84%B1%20%EC%8B%9C%EC%8A%A4%ED%85%9C/image%205.png)

1. 검색 질의가 로드밸런서로 전송됨
2. 로드밸런서는 해당 질의를 API 서버로 보냄
3. API 서버는 트라이 캐시에서 데이터를 가져와 해당 요청에 대한 자동완성 검색어 제안 응답을 구성함
4. 데이터가 트라이 캐시에 없는 경우
데이터를 데이터베이스에서 가져와 캐시에 채움
    
    → 질의가 오면 캐시에 보관된 데이터를 사용해 처리 가능
    

### 최적화 방안

- **AJAX 요청**
    - AJAX 요청으로 자동완성된 검색어 목록 가져옴
    - 요청을 보내고 받기 위해 페이지 새로고침할 필요 없음
- **브라우저 캐싱**
    - 대부분의 자동완성 검색어는 자주 변화X
    - 제안된 검색어를 캐시에 넣어두면 후속 질의 결과를 캐시에서 바로 가져갈 수 있음
- **데이터 샘플링**
    - 모든 질의 결과를 로깅하면 과도한 CPU 자원과 저장공간 소진
    - N개 요청 중 1개만 로깅하도록 함
