# 12장. 채팅 시스템 설계

# 1단계 문제 이해 및 설계 범위 확정

---

- 응답지연이 낮은 일대일 채팅 기능
- 최대 100명까지 참여할 수 있는 그룹 채팅 기능
- 사용자의 접속상태 표시 기능
- 다양한 단말 지원. 하나의 계정으로 여러 단말에 동시 접속 지원
- 푸시 알림
- 5천만 DAU를 처리할 수 있을 것

# 2단계. 개략적 설계안 제시 및 동의 구하기

## 채팅 시스템의 통신 방식

---

> 클라이언트-채팅 서비스 간 통신
> 
- 채팅 서비스가 제공하는 기능
    - 클라이언트들로부터 메시지 수신
    - 메시지 수신자 결정 및 전달
    - 수신자가 접속 상태가 아닌 경우 접속할 때까지 메시지 보관

![클라이언트-채팅 서비스 관계](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/bc4e9f58-3354-412b-af6c-046507ff9f55.png)

클라이언트-채팅 서비스 관계

- 메시지 전송 시나리오
    - 일반적으로 **HTTP 프로토콜**을 사용하여 메시지 전달
    - 접속시 keep-alive 헤더를 사용하여 연결 유지
- 메시지 수신 시나리오
    - 메시지 전송 시나리오보다 복잡

### 폴링

> 클라이언트가 주기적으로 서버에게 새 메시지가 있는지 묻는 방식
> 

![폴링 방식 예시](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image.png)

폴링 방식 예시

- 폴링을 자주할수록 비용 증가
- 메시지가 없는 경우 서버 자원 낭비

### 롱 폴링

> 새 메시지 반환되거나 타임아웃 될 때까지 클라이언트 연결 유지
> 

![image.png](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%201.png)

- 새 메시지 받으면 기존 연결 종료
- 서버에 새로운 요청 보내 모든 절차 다시 시작
- 단점
    - 메시지 송수신 클라이언트가 서로 다른 서버에 접속 가능
    - 서버는 클라이언트의 연결 여부를 확인하기 어려움
    - **비효율**. 메시지를 많이 받는 클라잉너트는 타임아웃마다 주기적으로 서버에 재접속함

### 웹소켓

> 서버가 클라이언트에 비동기 메시지 보낼 때 사용하는 기술
> 

![image.png](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%202.png)

- 클라이언트가 웹소켓 연결 시작
- 한 번 맺어진 연결은 항구적 & 양방향
- 처음 HTTP로 시작되어 웹소켓 연결로 업그레이드
- 방화벽이  있는 환경에서도 잘 동작
    
    ![웹소켓을 이용한 설계 예시](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%203.png)
    
    웹소켓을 이용한 설계 예시
    
- 장점
    - 메시지 송수신 시 동일한 프로토콜 사용 가능
    - 설계와 구현이 단순하고 직관적
- 유의점
    - 웹소켓 연결은 항구적으로 유지되어야 함
    - 서버 측에서 연결 관리를 효율적으로 해야 함

## 개략적 설계안

---

![image.png](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%204.png)

- 채팅 서버: 클라이언트 사이에 메시지 중계하는 역할 담당
- 접속상태 서버: 사용자 접속 여부 관리
- API 서버: 로그인, 회원가입, 프로파일 변경 등 나머지 전부 처리
- 알림 서버: 푸시 알림 전송
- 키-값 저장소: 채팅 이력 보관

### 무상태 서비스

> 로그인, 회원가입, 사용자 프로파일 표시 등을 처리하는 전통적인 요청/응답 서비스
> 

![image.png](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%205.png)

- 로드밸런서 뒤에 위치함
    - 로드밸런서: 요청을 그 경로에 맞는 서비스로 정확하게 전달
    - 모놀리틱 서비스, 마이크로서비스 가능

### 상태 유지 서비스

> 각 클라이언트는 채팅 서버와 독립적인 네트워크 연결 유지 가능
> 
- 클라이언트는 서버가 살아있는 경우 다른 서버로 연결 변경X
- 특정 서버에 부하가 몰리지 않도록 해야 함

### 제3자 서비스 연동

> 푸시 알림
> 
- 앱이 실행중이지 않아도 알림을 받아야 함

### 규모 확장성

> 대량 트래픽 처리시에도 모든 연결을 최신 클라우드 서버 한 대로 처리해야 한다
> 

### 저장소

> 데이터 유형과 읽기/쓰기 연산 패턴 고려하여 데이터베이스 결정
> 
- 채팅 시스템이 다루는 데이터
    - **일반 데이터**: 사용자 프로파일, 설정, 친구 목록 등
    - **고유 데이터**: 채팅 이력

→ 키-값 저장소를 이용할 것

- 키-값 저장소의 장점
    - 수평적 규모확장이 쉬움
    - 데이터 접근 지연시간이 낮음
    - 관계형 데이터베이스보다 롱 테일 데이터 잘 처리

# 3단계. 상세 설계

## 서비스 탐색

---

**예시- 아파치 주키퍼**

- 사용 가능한 모든 채팅 서버를 등록
- 클라이언트가 접속 시도시 사전 기준에 따라 최적의 채팅 서버 고를 것

![아파치 주키퍼 서비스 예시](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%208.png)

아파치 주키퍼 서비스 예시

1. 사용자가 시스템에 로그인 시도
2. 로드밸런서가 로그인 요청을 API 서버 중 하나로 전송
3. API 서버가 사용자 인증 처리 후 서비스 탐색 기능이 동작
4. 최적의 채팅 서버 찾아서 사용자에게 반환
5. 사용자와 채팅 서버 간 웹소켓 연결

## 메시지 흐름

> 종단 간 메시지 흐름 파악
> 

### 1:1 채팅 메시지 처리 흐름

![1:1 채팅에서 메시치 처리 흐름](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%209.png)

1:1 채팅에서 메시치 처리 흐름

1. 사용자 A가 채팅 서버1로 메시지 전송
2. 채팅 서버1은 ID 생성기를 사용해 해당 메시지 ID 결정
3. 채팅 서버1은 해당 메시지를 메시지 동기화 큐로 전송
4. 메시지가 키-값 저장소에 보관
5. 사용자 B가 접속 중인 경우 메시지는 사용자 B가 접속 중인 채팅 서버로 전송,
사용자 A가 접속 중이 아니ㅏㄹ면 푸시 알림 메시지를 푸시 알림 서버로 보냄
6. 채팅 서버2는 메시지를 사용자 B에게 전송. 
사용자 B와 채팅 서버2 사이에 웹소켓 연결로 이용

### 소규모 그룹 채팅에서의 메시지 흐름

![그룹 채팅 메시지 처리 흐름](12%EC%9E%A5%20%EC%B1%84%ED%8C%85%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%2011.png)

그룹 채팅 메시지 처리 흐름

# 4단계. 마무리

---

**추가 논의 사항**

- 채팅 앱을 확장하여 사진, 비디오 등의 미디어 지원하도록 하는 방법
    - 압축 방식, 클라우드 저장소, 썸네일 생성 등
- 종단 간 암호화
    - 메시지 발신인 수신인 제외 메시지 조회 금지
- 캐시
    - 클라이언트에 이미 읽은 메시지를 캐시하면 서버와 주고받는 데이터 양 줄일 수 있음
- 로딩 속도 개선:
    - 사용자 데이터, 채널 등을 지역적으로 분산하는 네트워크 구축
    - 앱 로딩 속도 개선
- 오류 처리
    - 채팅 서버 오류: 채팅 서버 하나에 수십만 사용자가 접속한 상황
    - 서버 하나가 죽으면 서비스 탐색 기능 동작
    - 클라이언트에게 새로운 서버 배정하고 접속 가능하도록 함
    - 메시지 재전송: 재시도, 큐 이용하여 안정적 전송 보장
