## 1. 개요 (Overview)

- 분산 시스템에서는 **중복되지 않는 고유 ID(Unique Identifier)** 가 필수적이다.
- 예: 트윗 ID, 주문번호, 이미지 ID, DB 레코드 등
- 서비스별로 요구사항이 다양하다.
    - **고정된 길이의 랜덤한 고유 ID**
        
        (예: 7자 길이의 단축 URL 또는 링크)
        
    - **순차적으로 증가하는 고유 ID**
        
        (예: 데이터베이스의 자동 증가 기본 키 Auto-increment)
        
    - **고정된 길이를 가지며 정렬 가능한 고유 ID**
        
        (예: 트윗 ID, 인스타그램 이미지 ID 등)

## 2. 기존 방식과 한계 (Existing Approaches & Limitations)

### GUIDs

> 고유함이 보장되는 매우 큰 숫자
> 

```java
30dd879c-ee2f-11db-8314-0800200c9a66
```

- 일반적으로 128비트 길이. 8–4–4–4–12 형태로 그룹화된 32개의 16진수 숫자로 구성
- 단점
    - 충돌 확률이 매우 작긴 하지만 **완전히 0은 아님**
    - **크기가 너무 크다** (**128비트)**
        
        (실제 요구는 64비트 이하)
        
    - 기본적으로 생성되는 GUID(Random GUID)는 **정렬 불가**
        
        (현재 시간을 기반으로 한 **Time-based GUID**를 사용은 가능)
        

---

### MongoDB ObjectId

> 타임스탬프를 포함한 96비트의 아이디
> 
- GUID와 비슷하게 동작
- ObjectId의 구성
    
    
    | 구성 요소 | 크기 | 의미 |
    | --- | --- | --- |
    | 타임스탬프 | 4바이트 | 생성된 시간을 Unix epoch 기준으로 초 단위 저장 |
    | 랜덤 값 | 5바이트 | 임의로 생성되는 랜덤 값 |
    | 증가 카운터 | 3바이트 | 랜덤 값으로 초기화된 증가하는 카운터 |
- 시간 기반이므로 정렬 가능 (ISO 시간 값을 포함)
    
    **→ 정렬 가능**
    
- 단점
    - 크기가 너무 크다 (96비트)
    - 저장 공간 및 인덱스 크기 증가
    - 캐싱 키로 사용할 경우 비효율

---

### **Ticker Server - Flickr Ticketing Service**

> 단일 서버에서 auto-increment 기능 활용
> 

**INSERT ON DUPLICATE KEY UPDATE**

- 한 행을 원자적으로 업데이트 후 auto-increment된 기본키 반환
- 고유한 64비트(또는 32비트) ID 생성 가능
- 장점
    - ID가 **순차적으로 증가**하므로 정렬 가능
    - ID 크기를 **32-bit 또는 64-bit로 제한할 수 있다.**
- 단점
    - **단일 데이터베이스가 단일 장애 지점(Single Point of Failure)이 된다.**
        
        (데이터베이스 다운되면 전체 시스템 오류)
        
    - **확장성이 부족하다.**
        
        (매우 짧은 시간동안 요청 폭증시 쓰기 처리 한계 초과 가능)
        
- 확장성을 이루는 방법
    1. SQL 테이블을 **수평 샤딩(horizontal sharding)** 하여 여러 DB 서버를 사용
        
        예: 두 티켓 서버에 대한 요청을 RR 방식으로 분산하여 **고가용성(High Availability)**과 장애에 대비
        
        → 문제점: **각 티켓 서버에서 같은 ID가 생성될 수 있다.**
        
    2. 두 서버가 **서로 다른 형태의 ID**만 생성하도록 한다.
        
        예: 하나는 홀수, 하나는 짝수
        
        ```
        TicketServer1:
        auto-increment-increment = 2
        auto-increment-offset = 1  (홀수: 1, 3, 5 ...)
        
        TicketServer2:
        auto-increment-increment = 2
        auto-increment-offset = 2  (짝수: 2, 4, 6 ...)
        ```
        
        - 단점: **정렬 가능성을 잃는다**
            
            예를 들어,
            
            - TicketServer1에서 생성된 ID: **101**
            - TicketServer2에서 생성된 ID: **202**
            
            숫자 자체는 101 < 202 이므로 101이 먼저 생성된 것처럼 보이지만,
            
            실제로는 ID 202가 더 먼저 생성되었을 수도 있다.

## 3. 대표 솔루션: Twitter Snowflake

> **목표:** 시간 순으로 정렬 가능하고, 대규모로 생성할 수 있는 64비트 ID 생성
> 

→ 서버 간 시간이 완벽하게 동일할 수 없기 때문에,

"완전한 정렬(sorted)"이 아닌 "**대략적인 시간 정렬(roughly-sorted)**"을 목표한다.

### Snowflake의 ID 구성

```
| 1 bit | 41 bits (timestamp) | 10 bits (machine ID) | 12 bits (sequence) |
```

1. **타임스탬프(Timestamp)** — 41비트
    - 밀리초(ms) 단위 정밀도
    - 트위터가 정한 custom epoch 기준
    - 최대 69년 동안 사용 가능
2. **머신 ID(worker/machine id)** — 10비트
    - 최대 **1024대의 서버**까지 확장 가능
3. **시퀀스 번호(sequence number)** — 12비트
    - 같은 밀리초 내에서 생성되는 ID를 구분
    - 서버당 **최대 4096개**까지 생성 가능
    - 같은 밀리초에서 rollover되지 않도록 제어

+) 1 비트는 미래 확장을 위해 예약되어 있으며 기본 값은 0이다

### 장점

- 64비트 고정 길이
- 타임스탬프를 이용하여 시간 순으로 정렬 가능
- 노드 장애가 발생해도 계속 동작할 수 있는 **분산형 시스템**

### 단점

- 노드와 머신 ID 관리를 위해 **Zookeeper** 같은 시스템이 필요
- 여러 Snowflake 서버가 필요하여 **구조가 복잡해지고**, 시스템 구성 요소가 증가

## 4. Instagram의 변형 구조

> **Snowflake 구조를 확장/변형**
> 

### 변경 사항

- **41비트** → 밀리초 단위의 시간 (Instagram의 epoch 기준, 약 41년 사용 가능)
- **13비트** → 논리적 샤드 ID (logical shard ID)
- **10비트** → 자동 증가 시퀀스(auto-increment sequence), mod 1024
    
    → 즉, 각 샤드 기준으로 **밀리초당 최대 1024개의 ID 생성 가능**
    

### Instagram의 아이디 생성 과정

- 날짜: 2011년 9월 9일 오후 5시
- Instagram epoch: 2011년 1월 1일 시작
    
    (epoch 이후 경과된 시간은 **1387263000 밀리초**라 가정)
    
1. **41비트 시간값 채우기**
    
    ```
    id = 1387263000 << (64 - 41)
    ```
    
2. **샤드 ID 채우기**
    
    : 샤딩 기준이 예를 들어 user ID라고 가정하고, 논리 샤드가 2000개 있으며, user ID = 31341일 때
    
    ```
    샤드 ID = 31341 % 2000 = 1341
    id |= 1341 << (64 - 41 - 13)
    ```
    
3. **시퀀스 번호 채우기**
    
    : 지금까지 5000개의 ID를 생성했고 다음 번호는 5001이라면:
    
    ```
    id |= (5001 % 1024)
    ```
    

→ 이렇게 하면 **41비트가 timestamp로 채워지므로 ID는 시간 기준 정렬 가능**

### 장점

| 항목 | 효과 |
| --- | --- |
| 데이터 재배치 필요 없음 | 논리 샤드 단위로 다른 DB로 옮기기만 하면 됨 |
| 초기 DB 서버 수가 적어도 시작 가능 | 트래픽 증가 시 물리 서버만 추가하면 됨 |
| Zookeeper 불필요(**논리적 샤드** 개념 사용) | 머신 ID 관리 시스템 없이도 확장 가능 |

### Postgres 기반 확장 구조

> PostgreSQL의 Schema 기능(테이블을 묶는 논리적 그룹)을 적극 활용함
> 
- 하나의 DB 안에 여러 Schema 생성 가능
- 각 Schema 안에 동일한 이름의 테이블 생성 가능
- 각 Schema가 Instagram에서 말하는 **논리적 샤드** 역할

각 샤드 내부의 테이블이 자체적으로 ID를 생성하도록 설정했고,

이를 위해 PostgreSQL의 내장 언어인 **PL/PGSQL**과 **auto-increment** 기능을 활용

## 결론 - 설계 시 고려사항

- **ID 크기:** 저장공간 및 인덱스 성능 고려
- **시간 동기화:** 서버 간 클럭 차이 최소화
- **노드 관리:** 각 서버의 ID 충돌 방지
- **확장성:** 노드 수 증가 시에도 ID 생성 논리 유지
- **정렬성:** 타임스탬프 기반 ID를 통한 시간 정렬
