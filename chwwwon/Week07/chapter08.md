**301 응답과 302 응답의 차이**

- 301 Permanentely Moved
    
    : 해당 URL에 대한 HTTP 요청의 처리 책임이 영구적으로 Location 헤더에 반환된 URL로 이전됨
    
    - 브라우저는 이 응답을 캐시함
    - 추후 같은 단축 URL에 요청을 보낼 필요가 있으면 캐시된 원래 URL로 요청 전달
    - 서버 부하↓ (첫 번째 요청만 단축 URL 서버로 전송됨)
- 302 Found
    
    : 주어진 URL로의 요청이 일시적으로 Location 헤더가 지정하는 URL에 의해 처리되어야 함
    
    - 클라이언트 요청은 반드시 단축 URL 서버 → 원래 URL로 리디렉션 되어야 한다
    - 트래픽 분석에 유용 (클릭 발생률/발생 위치 추적 등)

**URL 리디렉션의 구현 - 해시 테이블 이용**

- 원래 URL = hashTable.get(단축 URL)
- 301/302 응답 Location 헤더에 원래 URL을 넣은 후 전송

### 해시 함수

---

> 원래 URL을 단축 URL로 변환하는데 사용
> 

해시함수가 계산하는 단축 URL 값을 hashValue라 지칭

**해시 값 길이**

---

- hashValue는 [0-9, a-z, A-Z]의 문자로 구성됨 (총 62개)
    - hashValue의 길이는 62$^n$ ≥ 3650억인 최솟값 n이 된다

![hashValue의 길이/해시 함수가 만들 수 있는 URL 개수 사이의 관계](8%EC%9E%A5%20URL%20%EB%8B%A8%EC%B6%95%EA%B8%B0%20%EC%84%A4%EA%B2%8C/image%203.png)

hashValue의 길이/해시 함수가 만들 수 있는 URL 개수 사이의 관계

- hashValue의 길이를 **7로 가정**한다.
- 해시 함수 구현 방법: ‘해시 후 충돌 해소’, ‘base-62 변환’

**base-62 변환**

---

> 진법 변환. 수의 표현 방식이 다른 두 시스템이 같은 수를 공유하도록 함
> 

**62진법 사용 방법**

: 총 62개의 문자를 사용하는 진법

- 0 ~ 9 = 0 ~ 9
- 10 ~ 35 = a ~ z
- 36 ~ 61 = A ~ Z

![62진법 사용 예시](8%EC%9E%A5%20URL%20%EB%8B%A8%EC%B6%95%EA%B8%B0%20%EC%84%A4%EA%B2%8C/image%206.png)

62진법 사용 예시

**해시 후 충돌 해소 vs base-62 변환**

---

| 구분 | Hash + 충돌 해결 방식 | Base 62 변환 방식 |
| --- | --- | --- |
| **짧은 URL 길이** | 짧은 URL의 길이는 고정되어 있다. | 짧은 URL의 길이는 고정되어 있지 않고, ID가 증가함에 따라 길이도 달라진다. |
| **고유 ID 생성기 필요 여부** | 고유 ID 생성기가 필요하지 않다. | 고유 ID 생성기에 의존한다. |
| **충돌 가능성** | 충돌이 발생할 수 있으며 반드시 해결해야 한다. | ID가 고유하므로 충돌이 발생하지 않는다. |
| **다음 단축 URL 예측 가능성** | ID에 의존하지 않기 때문에 다음으로 생성될 단축 URL을 예측할 수 없다. | ID가 1씩 증가하는 경우 다음 단축 URL을 쉽게 예측할 수 있다. 그러나 이는 보안상 문제가 될 수 있다. |

> 시스템의 핵심 컴포넌트. 처리 흐름이 논리적으로 단순하고 언제나 동작 상태를 유지할 것.
> 

![62진법 변환 처리 흐름 예시](8%EC%9E%A5%20URL%20%EB%8B%A8%EC%B6%95%EA%B8%B0%20%EC%84%A4%EA%B2%8C/image%207.png)

62진법 변환 처리 흐름 예시

1. 입력으로 긴 URL을 받음
2. 데이터베이스에 해당 URL이 있는지 검사
3. 데이터베이스에 있다면 해당 URL에 대한 단축 URL을 만든 적이 있는 것
    
    → 데이터베이스에서 해당 단축 URL을 가져와서 클라이언트에 반환
    
4. 데이터베이스에 없는 경우 해당 URL은 새로 접수된 것이므로 유일한 ID 생성
    
    → 이 ID를 데이터베이스의 기본 키로 사용
    
5. 62진법 변환을 적용, ID를 단축 URL로 만든다.
6. ID, 단축 URL, 원래 URL로 새 데이터베이스 레코드 만든 후 단축 URL을 클라이언트에 전달

### URL 리디렉션 상세 설계

---

![URL 리디렉션 메커니즘의 상세한 설계 예시](8%EC%9E%A5%20URL%20%EB%8B%A8%EC%B6%95%EA%B8%B0%20%EC%84%A4%EA%B2%8C/eacc6494-d541-481a-8839-64d790461d55.png)

URL 리디렉션 메커니즘의 상세한 설계 예시

- <단축 URL, 원래 URL> 쌍을 캐시에 저장하여 성능을 높임
- 로드밸런서의 동작 흐름 요약
    1. 사용자가 단축 URL을 클릭
    2. 로드밸런서가 해당 클릭으로 발생한 요청을 웹 서버에 전달
    3. 단축 URL이 이미 캐시에 있는 경우 원래 URL을 바로 꺼내서 클라이언트에 전달
    4. 캐시에 해당 단축 URL이 없는 경우 데이터베이스에서 꺼냄
        
        → 데이터베이스에 없으면 잘못된 단축 URL을 입력한 것.
        
    5. 데이터베이스에서 꺼낸 URL을 캐시에 넣은 후 사용자에게 반환

# 4단계. 마무리

---

추가 논의 사항

1. **처리율 제한 장치**
    
    : 현재 시스템은 요청이 많아지면 무력화될 수 있다(잠재적 보안 결함)
    
    → 처리율 제한 징치를 이용해 요청을 거를 수 있다. (IP 주소 등 필터링 규칙 이용)
    

1. **웹 서버의 규모 확장**
    
    : 웹 계층은 무상태 계층이므로 웹 서버의 증설/삭제가 자유롭다
    

1. **데이터베이스의 규모 확장**
    
    : 데이터베이스를 다중화/샤딩하여 규모 확장 달성 가능
    

1. **데이터 분석 솔루션**
    
    : 성공적인 비즈니스를 위해 데이터가 중요
    
    → 데이터 분석 솔루션을 통합하여 사용자가 많이 클릭한 링크, 클릭한 시간 등을 알아낼 수 있다.
