# 7장 분산 시스템을 위한 유일 ID 생성기 설계

# 1단계. 문제 이해 및 설계 범위 확정

---

> 질문을 통해 모호함을 없애고 설계 방향 정하기
> 
- ID의 특성: 유일. 정렬 가능
- ID는 숫자로만 구성
- ID는 64비트로 표현될 수 있는 값
- ID는 발급 날짜에 따라 정렬 가능

## 유일성이 보장되는 아이디를 만드는 방법

---

- 다중 마스터 복제
- UUID
- 티켓 서버
- 트위터 스노플레이크 접근법

### 1) 다중 마스터 복제

---

> 다음 아이디의 값을 k만큼 auto_increment (k = 사용 중인 데이터베이스 서버 수)
> 

![다중 마스터 복제의 구성](7%EC%9E%A5%20%EB%B6%84%EC%82%B0%20%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%84%20%EC%9C%84%ED%95%9C%20%EC%9C%A0%EC%9D%BC%20ID%20%EC%83%9D%EC%84%B1%EA%B8%B0%20%EC%84%A4%EA%B3%84/image.png)

다중 마스터 복제의 구성

**장점**

- 규모의 확장성 문제 해결 가능
    
    → 초당 생산 가능 아이디 수도 증가하므로
    

**단점**

- 여러 데이터 센터에 걸쳐 규모를 늘리기 어려움
- 아이디의 유일성 보장O / 그 값이 시간 흐름에 맞추어 커지도록 보장X
- 서버 추가, 삭제시 제대로 동작하도록 만들기 어려움

### 2) UUID

---

> 컴퓨터 시스템에 저장되는 정보를 유일하게 식별하기 위한 128비트 수
> 

![image.png](7%EC%9E%A5%20%EB%B6%84%EC%82%B0%20%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%84%20%EC%9C%84%ED%95%9C%20%EC%9C%A0%EC%9D%BC%20ID%20%EC%83%9D%EC%84%B1%EA%B8%B0%20%EC%84%A4%EA%B3%84/image%201.png)

**장점**

- 생성이 단순. 서버 사이의 조율 없이 별도 ID 생성기를 이용하여 독립적으로 생성
- 동기화 이슈↓
- 각 서버가 자신이 쓸 아이디를 알아서 만드는 구조 → 규모 확장 용이
- 충돌 가능성 매우 낮음

**단점**

- 아이디가 128비트로 길다.
- 아이디 시간순으로 정렬 불가
- 숫자 아닌 값 포함 가능

### 3) 티켓 서버

---

> auto_increment를 갖춘 중앙 집중형 티켓 서버를 **하나만** 사용
> 

![image.png](7%EC%9E%A5%20%EB%B6%84%EC%82%B0%20%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%84%20%EC%9C%84%ED%95%9C%20%EC%9C%A0%EC%9D%BC%20ID%20%EC%83%9D%EC%84%B1%EA%B8%B0%20%EC%84%A4%EA%B3%84/image%202.png)

**장점**

- 유일성이 보장되는 숫자형 아이디 생성 쉬움
- 구현이 쉬움
- 중소 규모 애플리케이션에 적합

**단점**

- 티켓 서버가 SPOF(Single Point of Failure)가 된다
    
    → 티켓 서버 장애시 전체 서버에 영향
    
- 이슈 피하기 위해 여러 티켓 서버 필요
    
    → 동기화 문제 발생 가능
    

ex) 플리커- 분산 기본 키

### 4) 트위터 스노플레이크 접근법

---

> ID의 구조를 여러 절로 분할하여 사용. **divide and conquer 기법 적용**
> 

![스노플레이크에서 사용되는 64비트 아이디 구조](7%EC%9E%A5%20%EB%B6%84%EC%82%B0%20%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%84%20%EC%9C%84%ED%95%9C%20%EC%9C%A0%EC%9D%BC%20ID%20%EC%83%9D%EC%84%B1%EA%B8%B0%20%EC%84%A4%EA%B3%84/image%203.png)

스노플레이크에서 사용되는 64비트 아이디 구조

- sign bit: 1비트 할당. 양/음수 구별. 나중을 위해
- timestamp: 41비트 할당. 기원 시각(epoch) 이후 몇 밀리초가 경과했는지 나타냄.
- datacenter ID: 5비트 할당. 2$^5$ = 32개의 데이터센터 지원 가능
- machine ID(서버 아이디): 5비트 할당. 데이터센터당 32개의 서버 이용 가능
- sequence number(일련번호): 12비트 할당. 각 서버는 ID 생성할 때 마다 1만큼 증가. 1밀리초 경과마다 0으로 초기화

# 3단계. 상세 설계

---

### 주의점

---

- datacenter ID, machine ID는 시스템 시작시 결정. 운영중 변화X
- 데이터센터나 서버 ID를 잘못 변경하게 되면 ID 충돌 발생 가능
- 타임스탬프, 일련번호는 ID 생성기 동작 중 생성

### 타임스탬프

---

![image.png](7%EC%9E%A5%20%EB%B6%84%EC%82%B0%20%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%9D%84%20%EC%9C%84%ED%95%9C%20%EC%9C%A0%EC%9D%BC%20ID%20%EC%83%9D%EC%84%B1%EA%B8%B0%20%EC%84%A4%EA%B3%84/image%204.png)

- 시간 흐름에 따라 점점 큰 값을 가짐
    
    → 시간순 정렬 가능
    
- 41비트로 표현할 수 있는 최댓값: 2$^{41}$ - 1 = 약 69년
    
    → 69년 이후에는 기원 시각 또는 ID 체계를 다른 것으로 이전해야 함
    

### 일련번호

---

- 12비트로 표현할 수 있는 값: 2$^{12}$ = 4096개
- 어떤 서버가 같은 밀리초 동안 하나 이상의 아이디를 만들어 낸 경우에만 0보다 큰 값 가짐

# 4단계. 마무리

---

추가 논의 사항 

1. 시계 동기화: 아이디 생성 서버가 모두 같은 시계를 사용하지 않는다.
    
    (여러 코어에서 실해오디는 서버에 유효X 가능성)
    
    **→ NTP를 이용할 것**
    
2. 각 절의 길이 최적화: 동시성이 낮고 수명이 긴 애플리케이션
    
    **→  일련번호 절을 줄이고 타임스탬프 절 길이를 늘리자**
    
3. 고가용성: ID 생성기는 필수 불가결 컴포넌트이므로 높은 가용성을 제공해야 함
