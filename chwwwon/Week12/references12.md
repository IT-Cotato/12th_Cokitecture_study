# **Content Popularity for Open Connect
: 콘텐츠 전달 효율을 극대화하기 위한 인기 기반 전략**

---

> **Netflix가 자체 CDN인 Open Connect에서 콘텐츠 인기(popularity)를 어떻게 예측하고 활용하는가?**
> 

### Netflix가 콘텐츠 인기도에 주목하는 이유

- Netflix는 TV 프로그램과 영화의 **인기도(popularity)** 를 매우 중요하게 관리함
- **Open Connect(Content Delivery) 팀**의 관점에
    - 콘텐츠 인기도 예측이 **CDN 인프라 효율을 극대화**하는 핵심 요소

### Open Connect와 인기도 예측의 배경

- Netflix는 자체 CDN인 **Open Connect**를 운영
    
    **→ 콘텐츠 인기도를 왜, 어떻게 예측하는지**를 더 깊이 다룸
    

### Content Delivery 관점에서의 인기도 정의

- Netflix는 인기도를 좋아요/조회수가 아닌, **실제 시청 수**로 정의함

**[ 인기도 계산 방식 ]**

Popularity = (총 스트리밍된 바이트 수) / (콘텐츠 파일 크기)

- 같은 콘텐츠가 여러 번 재생될수록 스트리밍된 총 바이트 수가 증가
    
    → 이를 통해 **시청 횟수에 가까운 지표**를 얻음
    

## 콘텐츠 인기도는 CDN 최적화에 어떻게 활용되는가?

---

### 네트워크 거리 최소화 (**Minimizing network distance)**

- Open Connect 글로벌 CDN의 서버 구성
    - **ISP 데이터센터에 물리적으로 위치한 서버(ISP servers)**
    - **IXP 데이터센터에 위치한 서버(IX servers)**
- Netflix의 목표
    
    : **가능한 한 짧은 네트워크 경로를 통해 최대한 많은 콘텐츠를 제공하는 것**
    
    - 네트워크 지연 감소 효과
    - 더 나은 스트리밍 경험 제공 가능

![image.png](12%EC%A3%BC%EC%B0%A8%20%EB%B0%9C%ED%91%9C%EC%9E%90%EB%A3%8C/image.png)

- 제한된 디스크 공간
    - 각 서버는 유한한 디스크 용량을 가짐
    - 넷플릭스 전체 콘텐츠 카탈로그는 매우 방대
    - **모든 서버 클러스터에 전체 넷플릭스 콘텐츠를 저장하는 것은 불가능**
    
    → 클러스터에 가장 인기 있는 콘텐츠만 캐싱할 것
    

### 서버 티어(server tiers) 기반 콘텐츠 구성

> 트래픽 처리가 많은 지역에서 **티어드 인프라(tiered infrastructure)** 를 사용
> 
- **고처리량 서버 (최대 100Gbps)**
    
    → 매우 인기 있는 콘텐츠 제공
    
- **대용량 스토리지 서버 (200TB 이상)**
    
    → 상대적으로 덜 인기 있는 콘텐츠(카탈로그의 tail) 제공
    
- 효율적인 운영을 위해 콘텐츠를 **인기도 기준으로 정확히 랭킹할 것**

### 클러스터 내부 콘텐츠 복제 전략

- 하나의 클러스터 내에서 각 콘텐츠(타이틀)을 **N개의 서버에 복제**
    - N은 해당 콘텐츠의 인기도에 대략적으로 비례
- 인기가 높은 콘텐츠일수록 더 많은 서버에 복제됨
- **트래픽 부하를 분산 및 병목 현상 방지 가능**

## 왜 파일을 여러 개의 복사본으로 저장하는가?

---

### 문제점

- 인기 있는 파일은 하나의 서버에만 배포 시
    - 한 서버는 과도하게 사용
    - 다른 서버는 유휴 상태로 남음

### 최적화 방법

넷플릭스는 배호 환경에서

1. **네트워크 근접성 기반 트래픽 라우팅**
    - 트래픽을 네트워크상 가까운 서버로 분산시킴
    - 가장 인기있는 콘텐츠도 네트워크 전반에 지역별 수요를 분산
2. **메모리 기반 캐싱**
    - 인기 콘텐츠는 메모리에 고정되어 제공
    - 디스크 I/O가 서버 성능 병목의 원인이 되는 상황을 제거

***하지만 Netflix는 여전히 여러 복사본을 유지한다***

### 서버 간 트래픽 편차를 최소화하여 전체 처리량 극대화

- 클러스터 내 콘텐츠 배치는 **Consistent Hashing**을 사용
- 콘텐츠를 **서버 1대에만 배치 시** 서버 간 **트래픽 불균형(variance)** 이 커질 수 있음
- 예시
    - 큰 돌(인기있는 콘텐츠) 몇 개를 여러 양동이에 나누면 → 무게가 고르지 않음
    - 작은 자갈(여러 개의 복사본)을 나누면 → 훨씬 균등하게 분산 가능

**결론: 인기 콘텐츠를 여러 서버에 복제**

- 모든 서버가 **비슷한 수준으로 부하를 받도록 유지**
- 전체 트래픽이 증가 시 각 서버가 **동시에 최대 처리량에 도달**하도록 조절

### 서버 장애 및 갑작스러운 인기 폭증에 대한 복원력 확보

- 서버 장애 발생시 해당 서버로 향하던 트래픽이 이동
    - 같은 클러스터의 다른 서버
    - 더 먼 위치의 서버로 우회
- 같은 클러스터 내 복사본이 있는 경우
    - 네트워크 거리 증가 방지
    - 스트리밍 품질 저하 최소화 가능
- Netflix의 해결책
    
    : **일부 중간 인기(mid-tier) 콘텐츠도 복제하여 갑작스러운 트래픽 증가에 대비**
    

## 콘텐츠는 어떻게 구성되어 있는가?

---

- Netflix의 모든 콘텐츠(타이틀)는 **여러 가지 포맷의 인코딩 프로파일**로 인코딩됨
- 인코딩 프로파일 종류
    - 비디오 프로파일
    - 오디오 프로파일
    - 자막을 포함한 프로파일

### 화질(비트레이트)에 따른 인코딩

- 각 오디오 및 비디오 프로파일은 **여러 수준의 화질**로 인코딩
    - 초당 사용되는 비트 수가 높을수록 더 좋은 화질 제공
- 비트레이트 결정 요소
    - 사용자의 네트워크 연결 품질
    - 사용 중인 디바이스가 지원하는 인코딩 프로파일
    - 해당 콘텐츠(타이틀)의 특성
    - 사용자가 가입한 Netflix 요금제

### 언어별 오디오 및 자막

- 오디오 트랙과 자막을 **여러 언어**로 제공

### 캐싱해야 하는 파일의 규모

- Netflix는 **각 요소의 조합 대해 하나 이상의 파일 캐싱**
    - 타이틀, 인코딩 프로파일, 비트레이트, 언어
    
    ![image.png](12%EC%A3%BC%EC%B0%A8%20%EB%B0%9C%ED%91%9C%EC%9E%90%EB%A3%8C/image%201.png)
    

→ 하나의 콘텐츠라도 매우 많은 파일이 생성

## 콘텐츠 랭킹의 효과는 어떻게 평가하는가?

---

### 캐싱 효율 (Caching Efficiency)

> 특정 트래픽 구간에 대해 해당 클러스터가 **얼마나 많은 데이터를 직접 제공했는지**를 나타내는 비율
> 
- 계산
: 해당 클러스터에서 제공된 바이트 수 / 해당 트래픽 구간 전체에서 제공된 총 바이트 수
- Netflix의 측정 관점
    - **ISP 네트워크 단위**: ISP 파트너 관점의 효율 평가
    - **클러스터 단위**: Netflix 내부 인프라 최적화 목적
- 의미
    - 사용자와 **가장 가까운 위치**에서 캐싱 효율이 높을수록
        - 네트워크 홉 수 감소
        - 스트리밍 품질 향상
        - ISP 및 Netflix의 콘텐츠 전송 비용 절감
    - 인터넷 자원을 **효율적이고 책임감 있게 사용**하는 지표

### 콘텐츠 변동성 (Content Churn)

> 하루 단위로 **얼마나 많은 콘텐츠가 새로 업데이트 되는지**를 나타내는 지표
> 
- 배경
    - Netflix는 콘텐츠 업데이트를 스트리밍 트래픽이 적은 **비혼잡 시간(off-peak)** 에 수행
    - 매일 갱신되는 데이터 양은 최소화하려고 함
- 의미
    - Content churn이 낮을수록
        - 서버 간 데이터 이동 감소
        - 캐시 업데이트 비용 절감
        - ISP 파트너와 Netflix 모두에게 비용 효율적

## 콘텐츠 인기도는 어떻게 예측하는가?

---

### 기본 접근

- 미래 시청 패턴은 **과거 시청 패턴**을 기반으로 예측
- 단순 예측:
    - “오늘 본 콘텐츠 = 내일도 본다”
    - 정확도가 낮고 변동성에 취약

### 문제점

- 콘텐츠 인기도는 **자주 변함**
- 하루 단위 반응은 과도한 캐시 교체 및 불필요한 **content churn** 유발

### Netflix의 해결 방식

- 여러 날의 시청 데이터를 **종합**하여 데이터에 **스무딩(smoothing)** 적용
- 단기 노이즈를 제거 후 **안정적인 다음 날 인기도 예측** 수행

## 인기도 예측은 어떤 단위로 수행하는가?

---

Netflix는 서로 다른 **집계 수준**에서 콘텐츠 인기도를 계산하는 여러 모델을 사용

### 타이틀 단위

> 타이틀 전체를 하나의 랭킹 단위로 처리
> 
- 장점: 스트리밍에 필요한 모든 파일이 한 서버에 존재
- 단점: 인기 없는 파일도 함께 저장 → 스토리지 비효율

### 파일 단위

> 파일 단위로 개별 인기도 랭킹
> 
- 장점: 캐싱 효율 대폭 향상
- 성과: 2016년 전환 이후 **스토리지 50% 절감 + 동일한 캐싱 효율 달성**

### 지역 단위

> 국가/지역별 시청 선호 반영
> 
- 지역 맞춤형 콘텐츠 배치 가능

## 신규 콘텐츠는 어떻게 처리하는가?

---

### 신규 콘텐츠의 문제

- 과거 시청 데이터 없음
- 기존 히스토리 기반 랭킹 불가

### Netflix의 해결 방식

1. **사전 예측 기반**
    - 내부 + 외부 데이터를 활용한 성과 예측
2. **Organic Prediction과 정규화**
    - 실제 사용자 시청 패턴을 반영해 예측 보정
3. **마케팅 영향 반영**
    - 대규모 홍보 예정 콘텐츠는 인기도 상향 조정
4. **Human Judgment**
    - 기대작은 의도적으로 랭킹 상단에 배치
    - 초기 트래픽 폭증에도 안정적 서비스 제공 목적

### 향후 과제 (Future Work)

- 인기도 예측 알고리즘&스토리지 전략 지속적으로 평가하고 개선
