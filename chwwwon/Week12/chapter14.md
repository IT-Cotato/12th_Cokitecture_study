# 14장. 유튜브 설계

비디오 스트리밍 서비스

# 1. 문제 이해 및 설계 범위 확정

### 기능

- 빠른 비디오 업로드
- 원활한 비디오 재생
- 재생 품질 선택 가능
- 낮은 인프라 비용
- 높은 가용성과 규모 확장성, 그리고 안정성
- 지원 클라이언트: 모바일 웹, 웹브라우저, 스마트 TV

### 개략적 규모 추정

- 일간 능동 사용자(DAU): 500M
- 사용자 당 하루 평균 5개의 비디오 시청
- 10%의 사용자가 하루에 1비디오 업로드
- 비디오 평균 크기는 300MB
- 비디오 저장을 위해 매일 새로 요구되는 저장 용량: 500M*10%*300MB = 150TB
- CDN 비용
    - 클라우드 CDN: 데이터 양에 따라 과금
    - 아마존 클라우드프론트: 미국 트래픽 기준 1GB당 $0.02
    - 매일 발생하는 요금: 5백만*5비디오*0.3GB*$0.02 = $150,000

비디오 업로드 절차 컴포넌트 구성 예시

- **사용자**: 유튜브를 시청하는 이용자
- **로드밸런서**: API 서버 각각으로 고르게 요청을 분산하는 역할
- **API** **서버**: 비디오 스트리밍을 제외한 다른 모든 요청 처리
- **메타데이터** **데이터베이스**: 비디오의 메타데이터 보관. 샤딩&다중화 적용하여 성능 및 가용성 요구사항 충족
- **메타데이터** **캐시**: 비디오 메타데이터와 사용자 객체 캐시하여 성능 향상
- **원본** **저장소**: BLOB 저장소 이용. 원본 비디오를 대형 이진 파일 저장소에 보관
- **트랜스코딩** **서버**: 단말, 대역폭 요구사항에 맞는 최적의 비디오 스트림 제공을 위해 필요
    - 비디오 트랜스코딩: 비디오 인코딩. 비디오 포맷을 변환
- **트랜스코딩** **비디오** **저장소**: 트랜스코딩이 완료된 비디오 저장하는 BLOB 저장소
- **CDN**: 비디오 캐시 담당. 사용자가 비디오 재생시 비디오 스트리밍 제공
- **트랜스코딩 완료 큐**: 비디오 트랜스코딩 완료 이벤트를 보관
- **트랜스코딩 완료 핸들러**: 트랜스코딩 완료 큐에서 이벤트 보관

# 3. 상세 설계

## 비디오 트랜스코딩

---

> 단말간 호환을 위해 일정한 비트레이트와 포맷으로 저장
> 
- **비트레이트**: 비디오를 구성하는 비트가 얼마나 빨리 처리되어야 하는지를 나타내는 단위
    - 비트레이트가 높으면 고화질 비디오
    - 비트레이트가 높은 비디오 스트림 재생시 높은 성능의 컴퓨터 필요. 인터넷 회신 속도↑

### **비디오 트랜스 코딩의 중요성**

- 가공되지 않은 원본 비디오는 저장 공간을 많이 차지함
    
    ex) 초당 60프레임으로 녹화된 HD 비디오는 수백 GB의 저장공간 차지
    
- 상당수의 단말과 브라우저는 특정 종류의 비디오 포맷만 지원
    
    → 호환성 문제 해결을 위해 하나의 비디오를 여러 포맷으로 인코딩해둘 것
    
- 사용자에게 끊김 없는 고화질 비디오 재생 제공을 위함
    
    → 사용자의 대역폭에 맞는 비디오 품질 제공
    
- 모바일 단말은 네트워크 상황의 영향을 받음
    
    → 비디오 화질을 자동/수동으로 변경 가능하도록 할 것
    

**비디오 인코딩 포맷**

- **컨테이너**: 비디오 파일, 오디오, 메타데이터를 담는 일종의 바구니
    - 파일 확장자를 통해 컨테이너 포맷 확인 가능
- **코덱**: 비디오 화질은 보존하면서 파일 크기를 줄이는 압축/압축해제 알고리즘

### 자원 관리자

> 자원 배분을 효과적으로 수행하는 역할 담당
> 

![image.png](14%EC%9E%A5%20%EC%9C%A0%ED%8A%9C%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%2010.png)

- **작업 큐**: 실행할 작업이 보관된 우선순위 큐
- **작업 서버 큐**: 작업 서버의 가용 상태 정보가 보관되어 있는 우선순위 큐
- **실행 큐**: 현재 실행 중인 작업 및 작업 서버 정보 보관
- **작업 스케줄러**: 최적의 작업/서버 조합을 골라 해당 작업 서버가 작업 수행하도록 지시

**[ 작업 관리자 동작 순서 ]**

1. 작업 관리자가 작업 큐에서 가장 높은 우선순위의 작업 꺼냄
2. 작업 관리자는 해당 작업 실행을 위한 작업 서버 선정
3. 작업 스케줄러가 해당 작업 서버에게 작업 실행 지시
4. 작업 스케줄러는 해당 작업이 어떤 서버에게 할당되었는지에 관한 정보를 실행 큐에 넣음
5. 작업 스케줄러는 작업이 완료되면 해당 작업을 실행 큐에서 제거

### 안전성 최적화: 미리 사인된 업로드 URL

![image.png](14%EC%9E%A5%20%EC%9C%A0%ED%8A%9C%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%2016.png)

- 허가받은 사용자만 올바른 장소에 비디오 업로드 할 수 있도록 해야 함
- 업로드 절차
    1. 클라이언트는 HTTP 서버에 POST 요청하여 미리 사인된 URL 받음
        1. 해당 URL이 가리키는 객체에 대한 접근 권한을 가진 상태
    2. API 서버는 미리 사인된 URL을 돌려줌
    3. 클라이언트는 해당 URL이 가리키는 위치에 비디오 업로드

### 안전성 최적화: 비디오 보호

- 디지털 저작권 관리 시스템 도입: 애플 페어플레이, 구글 와이드바인, 마이크로소프트 플레이레디 등
- AES 암호화: 비디오 암호화하고 접근 권한 설정
    - 암호화된 비디오는 재생 시에만 복호화 함
    - 허락된 사용자만 암호화된 비디오 시청 가능
- 워터마크: 비디오 위에 소유자 정보를 포함하는 이미지 오버레이 올리는 것
    - 회사 로고나 이름 등 사용 가능

### 오류 처리

**[ 시스템 오류 종류 ]** 

- **회복 가능 오류**
    - 특정 비디오 세그먼트를 트랜스코딩하다 실패한 경우
    - 몇 번 재시도하여 해결 가능
    - 복구 실패시 클라이언트에 적절한 오류 코드 반환
- **회보 불가능 오류**
    - 비디오 포맷이 잘못된 경우
    - 해당 비디오에 대한 작업 중한 후 클라이언트에게 적절한 오류 코드 반환

**[ 오류에 대한 해결 방안 ]** 

- **업로드 오류**: 몇 회 재시도
- **비디오 분할 오류**: 전체 비디오를 서버로 전송 후 서버가 해당 비디오 분할을 처리하도록 함
- **트랜스코딩 오류**: 재시도
- **전처리 오류**: DAG 그래프 재생성
- **DAG 스케줄러 오류**: 작업을 다시 스케줄링
- **자원 관리자 큐에 장애 발생**: 사본 이용
- **작업 서버 장애**: 다른 서버에서 해당 작업 재시도
- **API 서버 장애**: 무상태 서버에서 신규 요청은 다른 API 서버로 우회
- **메타데이터 캐시 서버 장애**: 장애 발생 캐시를 새로운 것으로 교체
- **메타데이터 데이터베이스 서버 장애**
    - 주 서버 장애: 부 서버 중 하나를 주 서버로 교체
    - 부 서버 장애: 다른 부서버를 통해 읽기 연산 처리 후 죽은 서버는 새것으로 교체
