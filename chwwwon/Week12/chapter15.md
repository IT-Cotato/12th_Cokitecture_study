# 15장. 구글 드라이브 설계

- 파일 저장 및 동기화 서비스
- 문서, 사진, 비디오, 기타 파일을 클라우드에 보관

# 1단계. 문제 이해 및 설계 범위 확정

---

### 기능적 요구사항

- 파일 추가: 파일을 구글 드라이브로 드롭
- 파일 다운로드
- 여러 단말에 파일 동기화
- 파일 갱신 이력 조회
- 파일 공유
- 파일이 편집/삭제/새로 공유 시 알림 표시

### 비기능적 요구사항

- **안정성**: 데이터 손실 발생 불가
- **빠른 동기화 속도**: 파일 동기화에 너무 많은 시간이 걸리면 안됨
- **네트워크 대역폭**: 너무 많은 대역폭을 사용하지 않도록
- **규모 확장성**: 큰 트래픽을 처리 가능해야 함
- **높은 가용성**: 일부 서버에 장애/지연/네트워크 문제 발생 시 사용 가능하도록

### 개략적 추정치

- 가입 사용자는 5천만 명. DAU는 천만 명
- 모든 사용자에게 10GB의 무료 저장공간 제공
- 매일 각 사용자가 평균 500KB의 파일 2개 씩 업로드한다 가정
- 읽기 : 쓰기 = 1 : 1
- 필요한 저장공간 총량 = 5천만 사용자 * 10GB = 500 페타바이트
- 업로드 API QPS = 1천만 사용자 * 2회 업로드 / 24h /3600s = 약 240
- 최대 QPS = QPS*2= 480

# 2단계. 개략적 설계안 제시 및 동의 구하기

## 서버 구성

---

### 서버 종류

1. 파일 업로드/다운로드 처리
2. 사용자 데이터, 로그인 정보, 파일 정보 등의 메타데이터 보관 데이터베이스
3. 파일 저장소 시스템. 파일 저장을 위해 1TB의 공간 사용

### 디렉터리 구조

- **drive/** : 업로드되는 파일을 저장
- **namespace** : 특정 사용자가 올린 파일 보관
- 각 파일과 폴더는 상대 결로를 네임스페이스 이름과 결합하여 유일하게 식별 가능

## API

---

- 기본적으로 세 종류의 API 사용
- 사용자 인증 필요
- HTTPS 프로토콜 사용
    - 클라이언트와 백엔듣 서버가 주고받는 데이터 보호를 위함

### 1. 파일 업로드 API

- 단순 업로드: 파일 크기 작을 때 사용
- 이어 올리기: 파일 사이즈↑, 네트워크 문제로 업로드 중간 가능성 있을 때 사용

**절차**

1. 이어 올리기 URL을 받기 위한 최초 요청 전송
2. 데이터를 업로드하고 업로드 상태 모니터링
3. 업로드에 장애가 발생 시 장애 발생시점부터 업로드 재시작

**예시**

`https://api.example.com/files/upload?uploadType=resumable`

- 인자
    - uploadType = resumable
    - data: 업로드할 로컬 파일

### 2. 파일 다운로드 API

**예시**

`https://api.example.com/files/download`

- 인자
    - path: 다운로드할 파일의 경로

### 3. 파일 갱신 히스토리 제공 API

**예시**

`https://api.example.com/files/list_revisions`

- 인자
    - path: 갱신 히스토리를 가져올 파일 경로
    - limit: 히스토리 최대 길이

## 개략적 설계안

---

![image.png](15%EC%9E%A5%20%EA%B5%AC%EA%B8%80%20%EB%93%9C%EB%9D%BC%EC%9D%B4%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%204.png)

- **사용자 단말**: 사용자가 이용하는 웹브라우저나 모바일 앱 등의 클라이언트
- **블록 저장소 서버**: 파일 블록을 클라우드 저장소에 업로드
    - 블록 저장소: 클라우드 환경에서 데이터 파일을 저장하는 기술
    - 파일을 여러 블록으로 나눠 저장
    - 각 블록에 고유 해시값 할당
    - 해시값을 메타데이터 데이터베이스에 저장
    - 각 블록은 독립 개체로 취급. 클라우드 저장소 시스템(S3)에 보관
- **클라우드 저장소**: 파일을 블록 단위로 나누어 저장
- **아카이빙 저장소**: 비활성 데이터 저장
- **로드밸런서**: 요청을 모든 API 서버에 고르게 분산
- **API 서버**: 파일 업로드 제외 거의 모든 것 담당 (사용자 인증, 사용자 프로파일 관리, 파일 메타데이터 갱신 등)
- **메타데이터 데이터베이스**: 메타데이터 정보 관리. 실제 파일을 클라우드에 보관
- **메타데이터 캐시**: 자주 쓰이는 메타데이터 캐시하여 성능 향상
- **알림 서비스**: 특정 이벤트 발생을 알리는 발생/구독 프로토콜 기반 시스템
- **오프라인 사용자 백업 큐**: 오프라인에서 작업시 이후 큐의 데이터를 동기화

## 높은 일관성 요구사항

---

> 기본적으로 강한 일관성 모델 지원할 것
> 
- 메타데이터 캐시/데이터베이스 계층에 동일 원칙 적용
- 메모리 캐시는 일반적으로 **최종 일관성 모델** 지원
- 강한 일관성 모델 달성 방법
    - 캐시에 보관된 사본과 데이터베이스 원본이 일치할 것
    - 데이터베이스에 보관된 원본에 변경 발생시 캐시에 있는 사본 무효화
- 관계형 데이터베이스를 채택하여 높은 일관성 요구사항에 대응

### 메타데이터 데이터베이스

![image.png](15%EC%9E%A5%20%EA%B5%AC%EA%B8%80%20%EB%93%9C%EB%9D%BC%EC%9D%B4%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%207.png)

- user: 사용자에 관계된 기본 정보 저장
- device: 반말 정보 보관. 한 사용자가 여러 대의 단말 가질 수 있음

- file_version: 파일 갱신 이력 보관. 읽기 전용
- block: 파일 블록 정보 보관

### 업로드 절차

![image.png](15%EC%9E%A5%20%EA%B5%AC%EA%B8%80%20%EB%93%9C%EB%9D%BC%EC%9D%B4%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%208.png)

**파일 메타데이터 추가**

1. 클라이언트1이 새 파일의 메타데이터를 추가하기 위한 요청 전송
2. 새 파일의 메타데이터를 데이터베이스에 저장하고 업로드 상태를 ‘대기중’으로 변경
3. 새 파일 추가시 알림 서비스에 통지
4. 알림 서비스는 관련된 클라이언트에 파일 업로드되고 있음을 알림

**파일을 클라우드 저장소에 업로드**

1. 클라이언트1이 파일을 블록 저장소 서버에 업로드
2. 블록 저장소 서버는 파일을 블록 단위로 쪼갬/압축/암호화하여 클라우드 저장소에 전송
3. 업로드 이후 클라우드 스토리지는 완료 콜백을 호출하여 API 서버로 전송
4. 메타데이터 DB에 기록된 해당 파일 상태를 완료로 변경
5. 알림 서비스에 파일 업로드가 끝났음을 통지
6. 알림 서비스는 관련된 클라이언트에 파일 업로드 종료를 알림

### 다운로드 절차

> 파일 추가/편집시 자동 시작
> 
- 알림 서비스가 파일 변경을 알림
- 오프라인시 캐시에 보관 후 접속되면 새 버전 전송

![image.png](15%EC%9E%A5%20%EA%B5%AC%EA%B8%80%20%EB%93%9C%EB%9D%BC%EC%9D%B4%EB%B8%8C%20%EC%84%A4%EA%B3%84/image%209.png)

1. 알림 서비스가 클라이언트2에 파일 변경을 알림
2. 알림을 확인한 클라이언트2가 새로운 메타데이터 요청
3. API 서버가 메타데이터 데이터베이스에 새 메타데이터 요청
4. API 서버에 새 메타데이터 반환
5. 클라이언트2에 새 메타데이터 반환
6. 클라이언트2는 새 메타데이터를 받는 즉시 블록 다운로드 요청 전송
7. 블록 저장소 서버가 클라우드 저장소에서 블록 다운로드
8. 클라우드 저장소는 블록 서버에 요청된 블록 반환
9. 블록 저장소 서버는 클라이언트에 요청된 블록 반환
10. 클라이언트2가 전송된 블록을 사용하여 파일 재구성

### 알림 서비스

> 파일 수정을 감지하면 다른 클라이언트에 알려서 충돌 가능성을 줄임
> 
- 웹소켓: 양방향 통신 수행
- **롱 폴링**
    - 단방향 통신: 파일 변경시 서버 → 클라이언트 알림 전송
    - 특정 파일에 대한 변경 감지하면 해당 연결 종료
    - 새로 연결하며 최신 내역 다운로드

### 저장소 공간 절약

> 파일 여러 버전을 여러 데이터센터에 보관할 것
> 
- **중복 제거**
    - 중복된 파일 블록을 게정 차원에서 제거
    - 해시 값 비교로 동일 블록 여부 확인
- **지능적 백업 전략 도입**
    - 한도 설정: 보관 파일 버전 개수에 상한을 두고 오래된 순으로 삭제
    - 중요 버전만 보관
- **아카이빙 저장소 사용**
    - 자주 쓰이지 않는 데이터는 아카이빙 저장소에 보관

### 장애 처리

- **로드밸런서 장애**
    - 부 로드밸런서 활성화하여 트래픽 이어받음
    - 로드밸런서 간 박동 신호로 상태 모니터링
- **블록 저장소 서버 장애**
    - 다른 서버가 미완료/대기 상태의 작업을 이어받아 처리
- **클라우드 저장소 장애**
    - S3 버킷을 여러 지역에 다중화
    - 장애 발생시 다른 지역에서 파일 가져옴
- **API 서버 장애**
    - 트래픽을 차단하여 장애 서버 격리
- **메타데이터 캐시 장애**
    - 노드에 장애 발생시 다른 노드에서 데이터 가져옴
    - 장애 발생 서버는 새 서버로 교체
- **메타데이터 데이터베이스 장애**
    - 주 데이터베이스 서버 장애: 부 데이터베이스 서버를 주 데이터베이스로 교체. 부 데이터베이스 서버 추가
    - 부 데이터베이스 서버 장애: 다른 부 데이터베이스 서버가 연산 처리
- **알림 서비스 장애**
    - 접속 중인 모든 사용자는 알림 서버와 롱 폴링 연결 유지
    - 모든 사용자가 새로운 롱 폴링 연결 필요
- **오프라인 사용자 백업 큐 장애**
