# 4장. 처리율 제한 장치의 설계

## 처리율 제한 장치

---

- 클라이언트 / 서비스가 보내는 트래픽의 처리율을 제어하기 위한 장치
    
    Ex) 특정 기간 내에 전송되는 클라이언트의 요청 횟수 제한 → 임계치 초과시 이후 모든 호출은 block
    

### 처리율 제한 장치의 장점

- DoS 공격에 의한 자원 고갈 방지
    - 추가 요청에 대한 처리 중단하여 DoS 공격을 방지함
    - 대형 IT 기업은 대부분 처리율 제한 장치 가짐
- 비용 절감
    - 많은 서버 필요X
    - 우선순위 높은 API에 더 많은 자원 할당 가능
    - third-party API 이용할 때 특히 유용
- 서버 과부하 방지
    - bot, 잘못된 이용 등에서 유발된 트래픽 차단

## 1단계. 문제 이해 및 설계 범위 확정

---

- 여러 알고리즘을 이용하여 처리율 제한 장치 구현 가능
- 면접관과 소통하며 적합한 알고리즘을 찾아라

### 예제

- 처리율이 초과하면 정확하게 제한
- 가능한 한 적은 메모리 작성
- 낮은 응답시간: HTTP 응답시간에 영향 주면 안됨
- 분산형 처리율 제한: 한 처리율 제한 장치가 여러 서버(프로세스)에서 공유 가능
- 예외 처리: 요청 제한시 사용자에게 알림
- 높은 결함 감내성: 제한 장치에 장애 발생시 시스템에 영향 X

## 2단계. 개략적 설계안 제시 및 동의 구하기

---

- 기본적인 클라이언트-서버 통신 모델을 사용해라

### 처리율 제한 장치의 위치

- 클라이언트: 안정적X. 클라이언트 요청은 위변조가 쉬워 통제 어려움
- 서버
    - API 서버
    - 처리율 제한 미들웨어 생성
        - 처리율 초과시 미들웨어가 막고 HTTP 429 반환
    
    Ex) API 게이트웨이
    
    - 완전 위탁 관리형 서비스 (클라우드 업체의 유지 보수 서비스)
    - 처리율 제한, SSL 종단, 사용자 인증, IP 허용 목록 관리 등 지원
- 고려사항
    - 기술 스택 점검: 프로그래밍 언어, 캐시 서비스 등
    - 적절한 알고리즘 선택: 제3 사업자가 제공시 선택지 제한적
    - 마이크로서비스 기반, API 게이트웨이 사용 → 처리율 제한 기능을 게이트웨이에 포함해야 함
    - 상용 API 게이트웨이 사용

### 처리율 제한 알고리즘

**토큰 버킷**

- 인터넷 기업에서 보편적으로 사용됨 (아마존, 스트라이프 등)
- 토큰 버킷: 지정된 용량을 갖는 컨테이너
- 동작
    - 버킷에 사전 설정된 양의 토큰이 추기적으로 채워짐
    - 토큰이 다 차면 더이상 추가 X (초과된 토큰은 버려짐. overflow)
    - 요청 처리 시 하나의 토큰 사용
        - 토큰이 버킷에 있으면, 요청을 시스템에 전달
        - 토큰이 버킷에 없으면, 요청 버림 (dropped)
- 알고리즘
    - 파라미터
        - 버킷 크기: 버킷에 담을 수 있는 토큰의 최대 개수
        - 토큰 공급률: 초당 몇 개의 토큰이 버킷에 공급되는가
    - API 엔드포인트마다 버킷을 가짐
    - IP 주소별로 처리율 제한 적용시 IP 주소마다 버킷 할당
    - 처리율을 초당 n개 요청으로 제한시 모든 요청은 동일한 버킷 공유
- 장점
    - 구현이 쉬움
    - 효율적인 메모리 사용
    - 트래픽 처리 가능: 버킷에 토큰이 있기만 하면 요청 처리 가능
- 단점
    - 버킷 크기, 토큰 공급률 튜닝이 까다로움

**누출 버킷**

- 요청 처리율이 고정되어 있는 알고리즘
- FIFO 큐로 구현
- 동작:
    - 요청 도착시 큐에 공간 확인
    - 공간 있으면 요청 추가
    - 공간 없으면 요청 버림
    - 지정된 시간마다 큐에서 요청 꺼내서 처리
- 알고리즘
    - 파라미터
        - 버킷 크기: 큐 사이즈. 큐에 처리될 요청을 보관
        - 처리율(overflow): 시간당 처리할 요청의 수
- 장점
    - 큐 크기 제한으로 효율적인 메모리 사용 가능
    - 고정된 처리율로 안정적 출력에 적합
- 단점
    - 단시간에 많은 트래픽 몰리면 → 오래된 요청 누적. 최신 요청 버려짐
    - 버킷 크기, 처리율 튜닝 어려움

**고정 윈도 카운터**

- 동작
    - 타임라인을 고정된 간격의 윈도로 나눔 → 윈도마다 카운터 붙임
    - 요청 접수시 카운터 값 +1
    - 카운터 값이 임계치 도달시 요청을 버림
- 장점
    - 메모리 효율 좋음
    - 이해 쉬움
    - 윈도가 닫히는 시점에 카운터를 초기화하는 방식은 특정한 트래픽 패턴 처리에 적합
- 단점
    - 위도 경계 부근에서 일식적으로 트래픽이 몰리면 처리 한도보다 초과된 요청을 처리

**이동 윈도 로그**

- 고정 윈도 카운터의 단점을 해결
- 동작
    - 요청의 타임스탬프를 추척하여 캐시에 보관
    - 새 요청 받으면 만료된 타임스탬프(현재 윈도의 시작 시점보다 오래된 타임스탬프) 제거
    - 로그 크기가 임계치 이내면 요청 전달
    - 초과시 처리 거부
- 장점
    - 정교한 처리율 제한 메커니즘
    - 허용되는 요청의 개수가 시스템의 처리율 한도를 넘지 않음
- 단점
    - 다량의 메모리 사용 (거부된 요청의 타임스탬프도 보관)

**이동 윈도 카운터**

- 고정 윈도 카운터 + 이동 윈도 로깅
- 장점
    - 평균 처리율에 따라 윈도 상태 계산하여 트래픽 대응 가능
    - 메모리 효율이 좋음
- 단점
    - 직전 시간대의 요청이 균등하다 가정후 추정하므로 느슨

### 개략적인 아키텍처

- 추적 대상 결정
    - 사용자

    - 서비스 단위

## 3단계. 상세 설계

---

- 처리율 제한 규칙은 어떻게 만들어지고 어디에 저장되는가?
- 처리가 제한된 요청들은 어떻게 처리되는가?

### **처리율 제한 규칙**

: 설정 파일 형태로 디스크에 저장

### **처리율 한도 초과 트래픽의 처리**

: 클라이언트에 HTTP 429 응답 전송. 나중에 처리하기 위해 큐에 보관

- HTTP 헤더 이용
    - 클라이언트에 HTTP 헤더 전송하여 처리율 제한 여부 알림
    - 헤더 종류
        - X-Ratelimit-Remaining: 윈도 내 남은 처리 가능 요청의 수
        - X-Ratelimit-Limit: 매 윈도마다 클라이언트가 전송할 수 있는 요청의 수
        - X-Ratelimit-Retry-After: 한도 제한에 걸리지 않기 위해 기다리는 시간
    
    → 요청 초과시 429 오류를 X-Ratelimit-Retry-After 헤더와 함께 전송

### 분산 환경에서 처리율 제한 장치 구현

: 경쟁 조건 & 동기화 문제 해결 필요

**경쟁 조건**

- 병행성이 심한 경우 발생
- 서로 다른 두 스레드가 각각 카운터 값을 변경시킨 경우 문제 발생
- 해결 방안
    - 락: 시스템 성능 저하
    - 루아 스크립트
    - 정렬 집합

**동기화 이슈**

- 여러 처리율 제한 장치 서버 사용시 동기화 필요
- stateless 웹 계층에서 처리율 제한 장치 사용시 동기화해야 올바른 제한 가능
- 해결 방안
    - 고정 세션: 한 클라이언트의 요청은 항상 동일한 처리율 제한 장치로 전송
        - 확장 불가능. 유연 X
    - 중앙 집중형 데이터 저장소 사용 (Redis 등)

**성능 최적화**

- 지연 시간 관리
    - 데이터 센터와 물리적 거리가 먼 사용자 지원시 지연 시간 증가
    - 에지 서버 이용으로 해결
    - 트래픽을 사용자와 가까운 에지 서버로 전달하여 지연시간 줄임
- 최종 일관성 모델
    - 제한 장치 간 데이터 동기화에 사용

**모니터링**

- 채택된 처리율 제한 알고리즘이 효과적인지 확인
- 정의한 처리율 제한 규칙이 효과적인지 확인

## 4단계. 마무리

---

추가 언급 사항

- **Hard / Soft 처리율 제한**
    - Hard(경성) 처리율 제한: 요청의 개수는 임계치 초과 절대 불가
    - Soft(연성) 처리율 제한: 요청 개수는 잠시 동안은 임계치 초과 가능
- **다양한 계층에서의 처리율 제한**
    - 현재 애플리케이션 계층에서의 처리율 제한만 다룸
    - IP 주소에 처리율 제한 적용 가능
- **처리율 제한 회피 방법**
    - 클라이언트 측 캐시를 사용하여 API 호출 횟수 줄임
    - 처리율 제한의 임계치를 이해하고, 짧은 시간동안 너무 많은 메시지를 보내지 않음
    - 예외나 에러 처리 코드 도입 → 예외 상황을 유연하게 처리
    - 재시도(retry) 로직 구현시 충분한 back-off 시간 부여
