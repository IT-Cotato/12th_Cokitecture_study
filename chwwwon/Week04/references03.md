## 마이크로서비스 개요

---

### 정의

하나의 애플리케이션을 여러 개의 느슨하게 결합된, 독립적으로 배포 가능한 작은 서비스들로 구성하는 클라우드 네이티브 아키텍처 접근 방식

### 특징

- 자체 기술 스택(DB, 데이터 관리 모델 등)을 보유
- REST API, 이벤트 스트리밍, 메시지 브로커 등을 이용해 서비스 간 통신 수행
- 비즈니스 역량 별 서비스 조직
    - 비즈니스 역량: 회사/서비스가 하는 일의 기능 단위 (회원, 주문, 결제 등)
- 서비스 간 경계를 명확히 정의함
    - 서비스 간 경계 구분의 중요성
        1. **서비스 간 독립성** 확보 → 서로 다른 팀이 동시에 개발/배포 가능
        2. **데이터 일관성 충돌 방지** → 각 서비스가 자기 데이터만 책임짐
        3. **확장성과 유연성** → 특정 비즈니스 기능만 따로 교체/업데이트 가능

### 비즈니스 및 조직적 이점

- 코드 업데이트 용이성: 전체 애플리케이션 수정 없이 기능/성능 추가 가능
- 서로 다른 구성 요소에 대해 서로 다른 스택/프로그래밍 언어 사용 가능
- 각 구성 요소의 독립적 확장 가능: 부하 발생시 전체 확장 부담 감소

### 기존 아키텍처와의 차이점

- **모놀리식 아키텍처 (monolithic)**
    - 모놀리식: 전체 애플리케이션이 하나의 큰 덩어리로 구성
    - 마이크로서비스: 작은 서비스 여러개가 느슨한 결합을 이룸
- **서비스 지향 아키텍처(SOA)**
    - SOA: 조직 전체의 웹 서비스가 통합됨. ESB를 통해 구현
        
        → 많은 서비스를 묶어 하나의 큰 기업 서비스 생태계로 정리하는 방법에 초점을 둠
        
        - ESB: 여러 서비스 간 통신을 중앙에서 관리하는 **중앙 허브(버스)**
    - 마이크로서비스: 애플리케이션 단위 중심 설계
        
        → 큰 앱 하나를 잘게 나눠 유연하게 운영하는 방법에 초점
        
        - 중앙 집중 허브 없이 **분산적·자율적으로 통신**

## 마이크로서비스의 장점

---

**원하는 운영 모델을 잘 지원하는 아키텍처 모델**

→ 소프트웨어 개발 팀뿐만 아니라 비즈니스 리더 또한 선호한다

### 1. 독립적 배포 가능

- 코드 수정/기능 추가 등에 복잡한 절차가 필요 없다
- cross-functional 팀 구성에 유리한 모델이다
    - cross-functional 팀: 비즈니스 문제, 서비스/제품 등을 기준으로 팀 구성
        
        → 개발자, 운영자, 기획자를 한 임에 묶어 팀이 자율적으로 서비스를 관리
        
    - **조직 구조와 아키텍처의 부합**
        - 서비스 단위가 작은 마이크로서비스와 팀 단위가 잘 맞는다
- **장애 격리**, **복원력 구축**에 유리
    - 느슨한 결합(서비스가 독립적으로 동작)으로 한 서비스의 고장이 서로 다른 서비스에 영향 X
    - 특정 서비스 고장시 장애 회복이 가능하다
    - 코드 규모가 작아 신규 팀원이 쉽게 이해 가능하다
        
        → 개발 속도, 팀 사기 측면에서 긍정적 영향
        

### 2. 기술 선택의 자유

**전통적 n-tier 아키텍처**

- 하나의 공통 스택틀 전체 애플리케이션이 공유
    - 같은 프로그래밍 언어, 프레임워크, 하나의 큰 RDBMS 사용
- 문제점
    
    : 기능별로 효율적인 기술이 달라도 동일 기술 스택을 사용한다
    
    → **비효율적인 아키텍처**가 됨
    

**마이크로서비스 모델**

- 앱을 작은 서비스로 분리
- 서비스 별 독립 배포 가능
- 장점
    
    : 서비스바다 최적화된 스택 사용 가능
    
    → 새로운 기술 도입이 쉬워 **유연하고 진화 가능한 시스템 구현** 가능
    

### 3. 독립적 스케일링

**서비스 별 독립적 확장 가능**

부하가 큰 서비스를 개별적으로 스케일링하여 인프라 낭비를 줄인다

## 마이크로서비스의 단점

---

1. **관리 복잡성 증가**
    - 모놀리식은 앱 하나만 관리하면 됐는데,
    - 마이크로서비스는 **수십, 수백 개 서비스**를 따로 관리해야 함 → 훨씬 복잡해짐.
2. **서비스 간 문제 전파**
    - 한 서비스가 문제 생기면, 연결된 다른 서비스도 같이 영향을 받을 수 있음.
3. **로그/모니터링 어려움**
    - 서비스가 많아지면 로그도 방대해짐
    - 서비스마다 로그 형식이 달라서 분석이 힘듦
4. **버전 호환성 문제**
    - 새 버전을 배포했는데, 예전 버전이랑 잘 안 맞을 수 있음
5. **네트워크 문제**
    - 서비스 간 통신이 많아지니까 → 지연(latency), 연결 오류 발생 확률도 커짐
6. **DevOps 필요성**
    - 이런 복잡성을 해결하려면 자동화/CI/CD/운영 문화(DevOps)가 필요
    - 하지만 DevOps 자체도 도입하기 쉽지 않음

**그럼에도 불구하고,**

미도입 기업의 56%는 마이크로서비스를 도입하고자 한다.

도입 기업의 78%는 마이크로서비스에 투자를 늘리고자 한다.

## 마이크로서비스와 데브옵스

---

**마이크로서비스는 데브옵스를 가능하게 할 뿐만 아니라 필수로 만든다.**

→ 복잡하고 가변 요소가 많은 마이크로서비스를 도입하려면 배포, 모니터링, 라이프사이클 자동화를 고려해야 한다.

**DevOps란?**

개발과 운영을 하나로 묶어 자동화와 협업을 통해 빠르고 안정적인 소프트웨어 배포를 가능하게 하는 문화·방법론

→ 마이크로서비스에 데브옵스를 가능하게 하는 몇가지 핵심 도구가 있다.

## 아키텍처 구성 요소 및 핵심 역량

---

### 1. Containers, Docker and Kubernetes

- 마이크로서비스는 규모가 **작다.**
    - 각 서비스를 관리하기 위해 작은 단위의 실행 환경을 필요로 함
- **Docker**
    - 컨테이너: 프로그램 + 실행환경을 묶어서 실행하는 기술
    - 기존 VM보다 작고 가볍다
        
        ↔ VM은 OS를 같이 띄워야 함
        
    - **마이크로서비스와 같은 작고 가벼운 서비스에 적합**
- **Kubernetes**
    - 컨테이너를 자동으로 관리·배치해주는 **오케스트레이션 도구**
    - 도커의 문제: 서비스가 많아지면 컨테이너도 많아진다.
        - 누가 어디에 배치하는가?
        - 언제 켜고 끄는가?
        - 장애를 어떻게 복구하는가?
    - 대규모 마이크로서비스 환경을 안정적으로 운영 가능

### 2. API gateways

- 문제 상황
    - 마이크로서비스는 서비스의 수가 많다.
    - 클라이언트가 모든 각각의 서비스 주소를 알고 요청 보내는 것이 어렵다
    - 보안 문제 발생 가능
- API gateway의 역할
    - **서비스 앞에 있는 ‘문지기’ 같은 존재**
    - **요청 라우팅**
        - 클라이언트가 게이트웨이에 요청 전달
        - 게이트웨이가 알아서 적절한 서비스로 요청을 라우팅한다.
    - **요청 분산(Fan-out)**
        - 하나의 요청을 여러 서비스로 분산할 수 있다.
    - **보안·인증**
        - 게이트웨이가 인증/권한을 체크한다
        - 내부 서비스 보호
    - **Reverse Proxy 역할**
        - 게이트웨이가 중간에서 통신을 대신 수행한다
        - 일반 프록시: 클라이언트 측에서 클라이언트를 대신함
- 구현 기술
    - API 관리 플랫폼 사용
    - **Ingress 리소스 활용** (대표적)
        - 컨테이너 + 쿠버네티스 환경에서 주로 사용
    - Istio 활용
        - 트래픽 관리, 보안, 모니터링을 포함한 최신 기술

### 3. Messaging and event streaming

- 문제 상황
    - 마이크로서비스는 stateless로 설계하는 것이 이상적이다
    - 실제로 **state 서비스가 존재한다**.
    - 단순 API 호출 방식 (= Polling)은 비효율적이다
        - 리소스 낭비 ↑
- 해결 방법
    - Messaging & Event streaming 방식 사용
    - **서비스가 직접 event를 수신** (상태 변경 시)
    - **다른 서비스는 listen 후 반응**
    
    ⇒ **Push 방식으로 상태 변경을 알린다.**
    
- 구현 기술
    1. **메시지 브로커(Message Broker)**
        - 예: RabbitMQ, ActiveMQ
        - 서비스들이 메시지를 안전하게 주고받을 수 있도록 중간에서 관리
    2. **이벤트 스트리밍(Event Streaming)**
        - 예: Apache Kafka
        - 대량의 이벤트를 **실시간으로 빠르게 처리**해야 할 때 사용
            - 실시간 로그 수집, 주문 트래킹, IoT 센서 데이터 처리 등
- 장점
    - 서비스 간 최신 상태 자동 공유
    - 실시간 처리 가능 → 대규모 시스템에도 적합
    - 장애에 강함
    - 확장 용이

### 4. Serverless

- 서비스를 더 잘게 쪼개서 함수 단위로 실행
- 특징
    1. **마이크로서비스보다 더 작은 단위로 구성**
        - 코드가 짧다
    2. **자동 스케일링**
        - 요청이 많아지면 자동으로 늘어남
        - 요청 없으면 실행 X
        - 사용한 만큼만 과금 (pay-per-user)
    3. **운영 부담 ↓**
        - 서버 관리 필요 없음 (클라우드 제공자가 다 처리)
        - 개발자는 코드만 작성

## Microservices and cloud services

---

**마이크로서비스는 클라우드와 함께 사용시 장점이 커진다.**

### 1. 비용 & 확장성

- 마이크로서비스: 작게 나눠진 서비스를 독립적으로 확장 가능
- 클라우드: 필요할 때만 자원 사용. 사용한 만큼만 요금 지불
- 결론
    - 특정 서비스에 트래픽 많으면 그 부분만 확장
    - 전체 서버 확장 X
    - 비용 절감 & 효율 극대화

### 2. 기술 스택 측면

- 클라우드는 각 기술을 관리형 서비스로 제공한다
- 복잡성 ↓
- 팀 비즈니스 로직에 집중 가능

## Common patterns

---

마이크로서비스 아키텍처의 일반적인 설계 패턴

### 1. Backend-for-Frontend (BFF) 패턴

**프론트엔드별로 최적화된 백엔드 제공**

- 기존 패턴
    - 사용자 경험 - 호출 리소스 사이에 레이어 삽입
    - 각 인터페이스 별로 화면 크기, 성능, 사용 방식이 다르다
    - 하나의 백엔드 유형을 모든 기기에 맞춤
    - 성능 & 편의성 저하 발생
- BFF 패턴
    - 프론트엔드 종류별로 전용 백엔드 생성
    - 각 백엔드가 그 프론트엔드에 최적화된 데이터 제공

### 2. Entity & Aggregate 패턴

**데이터를 개별 객체와 묶음 단위로 나눠 의미 있게 관리**

- Entity: 고유하게 식별되는 개체
- Aggregate: 관련된 엔터티들을 묶은 하나의 단위
- 데이터를 의미있는 단위로 묶어서 관리 가능
- 비즈니스 로직을 더 자연스럽게 표현 가능

### 3. Service Discovery 패턴

**서비스가 계속 변해도 서로 자동으로 찾아서 연결할 수 있게 해주는 패턴**

- 문제점
    - 마이크로서비스는 계속 바뀐다
    - 트래픽 많으면 새로운 인스턴스 생김 (scaling)
    - 업데이트/장애로 서비스가 내려갔다 다시 켜짐
    - 서비스 주소(IP/포트)가 고정되어 있지 않다
- **서비스를 레지스트리에 연결**
    - 다른 서비스 필요시 레지스트리에서 찾아서 연결한다
    - 로드밸런서가 죽은 서비스는 제외하고, 살아있는 서비스로 트래픽 분배

### 4. Adapter Microservices 패턴

**서로 안 맞는 시스템 사이를 번역해주는 중간자 역할 서비스**

- 타사 API에 의존하는 애플리케이션 사용시 유용

### 5. Strangler Application 패턴

**모놀리식을 점진적으로 마이크로서비스로 바꿔가는 전략**

- 전체 모놀리식 애플리케이션을 한번에 바꾸기 어렵다
- 기존 기능을 하나씩 마이크로서비스로 교체

## Antipatterns

---

마이크로서비스 설계 시 하지 말아야 할 것들

### 1. 처음부터 마이크로서비스로 시작하지 마라

- 작은 앱은 마이크로서비스가 필요 없다
- **복잡한 앱에서 문제 해결을 위해 도입할 것.**

### 2. DevOps나 클라우드 없이 마이크로서비스 하지 마라

- 마이크로시스템은 관리가 어렵다
- DevOps, 클라우드 없이 관리 어려움

### 3. 너무 잘게 쪼개지 마라

- 지나친 micro 추구는 관리/통신 오버헤드 증가 유발하여 손해가 더 커진다
- 필요한 상황에서만 쪼개라.
    - 배포가 느릴 때
    - 데이터 모델이 복잡해질 때
    - 부하가 부분적으로만 커질 때

### 4. 마이크로서비스를 SOA처럼 만들지 마라

- SOA는 기업 전체의 IT 서비스를 통합하는 것
- 범위를 SOA 단위로 잡으면 관리가 불가능하다
- 마이크로서비스는 **앱 단위 최적화**이다.

### 5. 넷플릭스 따라 하지 마라

- 넷플릭스의 사례
    - 인터넷 트래픽의 1/3을 차지하는 규모 → 특수 상황
    - 자체적으로 수많은 맞춤형 도구/서비스 개발 필요
- 일반적인 회사/앱 규모는 넷플릭스보다 작다
- **필요한 만큼의 상용 툴&서비스를 최대한 활용할 것**
