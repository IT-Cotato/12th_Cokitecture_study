# 10장. 알림 시스템 설계

### 알림 시스템

---

- 최근 많은 프로그램이 채택
- 고객에게 중요한 정보(최신 뉴스, 제품 업데이트, 이벤트 등)를 **비동기적으로 제공**
- 종류: 모바일 푸시 알림, SMS 메시지, 이메일 등

# 1단계. 문제 이해 및 설계 범위 확정

---

- 백만 건 이상의 알림 처리 시스템 구축을 위해 확장성 높은 시스템 필요
- 특징
    - 지원 알림 종류: 푸시 알림, SMS 메시지, 이메일
    - 연성 실시간(soft real-time) 시스템
    - 높은 부하에서 약간의 지연 허용
    - iOS, 안드로이드, 랩톱/데스크톱 단말 지원
    - 알림 스케줄링 주체: 클라이언트 애플리케이션, 서버
    - 알림 수신 여부 설정 가능
    - 하루 당 보내는 알림 수
        - 모바일 푸시 알림: 천만 건
        - SMS 메시지: 백만 건
        - 이메일: 5백만 건

## 알림 전송 및 수신 절차

---

### 개략적 설계안

---

![설계 초안 예시](10%EC%9E%A5%20%EC%95%8C%EB%A6%BC%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%206.png)

설계 초안 예시

- **1부터 N까지의 서비스**: 각 서비스는 모두 다름
    - 마이크로서비스, 크론잡, 분산 시스템 컴포넌트 등 가능
- **알림 시스템**: 알림 전송/수신 처리의 핵심
    - 서비스 1~N에 알림 전송을 위한 API 제공 필요.
    - 제3자 서비스에 전달할 알림 페이로드를 만들 수 있어야 함
- **제3자 서비스**: 사용자에게 알림을 실제로 전달하는 역할 수행
    - **확장성** 고려 필요: 새로운 서비스 통합, 기존 서비스 제거가 용이해야 함.
    - 일부 시장에서만 이용 가능. ex) 중국에서는 Jpush, PushY 등 이용
- **iOS, 안드로이드, SMS**, 이메일 단말: 사용자는 자기 단말에서 알림 수신

**문제점**

1. **SPOF**: 알림 서비스에 단일 서버 사용 
    
    → 서버 장애시 전체 시스템 장애로 이어짐
    
2. **규모 확장성**: 한 대 서비스로 푸시 알림에 관계된 모든 것 처리 
    
    → 중요 컴포넌트 규모의 개별적 확장 불가능
    
3. 성능 병목: 알림 처리/전송에 많은 자원 필요↑
    
    → 트래픽 몰리면 시스템 과부하 발생 가능
    

**개선**

1. 데이터 베이스와 캐시를 알림 시스템의 주 서버에서 분리
2. 알림 서버를 증설하고 자동으로 수평적 규모 확장을 이룸
3. 메시지 큐를 이용해 시스템 컴포넌트 사이의 강한 결합 제거

![시스템 개선안 예시](10%EC%9E%A5%20%EC%95%8C%EB%A6%BC%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%207.png)

시스템 개선안 예시

- **1부터 N까지의 서비스**: 알림 시스템 서버 API를 통해 알림 보낼 서비스
- **알림 시스템**
    - **알림 전송 API**: 스팸 방지를 위해 사내 서비스/인증된 클라이언트 주로 이용
    - **알림 검증**: 이메일 주소, 전화번호 등에 대한 기본적 검증 수행
    - **데이터베이스 또는 캐시 질의**: 알림에 포함시킬 데이터 가져오는 기능
    - **알림 전송**: 알림 데이터를 여러 메시지 큐에 넣어서 병렬 처리
    
    ![이메일 알림 POST API 예시](10%EC%9E%A5%20%EC%95%8C%EB%A6%BC%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/image%208.png)
    
    이메일 알림 POST API 예시
    
- **캐시**: 사용자 정보, 단말 정보, 알림 템플릿 등 캐시
- **데이터베이스**: 사용자, 알림, 설정 등 다양한 정보 저장
- **메시지 큐**: 시스템 컴포넌트 간 의존성 제거를 위해 사용
    - 다량 알림 전송시 버퍼 역할 수행
    - 제3자 서비스 중 하나에 장애 발생시 다른 알람은 정상 동작
- **작업 서버**: 메시지 큐에서 전송할 알림을 꺼내서 제3자 서비스에 전달
- **제3자 서비스**: 사용자에게 알림을 실제로 전달하는 역할 수행
- **iOS, 안드로이드, SMS**, 이메일 단말: 사용자는 자기 단말에서 알림 수신

**알림 전송 과정**

1. API를 호출하여 알림 서버로 알림 전송
2. 알림 서버는 사용자 정보, 단말 토큰, 알림 설정 등 메타데이터를 캐시, DB에서 가져옴
3. 알림 서버는 전송할 알림에 맞는 이벤트를 만들고 해당 이벤트를 위한 큐에 저장
    
    (ex. iOS 푸시 알림 이벤트는 iOS 푸시 알림 큐에 넣음)
    
4. 작업 서버는 메시지 큐에 알림 이벤트 꺼냄
5. 작업 서버는 알림을 제3자 서비스로 전송
6. 제3자 서비스는 사용자 단말로 알림 전송

## 안정성

---

> 분산 환경에서 필수 고려사항
> 

### 데이터 손실 방지

“어떠한 상황에서도 알림시 소실되면 안 된다”

- 알림 지연, 순서 오류 가능. **소실 불가**

→ **알림 데이터를 DB에 보관 & 재시도 매커니즘 구현 필요**

![image.png](10%EC%9E%A5%20%EC%95%8C%EB%A6%BC%20%EC%8B%9C%EC%8A%A4%ED%85%9C%20%EC%84%A4%EA%B3%84/78781fca-4e83-4f5f-b471-4649a3425661.png)

### 알림 중복 전송 방지

“분산 시스템에서 완전한 중복 방지는 불가능, 빈도를 줄이자”

- 중복 탐지 매커니즘 도임. 오류 신중히 처리
- 중복 방지 로직
    1. 알림 도착
    2. 이벤트 ID 검사하여 중복 여부 검사
    3. 중복 이벤트는 버림
    4. 중복X이면 알림 전송

## 추가 컴포넌트 및 고려사항

---

### 알림 템플릿

“알림 메시지 유사성을 고려하여 템플릿 구현”

- 대부분의 알림 메시지는 형식이 비슷함
- 파라미터, 스타일, 추적 링크를 조정하여 형식에 맞는 알람 생성

### 알림 설정

“사용자가 알림 설정을 조정할 수 있도록”

- 알림 설정 테이블에 조정 가능한 필드 추가
- 알림 전송 전 알림 설정을 확인

### 전송률 제한

“한 사용자가 받을 수 있는 알림의 빈도 제한”

- 사용자가 알림 기능을 끄지 않도록 함

“인증/승인된 클라이언트만 알림 전송 API이용 가능”

# 4단계. 마무리

---

- **안정성**: 메시지 전송 실패율을 낮추기 위해 안정적인 재시도 메커니즘 도입
- **보안**: 인증된 클라이언트만이 알림 전송 가능하도록 appKey, appSecret 메커니즘 이용
- **이벤트** **추적** **및** **모니터링**: 알림 생성 후 전송 성공까지의 과정을 추적.
    - 시스템 상태 모니터링을 위해 각 전송 단계마다 이벤트 추적하고 모니터링 가능한 시스템 통합
- **사용자** **설정**: 사용자가 알림 수신 설정 조정할 수 있도록 함
    - 알림 전송 전 해당 설정 확인하도록 설계
- **전송률 제한**: 사용자에게 알림을 전송하는 빈도 제한
