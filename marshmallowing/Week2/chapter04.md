# 4장: 처리율 제한 장치의 설계

### 처리율 제한 장치(rate limiter)

> 클라이언트 또는 서비스가 보내는 트래픽의 처리율(rate)을 제어하기 위한 장치
>
- HTTP 경우 특정 시간 내에 전송되는 클라이언트의 요청 횟수 제한
    - API 요청 횟수가 제한 장치의 임계치를 넘어서면 추가로 도달한 모든 호출은 처리가 중단(block)된다
    - 사용자는 초당 2회 이상의 새글 게시 불가
    - 같은 IP 주소로 하루에 10개 이상의 계정 생성 불가능

- 처리율 제한 장치의 장점
    - DoS(Denial of Service) 공격에 의한 자원 고갈 방지
        - Dos 공격: 짧은 시간에 엄청나게 많은 요청을 서버에 보내서, CPU·메모리·네트워크 같은 서버 자원을 다 써버리게 만드는 것
    - 비용 절감 가능
        - 서버를 많이 두지 않아도 되고, 우선순위가 높은 API에 더 많은 자원 할당 가능
        - 제3자 API에 사용료 지불할 경우, 횟수를 제한함으로써 비용 절감
    - 서버 과부하를 막음
        - 봇(bot)에서 오는 트래픽(매크로..)이나 사용자의 잘못된 이용 패턴으로 유발된 트래픽 거를 수 있음

### 1단계 문제 이해 및 설계 범위 확정

- 예시 요구사항
    - 설정된 처리율을 초과하는 요청은 정확하게 제한한다
    - 낮은 응답 시간
        - 이 처리율 제한 장치는 HTTP 응답시간에 나쁜 영향을 주어선 안된다
    - 가능한 한 작은 메모리를 써야한다
    - 분산형 처리율 제한
        - 하나의 처리율 제한 장치를 여러 서버나 프로세스에서 공유할 수 있어야 한다
    - 예외 처리
        - 요청이 제한되었을 때는 그 사실을 사용자에게 분명하게 알려준다
    - 높은 결함 감내성
        - 제한 장치에 장애가 생기더라도 전체 시스템에 영향을 주어서는 안된다

### 2단계 개략적 설계안 제시 및 동의 구하기

- **처리율 제한 장치를 어디에 둘 것인가?**
    - 클라이언트 측에 둘 경우
        - 비권장 (안정적 X)
        - 클라이언트 요청은 쉽게 위변조 가능, 통제 어려움
    - 서버 측에 둘 경우
      - 서버 측에 제한 장치를 두는 방식
      - 처리율 제한 미들웨어를 통해 API 서버로 가는 요청 통제
      - HTTP 상태코드 429 반환 (Too many requests)
        - API 서버의 처리율이 초당 2개의 요청으로 제한돼있을 시 세번째 요청은 처리율 제한 미들웨어에 의해 가로막히고 클라이언트로 HTTP 상태코드 429 반환

- **API 게이트웨이**
    - 처리율 제한, SSL 종단, 사용자 인증, IP 허용 목록 관리 등을 지원하는 완전 위탁관리형 서비스
    - 클라우드 업체가 유지 보수를 담당하는 서비스
    - 처리율 제한을 지원하는 미들웨어

### 처리율 제한 알고리즘

- **토큰 버킷 알고리즘**
    - 토큰 버킷은 지정된 용량을 갖는 컨테이너이다
        - 이 버킷에는 사전 설정된 양의 토큰이 토큰 공급기(refiller)로부터 주기적으로 채워진다
        - 토큰이 꽉 찬 버킷에는 더 이상의 토큰이 추가되지 않고 버려진다 (overflow)

    - 각 요청은 처리될 때 마다 하나의 토큰을 사용한다
        - 충분한 토큰이 있는 경우, 버킷에서 토큰 하나를 꺼낸 후 요청을 시스템에 전달한다
        - 충분한 토큰이 없는 경우, 해당 요청은 버려진다 (dropped)

    - 토큰 버킷 알고리즘의 인자
        - 버킷 크기: 버킷에 담을 수 있는 토큰의 최대 개수
        - 토큰 공급률(refill rate): 초당 몇 개의 토큰이 버킷에 공급되는가
    - 적절한 버킷 개수
        - 통상적으로 API 엔드포인트마다 별도의 버킷
            - ex) 하루에 한번 포스팅/친구 최대 150명 추가 가능/좋아요 최대 5번 ⇒ 사용자마다 3개의 버킷 필요
        - IP 주소별로 처리율 제한을 적용해야 한다면 IP 주소마다 버킷을 하나씩 할당
        - 시스템의 처리율을 초당 1000개의 요청으로 제한하고 싶을 시, 모든 요청이 하나의 버킷 공유하게 설계 (전체 시스템 차원에서 요청 수를 합산해서 관리)
    - 장점
        - 구현 쉬움
        - 메모리 사용 측면에서 효율적
        - 짧은 시간에 집중되는 트래픽 처리 가능 (버킷에 남는 토큰 있기만 하면 요청 전달됨)
    - 단점
        - 버킷 크기와 토큰 공급률이라는 두 인자를 적절하게 튜닝하는 것이 까다로움
  
- **누출 버킷 알고리즘**

    > 요청 처리율이 고정되어 있다
    >

    - FIFO 큐로 구현
    - 요청이 도착하면 큐가 가득 차 있는지 확인
    - 빈자리가 있는 경우 큐에 요청 추가, 큐가 가득차 있는 경우 새 요청을 버림
    - 지정된 시간마다 큐에서 요청을 꺼내어 처리

    - 누출 버킷 알고리즘의 인자
        - 버킷 크기: 큐 사이즈와 같은 값 (큐에는 처리될 항목들이 보관됨)
        - 처리율(outflow rate): 지정된 시간당 몇개의 항목을 처리할지 지정하는 값 (초단위)
    - 장점
        - 큐의 크기가 제한되어 있어 메모리 사용량 측면에서 효츌적
        - 고정된 처리율을 갖고 있어 안정적 출력이 필요한 경우 적합
    - 단점
        - 단시간에 많은 트래픽이 몰리는 경우 큐에는 오래된 요청들이 쌓임
        - 요청들을 제때 처리하지 못하면 최신 요청들은 버려진다
        - 두 개의 인자 적절한 튜닝 까다로움
- **고정 윈도우 카운터 알고리즘**
    - 타임라인을 고정된 간격의 윈도우로 나누고, 각 윈도우마다 카운터를 붙임
    - 요청이 접수될 때마다 이 카운터의 값은 1씩 증가
    - 이 카운터의 값이 사전에 설정된 임계치에 도달하면 새로운 요청은 새 윈도우가 열릴때 까지 버려짐


  매초 마다 열리는 윈도우에 3개 이상의 요청이 밀려오면 그 초과분은 버려짐

    <aside>
    > 💡 이 알고리즘의 가장 큰 문제는 윈도우의 경계 부근에 순간적으로 많은 트래픽이 집중될 경우 윈도우에 할당된 양보다 더 많은 요청이 처리될 수 있다는 점
    </aside>

  최대 5개 요청 허용 시스템, 매 분마다 초기화 ⇒ 경계부근 1분간 처리한 요청은 허용 한도의 2배
    - 장점
        - 메모리 효율이 좋다
        - 이해하기 쉽다
        - 윈도우가 닫히는 시점에 카운터를 초기화하는 방식은 특정한 트래픽 패턴을 처리하기에 적합
    - 단점
        - 윈도우 경계 부근에서 일시적으로 많은 트래픽이 몰려드는 경우, 처리한도 보다 많은 양의 요청 처리하게 됨

- **이동 윈도우 로깅 알고리즘**

    <aside>
    💡 고정 윈도우 카운터 알고리즘의 경계 부분 문제 극복
    </aside>

    - 요청의 타임스탬프를 추적
        - 타임스템프는 데이터는 보통 레디스의 정렬 집합(sorted set) 같은 캐시에 보관
    - 새 요청이 오면 만료된 타임스탬프는 제거됨
        - 현재 윈도우의 시작 지점보다 오래된 스탬프
    - 새 요청의 타임스탬프를 로그에 추가
    - 로그의 크기가 허용치보다 같거나 작으면 요청은 시스템에 전달, 그렇지 않을 시 처리 거부

  요청의 타임스탬프가 로그에 추가되는 모습, 분당 최대 2회 요청 처리

    - 3번째 그림의 경우 로그 추가 직후 로그의 크기는 3으로 허용 한도보다 크므로, 타임스탬프에 로그는 남지만 요청은 거부됨
    - 4번째 그림의 경우 새로운 요청이 들어온 시간 이전의 타임스탬프는 모두 만료된 값 ⇒ 로그에서 삭제, 신규 요청 시스템에 전달

    - 장점
        - 알고리즘을 구현하는 처리율 제한 메커니즘 정교
            - 허용되는 요청의 개수는 시스템의 처리율 한도를 넘지 않음
    - 단점
        - 거부된 요청의 타임스탬프도 보관하여 다량의 메모리 사용
- **이동 윈도우 카운터 알고리즘**

    > 💡 고정 윈도우 카운터 알고리즘과 이동 윈도우 로깅 알고리즘을 결합한 것
    >

  < 예시 >

    - 처리율 제한 장치의 한도는 분당 7개
    - 현재 1분의 30% 시점에 도착한 새 요청의 경우, 현재 윈도우에 몇개의 요청이 온 것으로 처리?
        - 현재 1분간 요청수 + 직전 1분간 요청 수 X 이동 윈도우와 직전 1분이 겹치는 비율

          ⇒ 현재 윈도우에 들어있는 요청은 3+5X60% = **6.5개**

          ⇒ 현재 도착한 신규 요청은 시스템으로 전달될 것, 하지만 그 직후 한도 도달하여 X

    - 장점
        - 이전 시간대의 평균 처리율에 따라 현재 윈도우의 상태를 계산하므로 짧은 시간에 몰리는 트래픽에 잘 대응
        - 메모리 효율 좋음
    - 단점
        - 직전 시간대에 도착한 요청이 균등하게 분표돼있다고 가정한 상태에서 추정치를 계산 (심각한 문제는 X)

### 개략적인 아키텍처

> 처리율 제한 알고리즘은 얼마나 많은 요청이 접수되었는지를 추적할 수 있는 카운터를 추적 대상별로 두고, 이 카운터의 값이 어떤 한도를 넘어서면 도착한 요청을 거부하는 것

- 카운터는 주로 메모리상 동작하는 캐시에서 보관
    - 빠르고 시간에 기반한 만료 정책 지원
    - ex) 레디스(메모리 기반 저장장치)의 INCR(카운터 값 1증가), EXPIRE(카운터의 타임아웃 설정) 명령어

처리율 제한 장치의 개략적 구조

1. 클라이언트가 처리율 제한 미들웨어에 요청 보냄
2. 처리율 제한 미들웨어는 레디스의 지정 버킷에서 카운터를 가져와서 한도 도달 여부 검사
    - 도달 시 요청 거부됨
    - 도달하지 않았을 시 요청은 API 서버로 전달
    - 미들웨어는 카운터의 값을 증가시킨 후 다시 레디스에 저장됨

### 3단계 상세 설계

- **처리율 제한 규칙**

    ```
    domain: messaging
    descriptors:
    	- key: message_type
    		Value: marketing
    		rate_limit:
    				unit: day
    				requests_per_unit: 5
    ```

    - 시스템이 처리가능한 마케팅 메시지의 최대치를 하루 5개로 제한
- **처리율 한도 초과 트래픽의 처리**
    - 처리율 제한 장치가 사용하는 HTTP 헤더
        - 클라이언트에서 자신의 요청이 처리율 제한에 걸렸는지 여부, 얼마나 더 많은 요청을 보낼 수 있는 지 알 수 있음
        - X-RateLimit-Remaining: 윈도우 내에 남은 처리 가능 요청의 수
        - X-Ratelimit-Limit: 매 윈도우마다 클라이언트가 전송할 수 있는 요청의 수
        - X-Ratelimit-Retry-After: 한도 제한에 걸리지 않으려면 몇 초 뒤에 요청을 다시 보내야 하는지 알림
            - 사용자가 너무 많은 요청을 보매면 429 오류를 X-Ratelimit-Retry-After 헤더와 함께 반환

    ---

    - 상세 설계
        - 처리율 제한 규칙은 디스크에 보관
            - 작업 프로세스는 수시로 규칙을 디스크에서 읽어 캐시에 저장
        - 클라이언트가 요청을 서버에 보내면 요청은 먼저 처리율 제한 미들웨어에 도달
        - 처리율 제한 미들웨어는 제한 규칙을 캐시에서, 카운터 및 마지막 요청의 타임스탬프는 레디스 캐시에서 가져옴
            - 위 값들을 근거로 해당 요청이 처리율 제한에 걸리지 않은 경우 API 서버로 보내고 제한에 걸렸다면 429 에러를 클라이언트에 보냄 (해당 요청은 버려질 수도 있고 메시지 큐에 보관될 수도 있음)

- **분산 환경에서의 처리율 제한 장치의 구현**

    > 💡분산 환경에서 여러대의 서버와 병렬 스레드을 지원하도록 시스템을 확장할 때, 경쟁 조건과 동기화라는 두 문제가 있다
    >

  ### 경쟁 조건

    - 처리율 제한 장치의 동작
        - 레디스에서 카운터의 값을 읽는다(counter)
        - counter+1 의 값이 임계치를 넘는지 본다
        - 넘지 않는다면 레디스에 보관된 카운터 값을 1만큼 증가시킨다

      병행성이 심한 환경에서는 위와 같이 경쟁 조건 이슈 발생 가능

        - counter의 원래 값: 3
        1. 두개의 요청을 처리하는 스레드가 각각 병렬로 counter 값을 읽음
        2. 둘다 요청의 처리 상태 상관하지 않고 counter에 1을 더한 값을 레디스에 기록

           ⇒ counter값은 5가 돼야 하는데 4로 되는 문제

        > 위 경쟁 조건 문제를 해결하는 데 가장 유명한 해결책은 락(lock)이다. 하지만 이는 시스템의 성능을 상당히 떨어뜨리는 문제 존재한다. 위 설계의 경우 락대신 루아 스크립트, 정렬집합(레디스의 자료구조)을 쓰는 해결책들도 있다.

    ### 동기화 이슈

    > 💡처리율 제한 장치 서버를 여러 대 두게되면 동기화가 필요하다

    - 웹 계층은 무상태(stateless)이므로 클라이언트는 왼쪽, 오른쪽 모든 방식으로 요청 보낼 수 있음
        - 동기화를 하지 않는다면 제한 장치1의 경우 클라이언트 2에 대해 아무것도 모르므로 처리율 제한을 올바르게 수행 불가능
    - 고정세션을 활용해 같은 클라이언트로부터의 요청은 항상 같은 처리율 제한 장치로 보낼 수 있다. 하지만 규모에서 확장 가능하지 않고 유연하지 않아 비추천

    > 💡레디스와 같은 중앙 집중형 데이터 저장소를 사용해라

    - 성능 최적화
        - 여러 데이터 센터를 지원하는 문제는 처리율 제한 장치에 매우 중요한 문제
            - 멀리 떨어진 사용자의 경우 지연시간 증가해 대부분의 클라우드 서비스 사업자는 곳곳에 에지 서버(edge server) 배치
            - 사용자의 트래픽을 가장 가까운 에지 서버로 전달해 지연시간 줄임
        - 제한 장치 간에 데이터를 동기화할 때 최종 일관성 모델(eventual consistenty model)을 사용하는 것 고려
    - 모니터링
        - 채택된 처리율 제한 알고리즘이 효과적인지
        - 정의된 처리율 제한 규칙이 효과적인지

### 4단계 마무리

- 경성(hard) 또는 연성(soft) 처리율 제한
    - 경성 처리율 제한: 요청의 개수는 임계치를 절대 넘을 수 없다
    - 연성 처리율 제한: 요청의 개수는 잠시 동안은 임계치를 넘을 수 있다
- 다양한 계층에서의 처리율 제한다양한
    - 이번 장에서 알아본 것은 애플리케이션 계층(HTTP, 7번 계층)에서의 처리율 제한
    - Iptables를 사용하면 IP 주소(3번 계층)에 처리율 제한 적용 가능
- 클라이언트측의 처리율 제한 회피 방법
    - 클라이언트 측 캐시를를 사용해 API 호출 횟수 줄임
    - 처리율 제한의 임계치를 이해하고 짧은 시간동안 너무 많은 메시지 보내지 않도록 함
    - 예외나 에러를 처리하는 코드 도입
    - 재시도 로직을 구현할 때는 충분한 백오프 시간을 둔다
