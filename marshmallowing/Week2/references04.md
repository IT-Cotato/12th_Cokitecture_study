# 참고문헌

https://blog.cloudflare.com/counting-things-a-lot-of-different-things/

> Cloudflare의 수백만 도메인을 지원하는 레이트 리미팅 구축기를 읽고 정리한 글입니다.
>

> 💡 Cloudflare는 모든 고객(도메인)에 대해 **경계(edge)** 수준에서 레이트 리미팅 기능을 제공하기로 했다
> 

- 백엔드 서버가 과도한 트래픽이나 계층 7(L7) 공격에 휘둘리지 않음
- 레이트 리밋 처리를 네트워크 가장자리(edge)로 분산시켜 효율과 성능 향상

## 기본 작동 방식

- 사용자는 HTTP 요청에 맞는 여러 규칙을 설정 가능
    - 로그인 실패 시도 제한, 특정 API 호출 제한 등
- 규칙과 일치하는 요청을 해당 클라이언트 IP 단위로 카운트
- 그 카운트가 임계값(threshold)을 넘으면, 이후 요청은 원본 서버(origin)에 도달하지 않고 클라이언트에게 오류 응답을 줌

→ 이는 로그인 페이지에 대한 브루트포스 공격, L7 DoS 같은 악성 트래픽을 막는 단순하지만 효과적인 방법

> 💡 Cloudflare는 **수백만 개 도메인, 수천만 개 규칙**을 동시에 처리해야 했다. 엣지에서 빠르고 정확하게 동작하는 레이트 리미터가 필요했다.


### NGINX 모듈

Cloudflare 엣지 서버는 NGINX를 사용

```nginx
http {
    limit_req_zone $binary_remote_addr zone=ratelimitzone:10m rate=15r/m;

    server {
        location /api/expensive_endpoint {
            limit_req zone=ratelimitzone;
        }
    }
}
```

- 장점: 간단하고 효율적임
- 문제: **요청이 여러 서버로 분산되면 작동 불가**
- 해결책: 중앙 저장소를 사용해야 함 → 하지만 지연(latency) 문제 + 중앙 서비스의 가용성 보장 어려움

### Anycast

Cloudflare는 Anycast 네트워크를 사용

- 전 세계 어디서든 같은 IP 주소를 광고 → 클라이언트 요청은 항상 가장 가까운 데이터센터(PoP, Point of Presence)로 라우팅
    - 즉, 한 사용자가 특정 IP로 요청을 보내면 거의 같은 PoP(같은 지역 데이터센터)에 도달
        - 같은 PoP 안에서도 서버가 여러 대 있기 때문에 새로운 TCP 연결은 그 PoP 안의 다른 서버에 도착할 수도 있음

  ⇒  PoP 내부에 독립적인 카운팅 시스템을 만들 수 있고 지연 문제와 가용성 문제 크게 줄일 수 있다


> 💡 Anycast 덕분에 트래픽이 항상 같은 데이터센터(PoP)로 들어오기 때문에, 전 세계 단위가 아니라 데이터센터 단위에서만 트래픽 제어 로직을 만들면 된다

## 카운터 저장 방식

> 💡 같은 PoP 안에서도 서버는 여러 개 → 카운터 공유 필요


- Cloudflare는 과거 SSL 세션 ID 문제(과거 연결 정보 서버간 공유 필요)의 해결책으로 PoP마다 Twemproxy 클러스터를 만들었음
    - Twemproxy = 여러 Memcached 서버를 묶어서 프록시처럼 쓰게 해주는 시스템
    - 일관 해싱(consistent hashing)
        - 클러스터 크기를 조정(서버 추가/삭제)할 때, 모든 키를 다 새로 분배하는 대신 **일부 키만 다른 서버로 이동**되도록 해줌 ⇒ 안정적인 분산 저장 가능
- 같은 구조를 레이트 리미팅에도 재활용
    - 각 서버가 DB 샤드 보유
    - 키 분산은 일관 해싱으로 안정적 관리

## 알고리즘

- **Fixed Window (고정 윈도우 카운터)**
    - 각 샘플링 기간(카운터를 세는 기준 시간 단위)마다 카운터를 증가시키고, 주가이 끝나면 카운터를 0으로 리셋
    - 구현이 간단하지만, 경계 시점에서 트래픽 스파이크 발생 가능
- **타임스탬프 저장**
    - 모든 요청의 타임스탬프 저장 후 최근 윈도우 내 요청 수 계산
    - 정확하지만 메모리/연산 비용이 큼
- **Leaky Bucket (누수 버킷 알고리즘)**
    - 요청 시 카운터 증가, 일정 속도로 감소
        - 버킷에 카운터(물)를 채우고 시간이 지나면 일정 속도로 줄어듦(누수)
    - 버킷 용량 = 허용 가능한 버스트(burst) 트래픽
    - 정확하고 자원 사용도 효율적 (Nginx 기본 모듈이 이 방식)
    - 문제점
        - 평균 속도 + 버스트 2개 파라미터 튜닝 어려움
        - memcached 단일 연산(GET, SET, INCR)만으론 구현이 어려움
- **Sliding Window Approximation (슬라이딩 윈도우 근사법) -**Cloudflare가 선택한 방식
    - 이전 윈도우와 현재 윈도우를 가중 평균하여 요청률 추정
        - ex) 1분 제한(50회) → 현재 18회(15초 진행), 이전 42회
        - `rate = 42 * ((60-15)/60) + 18 = 49.5` →  추정값 50에 근접
    - 장점
        - 트래픽 스파이크 완화
        - 설정 단순
        - 메모리 사용량 최소(카운터 2개만 필요)
        - INCR + GET 연산만으로 구현 가능
        - 정확도 높음

## 최적화

> rate 계산을 위해 매번 Memcached에 조회하면 → 대규모 L7 공격 시 성능 저하, 정상 요청도 느려짐
>
- 요청 자체는 바로 처리 → 카운터 증가는 **비동기 작업**으로 처리

  요청과 비동기적으로 카운터 증가(Asynchronous increment)
    - 요청 응답 지연 방지
- 클라이언트가 임계 초과 시, PoP 내 모든 서버에 “이 클라이언트 차단” 상태 공유
    - 서버는 그 정보만 확인 → 카운터 조회 필요 없음
- 차단 캐싱
    - 차단 시작 후, 해당 정보는 서버 메모리에 캐싱

⇒ 대규모 L7 공격 방어 가능 + 정상 요청은 지연 거의 없음

## 요약

- **문제**: 수백만 도메인/규칙, 대규모 트래픽에서도 정확한 rate limiting 필요
- **해결 전략**: Anycast 라우팅 → PoP 단위 카운팅 → Twemproxy + Memcached 클러스터
- **알고리즘**: Sliding window approximation (정확도+효율 균형)
- **최적화**: 비동기 업데이트, PoP 내부 차단 캐싱
- **성과**: 초당 40만 요청 공격 방어, 정상 사용자 영ra향 최소
