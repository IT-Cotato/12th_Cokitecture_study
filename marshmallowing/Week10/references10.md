# LINE의 Timeline - Discover, Follow

## 타임라인

- 지인 기반의 다소 닫힌 SNS
- 친구 관계를 맺은 경우에 한해서만 구독이 가능
- 타임라인은 오로지 친구 관계인 경우에만 콘텐츠를 딜리버리
- 타임라인의 공개 포스트는 전체 공개로 권한이 설정됨
    - 누구나 읽을 수 있도록 권한이 설정됐다는 것이지 모든 사람에게 딜리버리된다는 뜻은 아님

## Discover

- 타임라인: 친구 또는 Follow의 콘텐츠를 구독하는 공간
- Discover: 그동안 나에게 도달하지 못했던, 내가 관심을 가질만한 콘텐츠를 탐색하고 소비하는 공간
    - 각 사용자가 관심 있게 본 콘텐츠를 기반으로 개인화가 진행되며, 사용자가 콘텐츠를 선택하면 다시 유사 콘텐츠를 계속해서 탐색할 수 있는 사용자 경험을 제공
    - 전체 공개 포스트를 작성하면 더 많은 사람들이 내 콘텐츠를 볼 수 있는 기회를 얻게 됨

### **Discover의 기본 동작 방식**

- PicCell과 같은 이미지/영상 분석 컴포넌트와 머신 러닝 기술을 조합해서, 유사한 사용자들을 묶어 세그먼트를 나누거나 비슷한 관심사를 가진 그룹으로 나누는 등의 작업
- 타임라인이 확보한 데이터를 기반으로 학습 모델을 만들고, 그 모델을 이용해 콘텐츠를 분류하고, 사용자가 원하는 관심사를 딜리버리

### **관심사 딜리버리 사이클과 주요 컴포넌트**

![image.png](attachment:95d2730c-b685-476a-88b3-4ea3d84c94e0:image.png)

1. 사용자가 본 콘텐츠 정보와 '좋아요'나 댓글과 같은 리액션 정보를 분석할 수 있는 형태로 데이터를 쌓음

2,3,4. 데이터는 Analysis Data Storage에 모이고, 추천 시스템이 여러 가지 모델의 ML(Machine Learning)을 통해 결과를 만들어냄

- 어떤 주제로 분류할지, 어떤 콘텐츠를 추천하지 않을지 결정
1. ML을 통해 만들어진 데이터 모델과 그 결과물은 서비스에 서빙할 수 있는 형태로 변환하여, 읽기 속도가 빠른 저장소(Recommended Contents Pool)에 저장
2. 클라이언트가 Discover 내용을 요청

7, 8, 9, 10. 요청은 Discover Curator Server에 전달되며, Recommender Server를 통해 추천 콘텐츠를 얻어낸 뒤 그리드 뷰를 구성

- 내 소셜 그래프를 참조하여 이미지와 영상 등을 적절히 배치
1. 데이터가 클라이언트에게 전송되어 화면에 렌더링

**Discover Curator Server**

Recommender Server와 Contents Storage에서 콘텐츠를 가져와 Discover 그리드 뷰를 만드는 역할

- 광고나 LINE Story 등 다양한 콘텐츠를 Discover에 붙일 수 있도록 확장 가능한 형태로 구현
- 실제로 사용자의 피드를 레이아웃해서 만들고 반환

**Recommender Server**

Recommended Contents Pool의 데이터를 서비스에서 사용할 수 있도록 딜리버리하는 역할

- Recommended Contents Pool의 데이터를 관리하고 정돈하며, 각종 요구 사항에 맞게 데이터를 분류하여 저장
- 추천 시스템과 서비스 시스템 사이의 API 게이트웨이 역할

**Recommender System**

공개 포스트 중 Discover에서 추천할 만한 품질을 가진 추천 후보군인 Recommended Contents Pool을 생성

- 콘텐츠 유사도를 측정하여 유사 콘텐츠 그래프를 만들고, 서빙에 필요한 로(raw) 데이터를 생성하여 Recommended Contents Pool에 저장
- Similarity Measurement: 사용자가 과거에 관심 있게 보았던 포스트와 유사한 포스트를 추천 후보군에서 찾아 유사 콘텐츠 목록 생성
- Ranking: 사용자의 선호에 맞게 콘텐츠 정렬
- Testing: 다양한 ML 기법을 통해 Discover 추천 모델 실험

**Contents & Reaction Storage**

타임라인 포스트, 라이크나 코멘트와 같은 리액션 정보, 외부 연동 콘텐츠 등 모든 콘텐츠를 저장하는 저장소

- 모든 데이터는 영구 보관이기 때문에 대용량 데이터 시스템으로 구성

**Social Graph DB**

친구와의 Follow 관계를 기반으로 사용자를 차단하거나 피드에서 숨기는 등의 상호 설정을 고려하여 최종 구독 관계를 저장하고 서빙

- 친구 / Follow / 차단 / 숨김 / 제한 등 모든 관계를 저장하는 그래프 DB

## Follow

LINE에서 친구 외에 다른 사람의 콘텐츠를 구독할 수 있는 새로운 소셜 관계를 제공

> 과연 소셜 미디어로서의 소셜 그래프는 LINE의 친구 목록과 같을까 다를까?
>

### **소셜 그래프의 분리 또는 확장**

- 기존 LINE의 친구 그래프(friend graph)
    - 친구 추가 기반의 양방향 관계로, 메신저 기반에는 효율적이지만 타임라인 구성면에서는 폐쇄적
- 팔로우 그래프(follow graph)
    - 일방향 구독 관계
    - SNS 피드에 최적화, 개방적

![image.png](attachment:e8b83dd9-fdc6-463c-a525-af294440def6:image.png)

<aside>
💡

라인은 기존에 친구 기반으로, 이는 타임라인 구성에선 적합하지 않다. 타임라인에 맞는 소셜 그래프를 형성하기 위해 기존의 방식과 바로 분리하는 것 보다, 확장하는 방식을 선택했다. 타임라인 전용 소셜 그래프(feed graph)를 친구 그래프 + 팔로우 그래프 = 2개의 그래프가 공존하는 방식을 사용하되, 장기적으로는 타임라인만의 소셜 그래프(=팔로우 기반)를 키워서 이를 분리시킬 계획이다.

</aside>

### 관계 정보의 데이터화

![image.png](attachment:0bed2229-be8e-4ba0-904b-50a12addec24:image.png)

- 타임라인: 콘텐츠 생산자와 소비자를 이어주는 공간
- 구독관계: 친구, Follow 관계와 각종 필터링 정보를 조합
    - 타임라인의 소셜 그래프는 단순 친구가 아니라 친구 + 팔로우 + 숨김/비공개 설정(필터링)이 합쳐진 결과

![왼쪽은 친구관계, 오른쪽은 포스트 작성 엣지까지 표현한 구독관계](attachment:f34386b8-52f3-4617-81a8-32f3edad2b84:image.png)

왼쪽은 친구관계, 오른쪽은 포스트 작성 엣지까지 표현한 구독관계

<aside>
💡

초기에는 구독 관계를 단순한 키-밸류 형태로 저장해 사용했으나, 기능을 수정하거나 확장할 때 매번 키를 재설계하거나 다시 만들어야 하는 일이 많았다. 현재는 이런 관계 정보를 효과적으로 저장하고 사용하기 위해 자체 개발한 **그래프 데이터베이스**를 사용하고 있다.

</aside>

### **그래프 데이터베이스**

- 새로운 관계가 생길 때 그에 대응하는 키-밸류 셋을 정의하고 개발할 필요 없이, 존재하는 버텍스(vertex)에 새로운 에지를 연결해 확장 가능
    - **노드(vertex) + 관계(edge)**

    ```css
    A → 친구 → B → 친구 → C → 작성한 포스트
    ```

- 버텍스와 에지의 CRUD(Create, Read, Update, Delete)와 그래프 탐색 방법을 제공하므로 매번 키-밸류를 다시 만들 필요가 없음
- 그래프 DB만으로는 대규모 팬아웃, 복잡한 그래프 탐색을 감당하기 힘들어 이를 그래프 서비스 레이어를 씌움
    - 그래프 DB: 관계를 저장하는 기본 계층
    - 그래프 서비스(고성능 레이어): 그래프 DB 위에서 캐싱, pre-computation (미리 계산), batch 처리

![친구 에지를 따라가서 친구의 친구가 작성한 포스트를 찾는 것](attachment:24508658-2ce7-400e-adad-869e760e18a3:image.png)

친구 에지를 따라가서 친구의 친구가 작성한 포스트를 찾는 것

> 실제 서비스는 이런 정보를 매우 빠르게 조회할 수 있어야 하기 때문에 그래프 데이터베이스에만 의존하지는 않는다. 깊이(depth)가 깊은 그래프 탐색은 성능이 나오지 않기 때문에 각 단계별 버텍스를 빠르게 찾아서 동시에 조회하는 **병렬성**이 핵심이다.
>
