> 관계형 데이터베이스의 키는 기본적으로 auto_increment 속성으로 설정되어 있다. 하지만 분산 환경에서는 고유성을 보장할 수 없고, ID를 동기화하기 위해선 네트워크 지연 시간이 발생하게 된다.
>

### ID의 특성

- ID는 유일해야 한다
- ID는 숫자로만 구성되어야 한다
- ID는 64비트로 표현될 수 있는 값이어야 한다
- ID는 발급 날짜에 따라 정렬 가능해야 한다
- 초당 10000개의 ID를 만들 수 있어야 한다

## 분산 시스템에서 유일 ID 생성기

### 다중 마스터 복제

![image.png](attachment:bb393afc-17d5-45ca-adbf-7ff0a1b52f3f:image.png)

| DB | auto_increment_increment | auto_increment_offset | 생성 ID |
| --- | --- | --- | --- |
| A | 3 | 1 | 1, 4, 7, 10, ... |
| B | 3 | 2 | 2, 5, 8, 11, ... |
| C | 3 | 3 | 3, 6, 9, 12, ... |
- 데이터베이스의 auto_increment 기능 활용
    - 다음 ID의 값을 구할 때 1이 아닌 k만큼 증가시킴
        - k는 현재 사용 중인 데이터베이스 서버의 수
- 해당 서버가 생성한 이전 ID 값에 전체 서버의 수를 더해서 다음 아이디 생성
- 각 서버가 다른 ID 범위를 생성

⇒ 데이터베이스 수를 늘리면 초당 생산 가능 ID의 수도 늘릴 수 있기 때문에 규모 확장성 문제 어느정도 해결 가능 (서버 수를 늘리면 병렬 생성 가능한 ID 수도 늘어나기 때문에)

- 단점
    - 여러 데이터 센터에 걸쳐 규모를 늘리기 어려움
    - ID의 유일성은 보장되겠지만 그 값이 시간 흐름에 맞춰 커지도록 보장 불가
    - 서버를 추가하거나 삭제할 때도 잘 동작하도록 만들기 어려움

### UUID

- 컴퓨터 시스템에 저장되는 정보를 유일하게 식별하기 위한 128비트짜리 수
- 충돌 가능성 매우 적음
- 서버 간 조율없이 독립적으로 생성 가능

![UUID의 구조](attachment:8e7f628a-8cc9-4713-81a1-f88f8e22ad01:image.png)

UUID의 구조

- 위 그림에서 각 웹 서버는 별도의 ID 생성기를 사용해 독립적으로 ID를 만들어낸다
- 장점
    - UUID를 만드는 것은 단순하다. 서버 사이의 조율이 필요 없고 동기화 이슈도 없다
    - 각 서버가 자기가 쓸 ID를 알아서 만드는 구조이므로 규모 확장 쉬움
- 단점
    - ID가 128비트로 길다
    - ID를 시간순으로 정렬 불가능
    - ID에 숫자가 아닌 값이 포함될 수 있다

### 티켓 서버

![image.png](attachment:075b0922-2889-45cc-ab5d-7c98ebd3547f:image.png)

- auto_increment 기능을 갖춘 데이터베이스 서버, 즉 티켓 서버를 중앙 집중형으로 하나만 사용
- 장점
    - 유일성이 보장되는 오직 숫자로만 구성된 ID를 쉽게 만들 수 있음
    - 구현이 쉽고 중소 규모 애플리케이션에 적합
- 단점
    - 티켓 서버가 SPOF가 된다
        - 이를 피하기 위해선 티켓 서버를 여러 대 준비해야 하지만, 그러면 데이터 동기화 같은 새로운 문제 발생

### 트위터 스노플레이크 접근법

![devide and conquer 전략 - ID의 구조를 여러 섹션으로 분할해 생성](attachment:6eb103ff-1c6d-4216-8c3b-2899958a5be8:image.png)

devide and conquer 전략 - ID의 구조를 여러 섹션으로 분할해 생성

- 사인 비트
    - 1비트를 할당
    - 음수와 양수를 구별하는 데 사용 가능
- 타임스탬프
    - 41비트를 할당
    - 기원 시각(epoch) 이후로 몇 밀리초가 경과헀는지를 나타내는 값
        - epoch: 컴퓨터가 시간을 계산할 때의 기준 시점(시계의 0초)
- 데이터센터 ID
    - 5비트 할당
    - 32개의 데이터센터를 지원 가능
- 서버 ID
    - 5비트 할당
    - 데이터센터당 32개의 서버 사용 가능
- 일련번호
    - 12비트 할당
    - 각 서버에서는 ID를 생성할 때마다 일련번호를 1만큼 증가시킴
    - 1밀리초가 경과할 때마다 0으로 초기화(reset)

## 트위터 스노플레이크 접근법 상세 설계

- 데이터센터 ID와 서버 ID는 시스템이 시작될 때 결정되며, 일반적으로 시스템 운영 중에는 바뀌지 않음
- 타임스탬프와 일련번호는 ID 생성기가 돌고 있는 중에 만들어지는 값

### 타임스탬프

- ID를 시간순으로 정렬 가능케 함
- 41비트로 표현 가능
    - 최대값은 2^41 - 1, 대략 69년
    - 따라서 이 ID 생성기는 69년동안만 정상 동작하는데, 기원 시각을 현재에 가깝게 맞춰 오버플로우가 발생하는 시점을 늦춰 놓은 것이다
    - 69년이 지나면 기원 시각을 바꾸거나 ID 체계를 다른것으로 이전해야 함

![이진 표현 형태로부터 UTC 시각을 추출하는 예시](attachment:8c6f3b35-3c5d-4875-965e-e95bca8f1180:image.png)

이진 표현 형태로부터 UTC 시각을 추출하는 예시

- 이진 형태를 10진수로 변환 후 트위터 기원 시각(epoch)를 더함 → 그 결과로 얻어진 밀리초 값을 UTC 시각으로 변환 ⇒ 역으로, 어떠한 UTC 시각도 상술한 타임스탬프 값으로 변환 가능

### 일련 번호

- 12비트이므로, 2^12개 값 가질 수 있음
- 어떤 서버가 같은 밀리초 동안 하나 이상의 ID를 만들어 낸 경우에만 0보다 큰 값을 갖게 됨

## 마무리

- 시계 동기화
    - 이번 설계에서는 우리는 ID 생성 서버들이 전부 같은 시계 사용한다고 가정
        - 하나의 서버가 여러 코어에서 실행될 경우, 서버가 물리적으로 독립된 여러 장비에서 실행되는 경우 유효하지 않을 수 있음
        - NTP(Network Time Protocol)은 이 문제를 해결하는 가장 보편적인 수단
- 고가용성
    - ID 생성기는 필수 불가결 컴포넌트이므로 아주 높은 가용성을 제공해야 한다
- 각 섹션 길이 최적화
    - 동시성이 낮고 수명이 긴 애플리케이션이라면 일련번호 절의 길이를 줄이고 타임스탬프 절의 길이를 늘리는 것이 효과적
