## **지연 시간(Latency)**

### **지연 시간이 중요한 이유**

- **시스템 병목 파악**
    - CPU, 메모리, 디스크, 네트워크 중 어디에서 병목이 생기는지 추적 가능
    - DB 쿼리가 느릴 때 → 디스크 I/O 때문인지, 네트워크 때문인지 분리
- **아키텍처 설계 기준 제공**
    - 캐싱, 샤딩, 비동기 처리 같은 최적화 전략을 세울 때 어디서 지연이 큰지를 알아야 설계가 합리적
- **사용자 경험(UX)에 직접 영향**
    - 수백 ms 차이가 검색·결제·게임 반응 속도에 치명적

### 주요 개념 및 대표 수치

- **CPU 사이클 접근 (Cycle Time)**
    - 명령어 실행 단위.
    - 예: L1 캐시 접근은 몇 나노초(ns) 단위
- **Cache (L1, L2) 접근 지연**
    - L1 캐시 접근은 매우 짧은 지연 (몇 나노초)
    - L2 캐시는 L1보다 더 긴 지연
- **메인 메모리 접근 (RAM)**
    - 캐시 미스 시 메모리에 접근해야 함 → 지연 수십 ~ 백 ns 수준
    - 캐시 미스 한 번 = 수십~수백 사이클 낭비
- **디스크 / SSD 접근 지연**
    - SSD 랜덤 읽기는 수십 ~ 수백 마이크로초(us) 수준
    - HDD(하드디스크)는 헤드 탐색, 회전 지연 등으로 밀리초(ms) 단위
    - HDD는 여전히 밀리초 단위 → 캐싱/버퍼링이 필수
- **네트워크 지연 / 왕복 시간 (RTT)**
    - 같은 데이터센터 내 RTT: 수십 ~ 수백 마이크로초
    - 국가 간 / 전세계 간 RTT는 수십 ~ 수백 밀리초
    - **물리적 거리**가 지연의 핵심 요인
- **데이터 전송 지연 (Transmission Delay)**
    - 패킷 크기 및 대역폭에 따라 달라짐 (전송 시간 = 패킷 크기 ÷ 대역폭)


### **네트워크 지연의 원인**

- 전송 매체
    - 전송 미디어 또는 링크는 데이터가 통과할 때 지연 시간에 가장 큰 영향을 미친다
    - 광케이블 < 유선 < 무선 (보통 무선이 더 느림)
- 네트워크 트래픽 이동 거리
    - 네트워크 엔드포인트 간의 거리가 멀면 네트워크 지연 시간이 증가
- 네트워크 홉 수
    - 중간 라우터가 여러 개인 경우 데이터 패킷에 필요한 홉 수가 증가하므로 네트워크 지연 시간 증가
- 데이터 볼륨
    - 동시 데이터 볼륨이 많으면 네트워크 디바이스의 처리 용량이 제한될 수 있으므로 네트워크 지연 시간 문제 증가
- 서버 성능
    - 서버가 느리게 응답하기 때문에 데이터 통신이 지연

### 설계 관점에서의 의미

- **캐싱**
    - 지연이 큰 DB/디스크 접근을 줄여서 성능 개선
- **분산 시스템**
    - 지역별 데이터센터 운영 → 사용자와 가까운 서버에서 응답
- **비동기/큐 기반 아키텍처**
    - 사용자-facing 서비스는 빠르게 응답하고, 느린 작업은 백그라운드 처리
- **트레이드오프**
    - Throughput(대역폭) ↑ vs Latency(반응속도) ↓ 는 별개
    - 대역폭을 늘려도 지연 시간(latency)이 크면 실시간 작업(자율주행, 금융 거래 등)에 효과 없음
