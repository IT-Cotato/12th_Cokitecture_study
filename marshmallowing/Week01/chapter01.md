## 단일 서버

1.  사용자는 도메인 이름을 이용해 웹사이트에 접속 ⇒ 접속을 위해선 도메인 이름을 도메인 이름 서비스(DNS)에 질의하여 IP 주소로 변환하는 과정 필요
2.  DNS 조회 결과로 IP 주소가 반환됨 (이 IP 주소는 그림 1-2의 웹서버 주소)
3.  해당 IP 주소로 HTTP(Hypertest Transfer Protocol) 요청이 전달됨
4.  요청을 받은 웹 서버는 HTML 페이지나 JSON 형태의 응답을 반환

## 데이터베이스

> 💡 웹/모바일 트래픽 처리 서버(웹 계층)와 데이터베이스 서버(데이터 계층)를 분리하면 그 각각을 독립적으로 확장해 나갈 수 있다

### 데이터베이스의 종류

-   관계형 데이터베이스(RDBMS)
    -   MySQL, 오라클 데이터베이스, PostgreSQL
    -   자료를 테이블과 열, 칼럼으로 표현
    -   SQL을 사용하면 여러 테이블에 있는 데이터를 그 관계에 따라 조인(join)하여 합칠 수 있다
-   비-관계형 데이터베이스(NoSQL)
    -   CouchDB, Neo4j, Cassandra, HBase, Amazon DynamoDB
    -   키-값 저장소, 그래프 저장소, 칼럼 저장소, 문서 저장소로 분류 가능
    -   일반적으로 조인 연산 지원 X
    -   비-관계형 데이터베이스가 적합한 상황
        -   아주 낮은 응답 지연시간(latency)이 요구됨
        -   다루는 데이터가 비정형(unstructured)이라 관계형 데이터가 아님
        -   데이터를 직렬화(serialize)하거나 역직렬화(deserial-ize) 할 수 있기만 하면 됨
        -   아주 많응 양의 데이터를 저장할 필요가 있음

---

### 수직적 규모 확장 vs 수평적 규모 확장

-   수직적 규모 확장: **스케일업(scale up)**
    -   고사양 자원(더 좋은 CPU, 더 많은 RAM)을 추가하는 행위
-   수평정 규모 확장: **스케일 아웃(scale out)**
    -   더 많은 서버를 추가하여 성능을 개선하는 행위
-   수직적 규모 확장의 장단점
    -   장점
        -   서버로 유입되는 트래픽 양이 적을 시 적합
        -   가장 큰 장점은 단순함
    -   단점
        -   한대의 서버에 CPU나 메모리를 무한대로 증설할 수 없음
        -   장애에 대한 자동복구 방안이나 다중화 방안을 제시하지 않음 → 서버에 장애 발생 시 완전히 중단됨

> 대규모 에플리케이션 지원에는 수평적 규모 확장이 적절하다!

### 로드밸런서

> 이전 설계에서 사용자는 웹 서버에 바로 연결된다. 즉, 웹 서버가 다운되면 사용자는 웹 사이트에 접속 불가하다. 또한 너무 많은 사용자가 접속하면 웹 서버의 응답 속도가 느려지거나 서버 접속 불가능해질 수 있다. 이러한 문제를 해결하기 위해 부하 분산기 또는 로드밸런서를 도입할 수 있다.

> 부하 분산 집합(load balancing set)에 속한 웹 서버들에게 트래픽 부하를 고르게 분산하는 역할

1.  사용자는 로드밸런서의 **공개 IP 주소**(public IP address)로 접속
    -   웹 서버는 클라이언트의 접속을 직접 처리 X
2.  로드밸런서는 웹 서버와 통신하기 위해 **사설 IP 주소**(private IP address) 이용
    -   사설 IP 주소는 같은 네트워크에 속한 서버 사이의 통신에만 쓰일 수 있는 주소로, 인터넷을 통해서는 접속 불가능 ⇒ 보안성 향상

-   부하 분산 집합(현재 웹 서버 2개)의 효과
    -   서버1 다운 시 모든 트래픽은 서버2로 전송됨 ⇒ 웹 사이트 전체가 다운되는 일 방지
        -   부하를 나누기 위해 더 많은 서버 추가 가능
    -   로드밸런서가 웹사이트로 유입되는 대량의 트래픽을 분산시켜줌
        -   웹 서버 계층에 더 많은 서버를 추가하기만 하면 됨
        -   로드밸런서가 자동적으로 트래픽을 분산

---

### 데이터베이스 다중화

> 서버 사이에 주(master)-부(slave) 관계를 설정하고 데이터 원본은 주 서버에, 사본은 부 서버에 저장하는 방식

-   쓰기 연산은 마스터에서만 지원
    -   데이터베이스를 변경하는 명령어(insert, delete, update)는 주 데이터베이스로만 전달돼야 함
-   부 데이터베이스는 주 데이터베이스로부터 그 사본을 전달받으며, 읽기 연산만은 지원
-   대부분의 애플리케이션은 읽기 연산의 비중이 쓰기 연산보다 훨씬 높음 ⇒ 부 데이터베이스의 수가 주 데이터베이스 수보다 많음


-   더 나은 성능
    -   주-부 다중화 모델에서 모든 데이터 변경 연산은 주 데이터베이스 서버로만 전달되는 반면 읽기 연산은 부 데이터베이스 서버들로 분산됨 ⇒ 병렬로 처리할 수 있는 질의(query)의 수가 늘어나고 성능 향상
-   안정성
    -   데이터를 지역적으로 떨어진 여러 장소에 다중화시켜 놓을 수 있음 ⇒ 데이터베이스 서버 일부가 파괴되어도 데이터는 보존됨
-   가용성
    -   데이터를 여러 지역에 복제해 둠 ⇒ 하나의 데이터베이스 서버에 장애가 발생하더라도 다른 서버에 있는 데이터를 가져와 계속 서비스 가능

---

#### 데이터베이스 서버 가운데 하나가 다운될 경우

-   부 서버의 다운
    -   부 서버가 한대 뿐인데 다운된 경우, 읽기 연산은 한시적으로 모든 주 데이터베이스로 전달
        -   새로운 부 데이터베이스 서버가 장애 서버를 대체
    -   부 서버가 여러대인 경우 읽기 연산은 나머지 부 데이터베이스 서버들로 분산됨
        -   새로운 부 데이터베이스 서버가 장애 서버를 대체
-   주 서버의 다운
    -   한 대의 부 데이터베이스만 있는 경우 해당 부 데이터베이스 서버가 새로운 주 서버가 된다
    -   모든 데이터베이스 연산은 일시적으로 새로운 주 서버 상에서 수행됨
    -   새로운 부 서버 추가

> 💡 부 서버에 보관된 데이터가 최신 상태가 아닐 수 있기 때문에 없는 데이터는 복구 스크립트를 돌려서 추가해야 함 (다중 마스터, 원형 다중화 방식 도입 시 이러한 상황 효과적으로 대처 가능)

1.  사용자는 DNS로부터 로드밸런서의 공개 IP 주소를 받음
2.  사용자는 해당 IP 주소를 사용해 로드밸런서에 접속
3.  HTTP 요청은 서버 1이나 서버 2로 전달됨
4.  웹 서버는 사용자의 데이터를 부 데이터베이스 서버에서 읽음 / 데이터 변경 연산은 주 데이터베이스로 전달 (데이터 추가, 삭제, 갱신)

## 캐시와 콘텐츠 전송 네트워크(CDN)

> 캐시(cache)를 붙이고 정적 콘텐츠를 콘텐츠 전송 네트워크(CDN)로 옮기면 응답시간 개선 가능

### 캐시

> 값비싼 연산 결과 또는 자주 참조되는 데이터를 메모리 안에 두고, 뒤이은 요청이 보다 빨리 처리될 수 있도록 하는 저장소

> 애플리캐이션의 성능은 데이터베이스를 얼마나 자주 호출하느냐에 크게 좌우되는데, 캐시는 그런 문제를 완화 

#### 캐시 계층

-   데이터가 잠시 보관되는 곳으로 데이터베이스보다 훨씬 빠르다
-   별도의 캐시 계층을 통해 성능이 개선되며 데이터베이스의 부하를 줄일 수 있음
-   캐시 계층의 규모는 독립적으로 확장시킬 수 있음

-   캐시 우선 읽기 전략
    -   캐시에 요청 받은 데이터가 저장돼 있을 시 캐시에서 반환
    -   캐시에 없는 경우 데이터베이스 질의를 통해 데이터를 찾아 캐시에 저장한 뒤 클라이언트에 반환

---

#### 캐시 사용 시 유의할 점

-   데이터 갱신은 자주 일어나지 않지만 참조는 빈번하게 일어날 시 캐시 적합
-   캐시는 데이터를 휘발성 메모리에 보관
    -   캐시 서버 재시작 시 캐시 내 모든 데이터 사라짐
    -   영속적으로 보관할 데이터는 지속적 저장소에 보관해두어야 함
-   캐시 만료 정책 필요
    -   만료 정책이 없을 시 계속 캐시에 남음
    -   만료 기한이 너무 짧을 시 데이터베이스를 너무 자주 읽게 되고, 너무 길 시 원본과 차이 날 가능성 높아지는 문제점 존재
-   일관성 유지
    -   데이터 저장소의 원본과 캐시 내의 사본이 같은지 여부
    -   원본을 갱신하는 연산과 캐시를 갱신하는 연산이 단일 트랜잭션으로 처리되지 않을 시 일관성 깨질 수 있음
-   캐시 서버를 한 대만 둘 시 단일 장애 지점 위험성
    -   단일 장애 지점(SPOF)이란 어떤 특정 지점에서의 장애가 전체 시스템의 동작을 중단시켜버릴 수 있는 해당 지점
    -   SPOF를 피하려면 여러 지역에 걸쳐 캐시 서버 분산 필요
-   캐시 메모리
    -   너무 작을 시 데이터가 자주 캐시에서 밀려나버려(eviction) 캐시 성능 저하
    -   캐시 메모리를 과할당하여 방지 가능
-   데이터 방출 정책
    -   캐시가 가득 찰 시 기존 데이터를 내보내어야 새로운 데이터 유입 가능
    -   LRU(Least Recently Used), LFU(Least Frequently Used), FIFO(First In First Out) ..

---

### 콘텐츠 전송 네트워크(CDN)

-   **정적 콘텐츠를 전송하는 데 쓰이는 지리적으로 분산된 서버의 네트워크**
-   이미지, 비디오, CSS, JavaScript 파일 등 정적 콘텐츠 캐시
-   사용자가 웹 사이트에 방문 시, 사용자에게 가장 가까운 CDN 서버가 정적 콘텐츠 전달CDN 사용 시 사이트 로딩 시간 개선 효

1.  사용자가 이미지 URL을 이용해 logo.jpg에 접근
    -   URL의 도메인은 CDN 서비스 사업자가 제공한 것
    -   ex) [https://mysite.cloudfront.net/logo.jpg](https://mysite.cloudfront.net/logo.jpg)
2.  CDN 서버의 캐시에 해당 이미지가 없는 경우 원본(Origin) 서버에 요청해 파일 가져옴
    -   원본 서버: 웹 서버, Amazon S3 ..
3.  원본 서버가 파일을 CDN 서버에 반환
    -   응답의 HTTP 헤더에는 해당 파일의 TTL 들어있음
4.  CDN 서버가 파일을 캐시하고 사용자에게 반환
    -   이미지는 TTL에 명시된 시간까지 캐시됨

---

### CDN 사용 시 고려해야 할 사항

-   비용: CDN은 보통 제3 사업자에 의해 운영됨
-   적절한 만료 시한 설정
-   CDN 장애 대한 대처 방안
-   콘텐츠 무효화(invalidation) 방법: 아직 만료되지 않은 콘텐츠라 하더라도 CDN에서 제거 가능
    -   CDN 서비스 사업자가 제공하는 API를 이용해 콘텐츠 무효화
    -   오브젝트 버저닝(object versioning) 이용
        -   콘텐츠의 새로운 버전 지정위해 URL 마지막에 버전 번호 인자를 준다 (v=2)

---

-   정적 콘텐츠는 더이상 웹 서버를 통해 서비스 하지 않으며, CDN을 통해 제공해 성능 향상
-   캐시가 데이터베이스 부하를 감소

## 무상태(stateless) 웹 계층

### 상태 정보 의존적인 아키텍처

> 서버는 클라이언트 정보, 즉 상태를 유지하여 요청들 사이에 공유되도록 한다

-   사용자 A를 인증하기 위해 HTTP 요청은 반드시 서버 1로 전송되어야 함
    -   요청이 서버 2로 전송되면 인증 실패
        -   서버 2에 사용자 A에 관한 데이터 보관 X **→ 같은 클라이언트로부터의 요청은 항상 같은 서버로 전송되어야 한다**
    -   로드밸런서의 고정 세션 기능으로 이용가능하지만 비권장

---

### 무상태 아키텍처

> 웹 서버는 상태 정보가 필요할 경우 공유 저장소로부터 데이터를 가져온다. 따라서 상태 정보는 웹 서버로부터 물리적으로 분리되어 있다. 이러한 구조는 단순하고, 안정적이며. 규모 확장이 쉽다.

-   사용자로부터의 HTTP 요청 어떤 웹 서버든 전달 가능

-   **세션 데이터(상태 정보)를 웹 계층에서 분리하고 지속성 데이터 보관소에 저장**
    -   공유 저장소로는 관계형 데이터베이스, 캐시 시스템, NoSQL 등 가능
-   **자동 규모 확장(autoscaling)**
    -   트래픽 양에 따라 웹 서버를 자동으로 추가하거나 삭제하는 기능
    -   상태 정보가 웹 서버와 분리돼 있으므로, 트래픽 양에 따라 웹 서버를 넣거나 빼어 자동으로 규모 확장 가능

## 데이터 센터

-   지리적 라우팅(geoDNS-routing)
    -   사용자의 위치에 따라 도메인 이름을 어떤 IP 주소로 변환할 지 결정할 수 있도록 해주는 DNS 서비스
    -   가장 가까운 데이터 센터로 안내됨
        -   같은 도메인인데 사용자 위치에 따라 다른 IP 주소로 변환
        -   한국에서 `mysite.com` 조회 → `203.0.113.10` (서울 데이터센터 IP)
        -   미국에서 `mysite.com` 조회 → `198.51.100.20` (캘리포니아 데이터센터 IP)

### 다중 데이터센터 아키텍처

-   트래픽 우회
    -   올바른 데이터 센처로 트래픽을 보내는 효과적인 방법을 찾아야 함
-   데이터 동기화
    -   데이터 센터 마다 데이터베이스가 별도일 시, 장애가 자동으로 복구되어 트래픽이 다른 데이터베이스로 우회한다 해도 찾고자 하는 데이터 없을 수 있음 ⇒ 데이터를 여러 데이터센터에 다중화하는 전략
-   테스트와 배포
    -   여러 위치에서의 테스트 중요
    -   자동화된 배포 도구는 모든 데이터 센터에 동일한 서비스 설치되는 데 중요한 역할

## 메시지 큐

> 메시지의 무손실을 보장하는 비동기 통신을 지원하는 컴포넌트로 메시지의 버퍼 역할

-   생산자(입력 서비스)
    -   메시지를 만들어 메시지 큐에 발행(publish)
-   소비자
    -   큐로부터 메시지를 받아 그에 맞는 동작 수행

-   메시지 큐를 이용시 서비스 또는 서버간 결합 느슨해짐
    -   규모 확장성이 보장되야 하는 안정적 애플리케이션 구성에 효과적
    -   생산자는 소비자 프로세스가 다운되어 있어도 발행 가능, 소비자는 생산자 서버가 비가용 상태여도 메시지 수신 가능
-   생산자와 소비자 서비스의 규모 각기 독립적으로 확장 가능
    -   큐가 커지면 더 많은 작업 프로세스 추가, 비어있으면 수 줄일 수 있음

## 로그, 매트릭 그리고 자동화

-   로그
    -   에러 로그 모니터링
    -   로그를 단일 서비스로 모아주는 도구 활용 시 더욱 편리
-   메트릭
    -   사업 현황, 시스템의 현재 상태 쉽게 파악 가능
    -   호스트 단위 메트릭: 개별 서버(호스트 머신)의 리소스 사용량을 모니터링
        -   CPU, 메모리, 디스크 I/O에 관한 메트릭
    -   종합 메트릭: 서비스 계층(데이터베이스, 캐시, API 등)을 전체적으로 바라본 성능 지표
        -   데이터베이스 계층의 성능, 캐시 계층의 성능
    -   핵심 비지니스 매트칙
        -   일별 능동 사용자, 수익, 재방문
-   자동화
    -   지속적 통합을 도와주는 도구 활용

---

-   메시지 큐는 각 컴포넌트가 보다 느슨히 결합될 수 있도록 하고, 결함에 대한 내성 높임
-   로그, 모니터링. 메트릭, 자동화 장치 추가

## 데이터베이스의 규모 확장

> 저장할 데이터가 많아지면 데이터베이스에 대한 부하 증가해 데이터베이스의 규모를 확장해야 한다

### 수직적 확장

> 기존 서버에 더 많은, 또는 고성능의 자원(CPU, RAM, 디스크)을 증설하는 방법

-   데이터베이스 서버 하드웨어에는 한계가 있으므로 무한 증설 불가
    -   사용자가 계속 늘어나면 한 대 서버로는 결국 감당 불가능
-   SPOF로 인한 위험성 큼
-   고성능으로 갈수록 고비용

---

### 수평적 확장(샤딩)

> 더 많은 서버를 추가함으로써 성능 향상

-   샤딩은 대규모 데이터베이스를 샤드(Shard)라고 부르는 작은 단위로 분할하는 기술
-   모든 샤드는 같은 스키마를 쓰지만 샤드에 보관되는 데이터 사이에는 중복이 없음

-   **샤딩 키(파티션 키)**
    -   데이터가 어떻게 분산될 지 정하는 하나 이상의 칼럼으로 구성
        -   위의 예시의 경우 샤딩 키는 user\_id
    -   샤딩 키를 통해 올바른 데이터베이스에 질의를 보내어 데이터 조회나 변경을 처리하므로 효율 높임
    -   데이터를 고르게 분할 할 수 있도록 하는 게 가장 중요
-   샤딩의 문제점
    -   데이터의 재샤딩
        -   데이터가 너무 많아져 하나의 샤드로 감당 불가능, 샤드간 데이터 분포 불균등에 의해 샤드 소진 현상 발생 시 샤드 키 계산 함수를 변경해 데이터를 재배치해야한다
        -   안정 해시 기법 활용
    -   유명인사 문제(핫스팟 키 문제)
        -   특정 샤드에 질의가 집중되어 서버에 과부화가 걸리는 문제
    -   조인과 비정규화
        -   하나의 데이터베이스를 여러 샤드 서버로 쪼개고 나면, 여러 샤드에 걸친 데이터를 조인하기 어려워짐 ⇒ 데이터베이스를 비정규화하여 하나의 테이블에서 질의가 수행될 수 있도록 해야한다

## 백만 사용자, 그리고 그 이상

-   웹 계층은 무상태 계층으로
-   모든 계층에 다중화 도입
-   가능한 한 많은 데이터를 캐시할 것
-   여러 데이터 센터를 지원할 것
-   정적 콘텐츠는 CDN을 통해 서비스할 것
-   데이터 계층은 샤딩을 통해 그 규모를 확장할 것
-   각 계층은 독립적 서비스로 분할할 것
-   시스템을 지속적으로 모니터링하고 자동화 도구들을 활용할 것
