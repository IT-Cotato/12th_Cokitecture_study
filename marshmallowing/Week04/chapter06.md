> 키-값 저장소는 키-값 데이터베이스라고도 불리는 비 관계형(non-relational) 데이터베이스이다. 이 저장소에 저장되는 값은 **고유 식별자를 키로 가져야한다**. 키와 값 사이의 이런 연결 관계를 “키-값” 쌍이라 한다.
>
- 키와 값 쌍에서 키는 유일해야 한다
    - 해당 키에 상응하는 값은 키를 통해서만 접근 가능
- 키의 예시
    - 일반 텍스트 키(”last_logged_in_at”) / 해시 키(25DDEC4)
- 키-값 저장소 예시
    - 아마존 다이나모, memcached, 레디스

## 문제 이해 및 설계 범위 설정

- put(key, value): 키-값 쌍을 저장소에 저장한다
- get(key): 인자로 주어진 키에 매달린 값을 꺼낸다

---

- 키-쌍 값의 크기는 10KB 이하
- 큰 데이터를 저장할 수 있어야 한다
- 높은 가용성을 제공해야 한다
    - 시스템에 장애가 생겨도 빠른 응답 가능해야 함
- 높은 규모 확장성
    - 트래픽 양에 따라 자동으로 서버 증설/삭제 가능
- 데이터 일관성 수준은 조정 가능해야 한다
- 짧은 응답 지연 시간

### CAP 정리

<aside>
💡

데이터 일관성, 가용성, 파티션 감내라는 세 요구사항을 동시에 만족하는 분산 시스템을 설계하는 것은 불가능하다

</aside>

- **일관성(consistency)**
    - 분산 시스템에 접속하는 모든 클라이언트는 **어떤 노드에 접속했느냐에 관계없이 언제나 같은 데이터를 보게 되어야 한다**
- **가용성(availability)**
    - 분산 시스템에 접속하는 클라이언트는 **일부 노드에 장애가 발생하더라도 항상 응답을 받을 수 있어야 한다**
    - 서버가 죽어도 응답할 수 있어야 한다
- **파티션 감내(네트워크 분할 내성, partition tolerance)**
    - 파티션은 두 노드 사이에 통신 장애가 발생하였음을 의미
    - 파티션 감내는 **네트워크에 파티션이 생기더라도 시스템은 계속 동작해야 한다는 뜻**
    - 서버끼리 연결이 끊겨도 작동할 수 있어야 한다

![image.png](attachment:782226c4-285d-4a14-99b6-070ba56f3a2e:image.png)

- CP 시스템: 일관성과 파티션 감내를 지원하는 키-값 저장소
    - 가용성을 희생
- AP 시스템: 가용성과 파티션 감내를 지원하는 키-값 저장소
    - 데이터 일관성을 희생
- CA 시스템: 일관성과 가용성을 지원하는 키-값 저장소
    - 파티션 감내는 지원하지 않음
    - 그러나 통상 네트워크 장애는 피할 수 없는 일이므로, 분산 시스템은 반드시 파티션 문제를 감내할 수 있도록 설계되어야 한다. ⇒ 분산 시스템은 P(Partition tolerance)를 **무조건 고려해서 설계해야 한다**

### 이상적 설계

> 이상적 환경이라면 네트워크가 파티션되는 상황은 절대 일어나지 않을 것이다. n1에 기록된 데이터는 자동적으로 n2와 n3에 복제된다. 데이터는 일관성과 가용성도 만족된다.
>

### 실세계의 분산 시스템

> 분산 시스템은 파티션을 피할 수 없다. 그리고 파티션 문제가 발생 시 일관성과 가용성 사이에서 하나를 선택해야 한다.
>

![image.png](attachment:aafca1a4-b58d-4ce6-a221-9837f1739fbe:image.png)

- n3에 장애가 발생하여 n1 또는 n2에 기록한 데이터는 n3로 전달되지 않음
- n3에 기록되었으나 n1 및 n2로 전달되지 않은 데이터가 있다면 n1, n2는 오래된 사본 보유 중인 상태
- CP 시스템 - 세 서버 사이에 생길 수 있는 데이터 불일치 문제를 피하기 위해 n1과 n2에 대해 쓰기 연산을 중단시켜야 한다 (가용성 희생, 일관성 유지)
    - 은행권 시스템은 보통 데이터 일관성을 희생하지 않음. 온라인 뱅킹 시스템이 계좌 최신 정보를 출력하지 못하면 큰 문제기 때문이다. 네트워크 파티션 때문에 일관성이 깨질 수 있는 상황이 발생하면 해결될 때까지 오류를 반환해야 함 → ‘데이터가 달라질 위험이 생기면 차라리 응답을 안 주겠다’
- AP 시스템 - 오래된 데이터를 반환할 위험이 있더라도 계속 읽기 연산을 허용 (일관성 희생, 가용성 유지)
    - n1과 n2는 계속 쓰기 연산을 허용, 파티션 문제가 해결된 뒤에 새 데이터를 n3에 전송

### 시스템 컴포넌트

### 데이터 파티션

> 대규모 애플리케이션의 경우 데이터를 작은 파티션으로 분할한 다음 여러 대 서버에 저장한다.
>
- 데이터를 파티션 단위로 나눌 때 고려사항
    - 데이터를 여러 서버에 고르게 분산할 수 있는가
    - 노드가 추가되거나 삭제될 때 데이터의 이동을 최소화할 수 있는가

⇒ ***안정 해시(consistent hash)*** 이용한 서버 구축 추천

![image.png](attachment:d6b85d02-4de1-477b-b4d7-336eee660967:image.png)

1. 서버를 해시 링에 배치
2. 키-쌍을 어떤 서버에 저장할 지는 우선 해당 키를 같은 링 위에 배치
3. 그 지점으로부터 링을 시계 방향으로 순회하다 만나는 첫번째 서버가 해당 키-값을 저장할 서버

- 안정 해시를 이용해 데이터 파티션 시 장점
    - 규모 확장 자동화(automatic scaling): 시스템 부하에 따라 서버가 자동으로 추가되거나 삭제 되도록 만들 수 있음
    - 다양성(heterogeneity): 각 서버의 용량에 맞게 가상 노드의 수를 조정 가능 ( 고성능 서버는 더 많은 가상 노드 갖도록 설정 가능)

### 데이터 다중화

> 높은 가용성과 안정성을 확보하기 위해 데이터를 N개 서버에 비동기적으로 다중화(replication)할 필요 있다. 여기서 N은 튜닝 가능한 값을 의미.
>
- N개 서버를 선정하는 방법
    - 키를 해시 링 위에 배치한 후, 그 지점으로부터 시계 방향으로 링을 순회하면서 만나는 첫 N개 서버에 데이터 사본을 보관
    - N=3으로 설정 시 아래 예제처럼 key0는 s1, s2,s3에 저장됨

      ![image.png](attachment:e7162055-ab36-442c-887a-4148caa1ae19:293c2464-ef3d-4caa-b96e-1b13552bcbbd.png)

- 가상 노드를 사용 시 선택한 N개의 노드가 대응될 실제 물리 서버의 개수보다 작아질 수 있는 위험 존재 ⇒ 같은 물리 서버를 중복 선택하지 않도록 해야 함
    - 같은 데이터 센터에 속한 노드는 정전, 네트워크 이슈 등의 문제를 동시에 겪을 가능성 존재

<aside>
💡

안정성을 담보하기 위해 데이터의 사본은 다른 센터에 보관하고, 센터들은 고속 네트워크로 연결한다

</aside>

### 데이터 일관성

> 여러 노드에 다중화된 데이터는 적절히 동기화되어야 하는데 정족수 합의(Quorum Consensus)프로토콜을 사용하면 읽기/쓰기 연산 모두에 일관성 보장 가능한다
>
- N=사본 개수
- W=쓰기 연산에 대한 정족수
    - 쓰기 연산이 성공적으로 간주되려면 적어도 W개의 서버로부터 쓰기 연산이 성공했다는 응답 받아야 함
- R=읽기 연산에 대한 정족수
    - 읽기 연산이 성공한 것으로 간주되려면 적어도 R개의 서버로부터 응답을 받아야 함

![N=3인 경우](attachment:24b7bf20-4d92-4490-b42e-61b91fb419c7:image.png)

N=3인 경우

- W=1
    - 쓰기 연산이 성공했다고 판단하기 위해 중재자(coordinator)는 최소 한 대 서버로부터 쓰기 성공 응답을 받아야 한다는 의미
    - s1으로 부터 성공 응답 받았을 시 나머지 응답 기다릴 필요 없음
    - 중재자는 클라이언트와 노드 사이에서 프록시(proxy) 역할

<aside>
💡

W, R, N의 값을 정하는 것은 응답 지연과 데이터 일관성 사이의 타협접을 찾는 과정이다.

</aside>

- W=1 / R=1
    - 중재자는 한대의 서버로부터만 응답 받으면 되니 응답 속도 빠름
- W나 R이 1보다 큰 경우
    - 데이터의 일관성을 향상되지만 중재자의 응답 속도는 가장 느린 서버로부터의 응답을 기다려야 하므로 느려짐
- W + R >N : 강한 일관성 보장
    - 일관성을 보증할 최신 데이터를 가진 노드가 최소 하나 겹침

- 요구되는 일관성 수준에 따라 W, R, N 값 조정
    - R=1, W=N
        - 빠른 읽기 연산에 최적화된 시스템
    - W=1, R=N
        - 빠른 쓰기 연산에 최적화된 시스템
    - W+R > N
        - 강한 일관성이 보장됨 (보통 N=3, W=R=2)
    - W+R ≤ N
        - 강한 일관성이 보장되지 않음

---

### 일관성 모델

> 일관성 모델(consistency model)은 데이터 일관성의 수준을 결정
>
- 강한 일관성
    - **모든 읽기 연산은 가장 최근에 갱신된 결과를 반환한다**
    - 모든 사본에 현재 쓰기 연산의 결과가 반영될 때까지 해당 데이터에 대한 읽기/쓰기를 금지
    - 새로운 요청의 처리가 중단되기 때문에 고가용성 시스템에 적합 X
        - 다이나모, 카산드라 저장소는 결과적 일관성 선택
- 약한 일관성
    - **읽기 연산은 가장 최근에 갱신된 결과를 반환하지 못할 수 있다**
- 결과적 일관성
    - **약한 일관성의 한 형태로, 갱신 결과가 결국에는 모든 사본에 반영(동기화)되는 모델**
    - 쓰기 연산이 병렬적으로 발생하면 시스템에 저장된 값의 일관성이 깨질 수 있는데, 이 문제는 클라이언트가 해결
        - 클라이언트 측에서 데이터의 버전 정보를 활용해 일관성이 깨진 데이터 읽기 않도록 함

### 비일관성 해소 기법: 데이터 버저닝

> 데이터를 다중화하면 가용성은 높아지지만 사본 간 일관성이 깨질 가능성은 높아진다. 버저닝(versioning)과 백터 시계(vector clock)는 그 문제를 해소하기 위해 등장한 기술이다.
>
- 버저닝
    - 데이터를 변경할 때마다 해당 데이터의 새로운 버전을 만듦
    - 각 버전의 데이터는 변경 불가능(immutable)하다


![image.png](attachment:867a003c-d1d9-40a9-9cc4-54a698c81c49:image.png)

- 어떤 데이터의 사본이 노드 n1, n2에 보관되어 있음
- 이 데이터를 가져오려는 서버1, 서버2는 get(”name”) 연산의 결과 같은 값을 얻음

  ![image.png](attachment:aadf6e0c-8243-4cc6-935e-e0c5d3c69bb8:image.png)

- 서버1, 서버2가 각각 name에 매달린 값을 변경하는 연산이 동시에 이루어짐
- 충돌하는 두 값을 갖게 되었고 그 각각을 버전 v1, v2라 함

<aside>
💡

두 버전 사이의 충돌을 발견하고 자동으로 해결해 낼 버저닝 시스템 필요한데, 이를 위해 백터 시계(vector clock)를 사용한다

</aside>

- 백터 시계
    - [서버, 버전]의 순서쌍을 데이터에 매단 것
    - 어떤 버전이 선행 버전인지, 후행 버전인지, 아니면 다른 버전과 충돌이 있는지 판별하는데 쓰임
    - ex) D([S1, v1], [S2, v2], …, [Sn, vn])
        - D=데이터,  vi=버전 카운터, Si=서버 번호
        - [Si, vi]가 있으면 vi를 증가시킴
        - 그렇지 않으면 새 항목 [Si, 1]을 만듦

  ![image.png](attachment:9cd48294-3698-4d34-b432-2f00e618048c:8a3fc2ed-ad8f-4a3b-8321-89005f0fc5fd.png)

    1. 클라이언트가 데이터 D1을 시스템에 기록
        - 이 쓰기 연산을 처리하는 서버는 Sx, 따라서 백터 시계는 **D1[(Sx, 1)]**으로 변한다
    2. 다른 클라이언트가 데이터 D1을 읽고 D2로 업데이트한 다음 기록
        - **D2[Sx, 2]**
            - D2는 D1에 대한 변경이므로 D1을 덮어쓴다 (쓰기 연산은 같은 서버 Sx가 처리한다고 가정)
    3. 다른 클라이언트가 D2를 읽어 D3로 갱신한 다음 기록(Sy가 처리)
        - **D3([Sx,2], [Sy,1])**
    4. 또 다른 클라이언트가 D2를 읽고 D4로 갱신한 다음 기록 (Sz가 처리)
        - **D4([Sx, 2], [Sz, 1])**
    5. 어떤 클라이언트가 D3, D4를 읽으면 데이터 간 충돌이 있다는 것을 알게됨


        | 구성요소 | D3 | D4 | 비교 결과 |
        | --- | --- | --- | --- |
        | Sx | 2 | 2 | 동일  |
        | Sy | 1 | 0 (없음) | D3이 큼  |
        | Sz | 0 (없음) | 1 | D4이 큼  |
        - D2를 Sy, Sz가 각기 다른 값으로 바꾸었기 때문
        - 이 충돌은 클라이언트가 해소한 후에 서버에 기록
        - **D5([Sx, 3], [Sy, 1], [Sz, 1])**

- 백터 시계를 사용하면 선후 버전, 충돌의 유무 쉽게 판단 가능
    - 버전 Y에 포함된 모든 구성요소 값이 X에 포함된 모든 구성요소 값보다 같거나 큰지만 보면 됨
        - D([s0, 1], [s1, 1])은 D([s0, 1], [s1 ,2])의 이전 버전이므로 두 데이터 사이에 충돌은 없음
    - 버전 X, Y 사이에 충돌이 있는지 보려면 Y의 백터 시계 구성요소 가운데 X의 백터 시계 동일 서버 구성 요소보다 작은 값을 갖는 것이 있는지 보면 됨
        - 두 버전이 같은 이전 버전에서 파생된 다른 버전들인지 보는 것
        - D([s0,1], [s1,2])와 D([s0,2], [s1,1])는 서로 충돌
- [서버: 버전]의 순서쌍 개수가 굉장히 빨리 늘어남
    - 길이에 대한 어떤 임계치를 설정하고, 그 이상 길어질 시 오래된 순서쌍을 백터 시계에서 제거하도록 해야함 → 단, 버전 간 선후 관계의 정확성 떨어져 충돌 해소 과정의 효율성 낮아질 수 있음

### 장애 처리

### 장애 감지

> 분산 시스템에서는 보통 두 대 이상의 서버가 똑같이 서버 A의 장애를 보고해야 해당 서버에 실제로 장애가 발생했다고 간주한다
>
- 멀티캐스팅

  ![image.png](attachment:c8f538d6-4043-497d-bbb8-e66e2b98bde3:664c6434-5500-45c1-a341-8bf60d38820e.png)

    - 모든 노드 사이에 멀티캐스팅 채널을 구축하는 것으로 서버 장애를 감지하는데 가장 쉬운 방법이다
        - 클러스터 내의 모든 서버들이 특정 멀티캐스트 주소에 가입하고 각 서버는 주기적으로 자신의 상태메시지를 해당 멀티캐스트 주소로 보냄
    - 서버가 많을 시 비효율적
        - 불필요한 메시지 수신, 패킷 폭증 위험 등 네트워크 리소스 낭비, 서버 처리 부하 낭비
- **가십 프로토콜: 분산형 장애 감지**
    - 각 노드는 멤버십 목록을 유지
        - 멤버십 목록은 각 멤버 ID와 그 박동 카운터 쌍의 목록
    - **각 노드는 주기적으로 자신의 박동 카운터를 증가**
    - 각 노드는 무작위로 선정된 노드들에게 주기적으로 자기 박동 카운터 목록을 전송
    - 박동 카운터 목록을 받은 노드는 멤버십 목록을 최신 값으로 갱신
    - 어떤 멤버의 박동 카운터 값이 지정된 시간 동안 갱신되지 않으면 해당 멤버는 장애(offline) 상태인 것으로 간주

  ![노드 s0는 왼쪽 테이블과 같은 멤버십 목록 가진 상태](attachment:538848ed-abe5-48b1-a206-1ca963dd8811:717d6da5-c0fc-440a-9d65-f0d300cef279.png)

  노드 s0는 왼쪽 테이블과 같은 멤버십 목록 가진 상태

    - 노드  s0는 노드 s2의 박동 카운터가 오랫동안 증가되지 않았음을 발견
    - 노드 s0는 노드 s2를 포함하는 박동 카운터 목록을 무작위로 선택된 다른 노드에게 전달
    - 노드 s2의 박동 카운터가 오랫동안 증가되지 않았음을 발견한 모든 노드는 해당 노드를 장애 노드로 표시

### 일시적 장애 처리

- 엄격한 정족수(strict quoroum) 접근법
    - 읽기와 쓰기 연산을 금지 시킴
- 느슨한 정족수(sloppy quorum) 접근법
    - 정족수 요구사항을 강제하는 대신, 쓰기 연산을 수행할 W개의 건강한 서버와 읽기 연산을 수행하는 R개의 건강한 서버를 해시 링에서 고른다 (장애 서버는 무시)
- 임시 위탁(hinted handoff) 기법
    - 장애 서버로 가는 요청은 다른 서버가 잠시 맡아 처리한다
        - 임시로 쓰기 연산을 처리한 서버에는 그에 관한 힌트를 남겨둠
    - 해당 서버가 복구되면 위 변경 사항들을 일괄 반영하여 데이터 일관성 보존한다

  ![장애 상태인 노드 s2에 대한 읽기 및 쓰기 연산은 일시적으로 노드 s3가 처리한다. s2가 복구되면 s3는 갱신된 데이터를 s2로 인계한다](attachment:d6471230-c18a-473f-8c49-d1e758f5685d:86d72fdf-48ad-4da1-9ef4-cfe619559109.png)

  장애 상태인 노드 s2에 대한 읽기 및 쓰기 연산은 일시적으로 노드 s3가 처리한다. s2가 복구되면 s3는 갱신된 데이터를 s2로 인계한다

### 영구 장애 처리

> 영구적인 노드의 장애 상태를 처리하기 위해선 **반-엔트로피(anti-entropy) 프로토콜**을 구현해 사본들을 동기화한다.
>
- **반-엔트로피 프로토콜은 사본들을 비교하여 최신 버전으로 갱신하는 과정을 포함한다**
- 사본 간의 일관성이 망가진 상태를 탐지하고 전송 데이터의 양을 줄이기 위해서는 **머클(Merkle) 트리**를 사용
    - 머클 트리(해시 트리): 각 노드에 그 자식 노드들에 보관된 값의 해시(자식 노드가 종단 노드인 경우) 또는 자식 노드들의 레이블로부터 계산된 해시 값을 레이블로 붙여두는 트리
    - 머클 트리를 사용하면 대규모 자료 구조의 내용을 효과적이면서도 안전한 방법으로 검증 가능

- 키 공간이 1~12일 때 머클 트리

![image.png](attachment:fc467adb-1a4e-4d3d-af46-46aa6ec26749:image.png)

1. 키 공간을 버킷으로 나눈다 (위 그림은 네 개의 버킷으로 나눈 모습)

![image.png](attachment:964824b3-b6f8-4311-b551-df0accaaa745:image.png)

1. 버킷에 포함된 각각의 키에 균등 분포 해시(uniform hash) 함수를 적용하여 해시 값을 계산한다

![image.png](attachment:8dede82e-1e1c-4958-80e4-a0a342b2d4c6:image.png)

1. 버킷별로 해시 값을 계산한 후, 해당 해시 값을 레이블로 갖는 노드를 만든다

![image.png](attachment:faa67e72-8000-4020-b57a-2e634c8368c1:image.png)

1. 자식 노드의 레이블로부터 새로운 해시 값을 계산하여, 이진 트리를 상향식으로 구성해 나간다

<aside>
💡

이 두 머클 트리의 비교는 루트 노드의 해시 값을 비교하는 것으로 시작

</aside>

- 루트 노드의 해시 값이 일치하면 두 서버는 같은 데이터를 갖는 것
- 값이 다른 경우 왼쪽 자식 노드의 해시 값을 비교하고, 그 다음으로 오른쪽 자식 노드의 해시 값을 비교한다.
    - 이렇게 아래 쪽으로 탐색하다 보면 다른 데이터를 갖는 버킷 발견 가능 → 그 버킷들만 동기화
- 머클 트리를 사용하면 동기화해야 하는 데이터의 양은 실제로 존재하는 차이의 크기에 비례할 뿐, 두 서버에 보관된 데이터의 총량과는 무관해진다
    - 단, 이 버킷 하나의 크기가 꽤 크다


| 구분 | 정족수 (Quorum) | 반 엔트로피 (Anti-Entropy) + 머클 트리 |
| --- | --- | --- |
| **목적** | **쓰기/읽기 중** 일시적 장애를 허용하고 데이터의 **가용성** 및 **일관성**을 높임. | **쓰기/읽기 후** 발생하는 데이터 불일치(Drift)를 감지하고 복구하여 **최종적 일관성**을 보장함. |
| **작동 시점** | **실시간** 트랜잭션 처리 중 | **백그라운드**에서 주기적으로 |
| **장애 유형** | **일시적** 장애 (네트워크 끊김, 일시적 노드 다운) | **영구적** 장애 (노드 완전 고장/삭제) 또는 장기간 네트워크 분할로 인한 데이터 불일 |

### 데이터 센터 장애 처리

> 데이터를 여러 데이터 센터에 다중화 하는 것 중요
>

### 시스템 아키텍처 다이어그램

- 클라이언트는 키-값 저장소가 제공하는 두 가지 단순한 API, 즉 `get(key)` 및 `put(key, value)`와 통신한다
- 중재자는 클라이언트에게 키-값 저장소에 대한 프록시(proxy) 역할을 하는 노드이다
- 노드는 안정 해시의 해시 링 위에 분포한다

![image.png](attachment:92896761-8430-473f-ba51-0f037e5d2214:image.png)

- 노드를 자동으로 추가 또는 삭제할 수 있도록, 시스템은 완전히 분산된다 (decentralized)
    - 수평적 확장성
- 데이터는 여러 노드에 다중화된다
- 모든 노드가 같은 책임을 지므로, SPOF(Single Point of Failure)는 존재하지 않는다

![image.png](attachment:34400dc4-bf6d-4ce1-8bb7-3c8a32e51a4e:image.png)

- 완전히 분산된 설계를 채택 했으므로, 모든 노드는 위 기능들을 전부 지원해야 한다
    - 시스템 내에 역할 분담이 없으며, 모든 노드가 동일한 책임과 기능을 가진다는 원칙

### 쓰기 경로

![카산드라 구조 사례](attachment:85145fcd-e398-4b12-bcf8-e8717fcd0417:image.png)

카산드라 구조 사례

1. 쓰기 요청이 커밋 로그 파일에 기록된다
2. 데이터가 메모리 캐시에 기록된다
3. 메모리 캐시가 가득차거나 사전에 정의된 어떤 임계치에 도달하면 데이터는 **디스크에 있는 SSTable**에 기록된다
    - SSTable은 Sorted-String Table의 약어로, (키, 값)의 순서쌍을 정렬된 리스트 형태로 관리하는 테이블

### 읽기 경로

![image.png](attachment:3841d135-3de9-440d-929e-aaa19498df84:image.png)

- 읽기 요청을 받은 노드는 데이터가 메모리 캐시에 있는지부터 살핀다
    - 메모리에 데이터가 있는 경우 위와 같이 바로 클라이언트에게 반환
    - 메모리에 없는 경우 디스크에서 가져온다
        - 어느 SSTable에 찾는 키가 있는지 효율적으로 알아내기 위해 블룸 필터(Bloom filter)가 흔히 사용된다

![데이터가 메모리에 없을 때 읽기 연산의 처리](attachment:1d49394a-40f8-4793-808c-44386cfcb710:image.png)

데이터가 메모리에 없을 때 읽기 연산의 처리

1. 데이터가 메모리에 있는지 검사한다. 없을 시 2로 간다
2. 데이터가 메모리에 없으므로 블룸 필터를 검사한다
3. 블룸 필터를 통해 어떤 SSTable에 키가 보관되어 있는지 알아낸다
4. SSTable에서 데이터를 가져와 해당 데이터를 가져온다
5. 해당 데이터를 클라이언트에게 반환한다

## 요약

- 대규모 데이터 저장
    - 안정 해시를 사용해 서버들에 부하 부산
- 읽기 연산에 대한 높은 가용성 보장
    - 데이터를 여러 데이터 센터에 다중하
- 쓰기 연산에 대한 높은 가용성 보장
    - 버저닝 및 백터 시계를 사용한 충돌 해소
- 데이터 파티션
    - 안정 해시
- 점진적 규모 확장
    - 안정 해시
- 다양성(heterogeneity)
    - 안정 해시
- 조절 가능한 데이터 일관성
    - 정족수 합의(quorum consensus)
- 일시적 장애 처리
    - 느슨한 정족수 프로토콜(sloppy quorum)과 단서 후 임시 위탁(hinted handoff)
- 영구적 장애 처리
    - 머클 트리(Merkle tree)
- 데이터 센터 장애 대응
    - 여러 데이터 센터에 걸친 데이터 다중화
