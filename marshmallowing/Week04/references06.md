Bloom Filter는 공간 효율적인 확률적 자료구조로서, 어떤 요소가 집합에 **존재하는지 여부**를 빠르게 판단하는 데 쓰인다.

- **False negative**: 절대 없음 (집합에 있는데 Bloom 필터가 없다 한다 → 없음)
- **False positive**: 가능성 있음 (집합에 없는데 Bloom 필터가 있다 하는 경우)

> 위와 같은 Bloom Filter를 이용한 membership query(어떤 집합과 어떤 원소가 포함 관계인지 묻는 것) 결과 값의 특징 때문에 Bloom Filter는 단독으로 사용하는 것보다는 확률적인 방법이 아닌 다른 방법을 보조하는 역할로 사용하는 것이 적합하다.
>

- Bloom Filter는 **m 비트(bit array)**로 구성되며, 초기에는 모두 0으로 설정
- **k개의 서로 다른 해시 함수(hash functions)**를 갖는다
    - 해시 함수는 각각의 입력 값에 대해 0 ~ m-1 사이의 비트 인덱스로 매핑해 반환

## 요소 추가

1. 요소 x를 추가할 때 x에 대해 k개의 해시 함수를 적용한다
2. 해시 함수의 출력에 모듈러 연산이 실행되어 k개의 인덱스(i₁, i₂, …, iₖ)를 구한다
3. 비트 배열에서 이 k개의 인덱스 위치에 해당하는 비트를 모두 1로 설정(set)한다

   ![블룸 필터에 항목 추가하기](attachment:f5cae846-b5e3-4fb6-afb7-9bd59ad15012:image.png)

   블룸 필터에 항목 추가하기

- 항목 red와  blue가 블룸 필터에 추가된다
    - 항목 red에 대해 1로 설정되어야 하는 버킷은 **계산된 해시 값에 모듈러 연산을 실행하여 식별된다**
        - `h1(red)(mod10)=1`
        - `h2(red)(mod10)=3`
        - `h3(red)(mod10)=5`
    - 항목 blue 도 위와 같은 방식으로 식별된다
        - `h1(blue)(mod10)=4`
        - `h2(blue)(mod10)=5`
        - `h3(blue)(mod10)=9`

## 요소 조회**(Membership Query)**

### Membership Query

- 어떤 집합에 어떤 원소 x가 포함되어 있는 관계인지 묻는 것
    - ex) java.util.Set 인터페이스의 `boolean contains(Object e)`
- Bloom Filter는 메모리를 매우 적게 사용하면서 membership query 결과를 알 수 있는 방법
    - 데이터베이스를 구현할 때나 네트워크 보안 시스템을 사용할 때 이용


### 요소 조회

- 원소 x에 대해 k개의 모든 해시 결과를 V에 대한 배열 인덱스로 사용하였을 때 해당 배열 값이 모두 1이면 '아마도 해당 집합의 원소인 것 같다'라고 판단한다.
- **이때 단 하나라도 원소 배열 값이 1이 아니라면, '집합에 포함된 원소가 확실히 아니다'라고 판단할 수 있다.**

![image.png](attachment:7f6bba44-c9a8-45aa-8b29-c81f4c16d779:image.png)

- 집합 {x,y,z}의 원소에 대해 각각의 해시 함수를 수행했을 때, 해시 함수 결과 값을 배열 인덱스로 한 값은 모두 1이다. ⇒ **이 경우 x,y,z는 '아마도 해당 원소의 집합인 것 같다'라고 판단**
- w에 대해서는 해시 함수의 실행 결과 값이 가리키는 배열 V의 원소 값이 모두 1은 아니다 ⇒ **w는 '집합에 포함된 원소가 확실히 아니다'라고 판단**

## 복잡도

> 블룸 필터는 공간(m)과 시간(k) 모두에서 상수 복잡도를 달성하며, 이는 대규모 시스템에서 성능을 최적화하는 핵심 이유다. 또한 블룸 필터의 성능은 사용된 해시 함수에 따라 달라진다. 해시 함수의 계산이 빠를수록 블룸 필터에서의 각 작업의 전체 시간이 더 빨라진다.
>
- **시간 복잡도** (k는 해시 함수의 개수)


    | 연산 | 시간 복잡도 |
    | --- | --- |
    | 항목 추가 (add item) | O(k) 또는 상수 |
    | 멤버십 조회 (membership query) | O(k) 또는 상수 |
    - 블룸 필터의 시간 복잡도는 블룸 필터에 이미 들어있는 항목 수와 독립적이다
    - 블룸 필터에서의 k번의 조회는 독립적이며 병렬로 처리될 수 있다
- **공간 복잡도**
    - 블룸 필터에 있는 항목 수에 관계없이, 항목당 할당되는 비트 공간이 정해져있어 상수 공간 복잡도 **O(1)**에 해당하는 일정한 수의 비트가 필요하다
    - 공간 사용량의 경우 블룸 필터의 전체 크기는 m 비트로, 초기 설정된 크기로 고정된다  - **O(m)**
    - 블룸 필터는 데이터 항목을 저장하지 않으므로 상수 공간 복잡도를 산출한다

## 장단점

- 장점
    - 상수 시간 복잡도, 상수 공간 복잡도
    - 작업의 병렬 처리 가능
    - 거짓 부정이 없음 (No false negatives)
    - 실제 항목을 저장하지 않아 개인 정보 보호 가능
- 단점
    - 블룸 필터는 삭제 연산을 지원하지 않는다
        - 어떤 요소가 설정한 비트를 추적할 수 없기 때문에 삭제가 불가능하다
        - 어떤 비트를 0으로 바꾸면 그 비트를 공유하는 다른 요소들도 영향을 받기 때문
    - 거짓 긍정률을 0으로 줄일 수 없다 (false positive 가능성 존재)
    - 요소 개수가 예상보다 많아지면 false positive가 급격히 증가
    - 블룸 필터를 디스크에 저장하려면 해시 함수에 의해 생성된 무작위 인덱스로 인해 무작위 접근(random access)이 필요하다

<aside>
💡

블룸 필터에서 항목 제거가 지원되지 않는 이유는 지워야 할 비트를 식별할 가능성이 없기 때문이다. 다른 항목이 같은 비트에 매핑되어 있을 수 있으며, 비트를 지우면 거짓 부정의 가능성이 생길 수 있다.

</aside>

## 블룸 필터의 확장

### **카운팅 블룸 필터 (Counting Bloom Filter)**

각 비트 대신 카운터(counter)를 두어 삭제(Delete)를 지원하도록 한 변형

![image.png](attachment:765639bb-90b4-40a9-b2d0-2c46870f6a0b:image.png)

- 항목이 카운팅 블룸 필터에 추가될 때 관련 비트의 카운터가 증가
- 카운팅 블룸 필터는 삭제 연산을 지원
- 추가 메모리가 카운터를 위해 할당되어야 하므로 고전적인 블룸 필터보다 훨씬 크다

### **확장 가능한 블룸 필터 (Scalable Bloom Filter)**

삽입될 요소 수가 미리 정해지지 않은 경우에도 false positive 확률을 보장하도록 설계된 변형

![image.png](attachment:3121be71-c561-49b9-a9fb-eb4a359c61d3:image.png)

- 블룸 필터는 멤버인 항목을 식별할 수 없기 때문에 포화(saturation) 상태에서 재조정할 수 없다
    - 따라서 확장성을 위해 블룸 필터를 쌓아 올릴 수 있다
- 블룸 필터가 포화되면, 현재 블룸 필터 위에 더 높은 용량의 **새로운 블룸 필터**가 생성

## 마무리

분산 키-값 저장소의 읽기 경로에 대해 공부하던 중, SSTable에서 키의 존재 여부를 파악하는 데 블룸 필터(Bloom Filter)가 흔히 사용된다는 사실을 접하고 이에 대해 깊이 있게 정리해 보았다. 처음 접하는 내용이라 생소했지만, 이미 많은 대규모 데이터베이스와 시스템에서 성능 최적화 도구로 활용되고 있음을 알 수 있었다

블룸 필터의 확률적 특성으로 단독으로 사용되긴 어렵지만 없는 것을 걸러내는 역할로 불필요한 디스크 접근을 생략할 수 있는 효율적인 기능이다.

블룸 필터는 **공간 효율적인 확률적 자료구조**로서, 다음과 같은 핵심적인 역할을 수행한다.

- **효율성 극대화:** 블룸 필터는 확률적 특성상 단독으로 사용되기는 어렵지만, **'없는 것을 걸러내는(Negative Screening)'** 역할에 매우 효율적이다. 이를 통해 불필요한 디스크 접근(Disk I/O)을 생략할 수 있어 시스템의 성능과 처리량을 획기적으로 향상시킨다.
- **광범위한 적용**
    - **데이터베이스:** **Hadoop, BigTable, Cassandra**와 같은 데이터 분석 및 로그 저장소는 디스크 접근 비용이 높기 때문에, 블룸 필터를 적용하는 것이 매우 유리하다
    - **보안 및 네트워크:** Chrome에서 보안 위험 URL 접근 여부를 판단하거나, 네트워크 시스템에서 블랙리스트를 필터링하는 목적으로도 폭넓게 사용된다

블룸 필터는 **O(1) 상수 공간 복잡도**와 **O(k) 상수 시간 복잡도**라는 강력한 장점을 제공하지만, 치명적인 단점 하나가 데이터 삭제가 불가능하다는 점이다.

- **데이터 삭제 불가:** 블룸 필터는 한 번 삽입한 원소를 **삭제(Delete)할 수 없다.** 이는 서로 다른 두 원소 x와 y가 비트 배열의 동일한 위치를 공유할 수 있기 때문이다. 만약 x를 삭제하기 위해 해당 비트를 0으로 되돌리면, y의 정보까지 손상되어 **거짓 부정(False Negative)** 오류가 발생할 수 있다.
