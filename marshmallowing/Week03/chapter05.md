# 5장: 안정 해시 설계

> 수평적 규모 확장성을 달성하기 위해서는 요청 또는 데이터를 서버에 균등하게 나누는 것이 중요하다.
>

### 해시 키 재배치(refresh) 문제

- 서버 부하를 균등하게 나누는 보편적 방법 : **해시함수**
    - `serverIndex = hash(key) % N`
    - N = 서버 수

  ![4개의 서버](attachment:dde37909-ac18-4f22-8e65-bf7670e8d95d:image.png)

  위 예제에서 키값이 서버에 분산되는 형태

    - 서버 풀(server pool)의 크기가 고정되어 있을때, 데이터 분포가 균등할 때 잘 동작
    - 서버가 추가되거나 삭제될 시 문제 발생
        - ex) 1번 서버가 장애로 동작 중단 → 서버 풀의 크기는 3 ⇒ 키에 대한 해시 값은 변하지 않지만 나머지 연산을 적용하여 계산한 서버 인덱스 값은 달라짐

### 안정 해시(consistent hash)

> 안정 해시는 위의 대규모 캐시 미스 문제를 해결하는 기술이다.
>
- **해시 테이블 크기가 고정될 때 평균적으로 오직 k/n개의 키만 재배치하는 해시 기술**
    - k = 키의 개수 / n = 슬롯의 개수
    - 전통적 해시 테이블은 슬롯의 수가 바뀌면 거의 대부분 키 재배치

### 해시 공간과 해시 링

- 해시 함수 f = SHA-1
    - 해시 공간 범위: 0~2^160-1
    - 출력 범위: x0,x1,…,xn (x0=0, xn=2^160-1)

### 가상 노드

> 실제 노드 또는 서버를 가리키는 노드로서, 하나의 서버는 링 위에 여러 개의 가상 노드를 가질 수 있다.
>
- 가상 노드의 수를 늘리면 키의 분포는 더 균등해진다
    - 표준 편차가 작아져서 데이터가 고르게 분배
        - 표준 편차: 데이터가 어떻게 퍼져나갔는지를 보이는 척도
    - 단, 가상 노드 데이터를 저장할 공간은 더 많이 필요하게 된다

![image.png](attachment:07a6db14-e0c8-41e6-ad39-019f7aee1dcc:image.png)

- 서버 0과 서버 1은 3개의 가상 노드를 갖는다
    - 서버 0을 링에 배치하기 위해 s0_0, s0_1, s0_2 세 개의 가상 노드 사용
- 각 서버는 하나가 아닌 여러 개 파티션을 관리해야 한다
    - s0로 표시된 파티션은 서버 0이 관리하는 파티션

![image.png](attachment:90e7d252-7388-4d04-893d-1c70f8ec1086:image.png)

- 키의 위치로부터 시계 방향으로 링을 탐색하다 만나는 최초의 가상 노드가 해당 키가 저장될 서버
    - k0가 저장되는 서버는 k0의 위치로부터 링을 시계방향으로 탐색하다 만나는 최초의 가상 노드 s1_1가 나타내는 서버, 즉 서버1이다

### 정리

- 안정 해시의 이점
    - 서버가 추가되거나 삭제될 때 재배치되는 키의 수가 최소화
    - 데이터가 보다 균등하게 분포하게 되므로 수평적 규모 확장성 달성 쉬움
    - 핫스팟 키 문제를 줄인다
        - 특정한 샤드에 대한 접근이 너무 빈번하면 서버 과부하 문제 발생 가능
        - 안정 해시는 데이터를 더 균등하게 분배하므로 이러한 문제 발생 줄임

- 안정 해시 기술의 예시
    - 아마존 DynamoDB의 파티셔닝 관련 컴포넌트
    - Apache Cassandra 클러스터에서의 데이터 파티셔닝
    - 디스코드 채팅 어플리케이션
    - 아카마이 CDN
    - Meglev 네트워크 부하 분산기
