# Dynamo: Amazon의 고가용성 키-값 저장소 요약

https://www.allthingsdistributed.com/files/amazon-dynamo-sosp2007.pdf

> Amazon이 내부 서비스용으로 만든 **고가용 분산 키-값 저장소(Dynamo)** 설계와 구현에 대한 논문을 읽고 정리한 글입니다.
>

## 목표 및 개요

> 아마존은 쇼핑 카트, 고객 프로필 등 핵심 서비스에 사용할 대규모 분산 데이터 저장소가 필요했다. 전통적인 관계형 데이터베이스(RDB)는 엄격한 **일관성(Consistency)**을 보장하지만, 서버 장애 시 **가용성(Availability)**과 **확장성(Scalability)**이 떨어지는 문제가 있었다.
>
- 2000년대 초반, Amazon은 **쇼핑 카트 서비스의 장애가 매출 손실로 직결**된다는 문제를 겪었다.
    - “사용자 카트가 열리지 않거나 주문이 실패하면 곧바로 매출 손실이다.”
    - 당시 관계형 DB 기반의 시스템은 트랜잭션 안정성은 높았지만, 하나의 노드 장애에도 전체 시스템이 영향을 받는 단일 실패 지점(Single Point of Failure)이 있었다.
- 서비스가 글로벌하게 확장되며, 데이터는 지리적으로 분산된 여러 서버에 존재해야 했고, 그 과정에서 **일관성(Consistency)** vs **가용성(Availability)** vs **분할 내성(Partition tolerance)** 간의 **CAP 트레이드오프**가 현실적인 문제가 되었다.

<aside>
💡

따라서 모든 상황에서도 사용자 요청을 거부하지 않는 **Always-On** 경험을 제공하기 위해, 네트워크 파티션(장애) 상황에서도 쓰기 요청을 거부하지 않고 데이터에 접근할 수 있는 **고가용성**과 **확장성**을 최우선으로 하는 분산 저장소 시스템을 설계했다

</aside>

- Amazon의 요구사항
    1. **항상 쓰기 가능**
    2. **높은 가용성**
    3. **지연 시간 보장**
    4. 다양한 애플리케이션별로 일관성/내결함성 등의 트레이드오프를 조절 가능해야 함

> 분산 시스템의 CAP 이론 중 **가용성(A)**과 **분할 내성(P)**을 선택하고, **엄격한 일관성(C)**을 포기하는 **최종적 일관성(Eventual Consistency)** 모델을 채택했다. 즉, 데이터는 비동기적으로 전파되지만 이에 따른 **데이터 충돌 문제**가 발생할 수 있다.
>

## 핵심 설계 원칙 및 기술

<aside>
💡

Dynamo는 고가용성을 달성하기 위해 여러 잘 알려진 기술들을 조합하여 사용했다. 특히 **일관된 해싱**은 데이터를 분산하고 확장성을 확보하는 핵심 기술이다

</aside>

- **다이나모가 사용하는 기술과 장점**

| 문제 | 적용 기술 | 장점 |
| --- | --- | --- |
| 파티셔닝 | **안정 해싱(Consistent Hashing)** | 점진적 확장, 부하 균등화 |
| 쓰기 고가용성 | **벡터 클록(Vector Clock)** | 데이터 버전 추적 및 충돌 감지 |
| 일시적 장애 | **느슨한 정족수(Sloppy Quorum)**, **힌티드 핸드오프(Hinted Handoff)** | 장애 발생 시에도 서비스 지속 |
| 영구적 장애 | **머클 트리(Merkle Tree)** 기반 안티 엔트로피(Anti-Entropy) | 백그라운드에서 레플리카 동기화 |
| 멤버십/장애 감지 | **Gossip 프로토콜** | 중앙 서버 없이 상태 정보 공유 |

## 1. **시스템 인터페이스**

> Dynamo는 단순한 인터베이스를 통해 키와 결합된 객체를 저장한다. 인터페이스에는 `get()`과 `put()` 두 개의 명령이 있다.
>
- `get(key)`
    1. 클라이언트가 `get(key)` 요청
    2. coordinator는 preference list 상위 N 노드에 요청
    3. R 개의 응답을 기다림
    4. 만약 동일한 버전이 여러 개면 모두 반환 (충돌 가능성 존재)
    5. 클라이언트가 받은 여러 버전을 병합하거나, 그 중 하나를 선택해서 write back
    6. 이 과정을 통해 일관성 정리 (reconciliation)
    - 저장소 시스템에서 키와 연결된 객체 레플리카를 찾고 충돌 버전의 단일 객체나 객체 목록을 컨텍스트와 함께 반환
- `put(key, context, object)`
    1. 클라이언트가 `put(key, context, value)` 요청
    2. coordinator 노드가 벡터 클록을 생성하거나 갱신
    3. coordinator는 우선 로컬에 write
    4. 그 뒤에 preference list 상의 상위 N 노드에 복제 전송
    5. W 개 노드가 응답하면 성공으로 간주
    6. 쓰기 실패한 노드가 있으면 hinted handoff 로 복구 가능하게 함
    - 결합된 키를 기준으로 객체의 레플리카 위치를 결정하고, 디스크에 저장한다.
    - 컨텍스트는 객체 버전처럼 호출자에게 공개되지 않는 객체에 대한 시스템 메타데이터를 인코딩한다
    - 컨텍스트 정보는 객체와 함께 저장되기 때문에 시스템은 `put` 요청에서 제공된 컨텍스트 객체의 유효성을 확인 가능

## 2.  데이터 파티셔닝: 안정 해싱 (Consistent Hashing)


키(Key)와 노드(Node) 모두를 하나의 큰 **해시 링(Hash Ring)** 상의 위치(*position*)로 매핑한다. 특정 키에 대한 요청이 들어오면, 키의 해시 위치에서 링을 시계 방향으로 돌며 데이터를 저장할 첫 번째 노드를 찾는다.

- 각 노드는 **링 위의 이전 노드와 자기 자신 사이의 영역**을 담당하게 된다.
- 안정 해싱의 주요 이점은 노드의 추가 또는 제거가 **인접 노드**에게만 영향을 미치고, 다른 노드들에게는 영향을 미치지 않는다는 것이다
    - 클러스터 전체에 미치는 영향을 최소화
    - 시스템의 **탄력성**(Elasticity)과 **확장성**을 극대화

### *⇒ 로드 밸런싱: 가상 노드 (Virtual Nodes)*

기본적인 안정 해싱 알고리즘에서 링에 있는 각 노드의 위치를 랜덤하게 배정하면 불균형한 부하 분산을 야기할 수 있다. 또한, 기본적인 알고리즘은 노드의 성능을 신경쓰지 않는다.

<aside>
💡

위 문제를 해결하기 위해 다이나모는 단순히 원 위에 하나의 점으로 노드를 매핑하는 것이 아니라,  **가상 노드(*virtual nodes*)**라는 개념을 사용해 **각 노드에 여러 개의 위치를 배정한다**

</aside>

- 가상 노드는 시스템에서 하나의 노드처럼 보이지만, 각 노드가 하나 이상의 가상 노드를 담당할 수 있으며, 새로운 노드를 시스템에 추가할 때 링에서 여러 위치(토큰)가 할당된다.
- 가상 노드가 많아질수록 링 상에 노드가 더 고르게 분포되어 데이터와 요청 부하가 노드 전반에 **균등하게 분산**
- 노드 용량에 따라 가상 노드 수를 조절하여 성능이 좋은 노드에 더 많은 부하를 할당 가능
- 장애나 정기적인 유지보수로 인해 노드를 사용할 수 없게 되면, 해당 노드가 처리하던 부하를 나머지 사용 가능한 노드들에게 고르게 분산 가능

> 즉, Dynamo의 안정 해싱과 가상 노드 구조는 **부하 분산 + 유연한 확장성 + 노드 장애 격리**를 동시에 달성하게 해준다. 이 원리는 이후 **Cassandra**, **Riak**, **Voldemort** 등 여러 NoSQL 시스템의 기본 구조로 채택되었다.
>

## 3. 고가용성 보장: 복제 및 정족수 (Replication & Quorum)

데이터 업데이트 중 레플리카(*replicas*) 사이의 일관성은 정족수(*quorum*) 기법과 분산 레플리카 동기화 프로토콜로 유지할 수 있다.

- **복제**
    - 모든 데이터는 N개의 노드에 복제된다. 해당 키를 책임지는 노드(N1)로부터 시작하여 링을 따라 시계 방향으로 이어진 N−1개의 노드에 데이터를 저장한다.
    - 이 N개 노드의 목록을 **선호 목록(Preference List)**이라 한다


### **Quorum Consensus(정족수 합의) 프로토콜**

여러 노드에 다중화된 데이터는 적절히 동기화가 되어야 한다. 정족수 합의 프로토콜을 사용하면 읽기/쓰기 연산 모두에 일관성을 보장할 수 있다.

- **R (Read Quorum):** 읽기 요청이 성공으로 간주되려면 응답해야 하는 최소 복제본 수
    - 읽기 성공 정족수 기준
- **W (Write Quorum):** 쓰기 요청이 성공으로 간주되려면 업데이트되어야 하는 최소 복제본 수
    - 쓰기 성공 정족수 기준


N=3인 경우

- **R=1, W=N** : 빠른 읽기 연산에 최적화된 시스템
- **W=1, R=N** : 빠른 쓰기 연산에 최적화된 시스템
- **R+W>N :** 읽기와 쓰기 작업이 항상 최소한 하나의 최신 버전 복제본을 공유하게 되므로 강한 일관성 보장
- **R+W≤N** : 강한 일관성이 보장되지 않는다

---

- 강한 일관성(Strong Consistency) : 모든 읽기는 최신 데이터이다. 낡은 데이터를 반환하는 경우는 없다.
- 약한 일관성(Weak Consistency) : 읽기 연산은 최신 데이터가 아닐 수 있다. 어느정도 일관성 손실을 감수한다.
- 최종 일관성(Eventual Consistency) : 읽기 연산은 최신 데이터가 아닐 수 있으나 시간이 지나면 모든 사본에 일관성이 맞춰진다. ⇒ **Dynamo**

- **Dynamo의 선택: 느슨한 정족수 (Sloppy Quorum) 시스템**

    <aside>
    💡

  Dynamo quorum의 특이점은 요청을 받아 노드에게 전파해주는 중개자(Coordinator)가 따로 없다. 요청을 보내는 key에 따라 안정해시에서 해당 데이터를 저장해야하는 **첫번째 노드가 중개자 역할을 하도록 구현되어있다.**

    </aside>

  > 요청을 받아 중개자가 된 노드는 복제해야하는 노드 수 만큼 다른 노드(preference list)에 데이터를 전파하는데 이 과정 속에서 실패 여부는 **느슨한 정족수(Sloppy Quorum)**를 사용해서 성공 응답을 클라이언트에게 반환한다.
  >
    - 쓰기 연산용 W개의 정상노드, 읽기 연산용 R개의 정상노드만 사용한다. 나머지 서버는 무시한다.
    - **느슨한 정족수를 사용해서 일부 노드가 실패해도 남은 노드들만으로 읽기/쓰기 연산을 수행하고, 나중에 복구된 노드에 데이터를 비동기적으로 동기화한다.**
    - Dynamo는 쓰기 요청이 들어올 때 모든 복제본이 업데이트될 때까지 기다리지 않는다. 이로 인해 잠시 동안 동일한 데이터에 대해 여러 버전이 공존하는 **데이터 충돌**이 발생할 수 있다.

## 4. 충돌 해결: 벡터 시계 (Vector Clocks)

> 최종 일관성등에서 일관성을 해소하는 방법 중 하나는 데이터 버저닝(versioning)과 벡터 시계(vector clock)을 사용하는 방법이 있다. 버저닝은 데이터 변경 시마다 데이터의 새로운 버전을 만드는 것으로 각 버전의 데이터는 불변하다.
>
- **벡터 시계:** 각 데이터 항목의 버전 관리를 위해 **(노드, 카운터)** 쌍의 목록인 벡터 시계을 사용
    - [서버, 버전]의 순서쌍을 시간 값처럼 데이터에 달아두는 것
    - 이 시계을 통해 어떤 버전이 선행(A가 B보다 먼저 발생)하는지, 아니면 동시적(concurrent)으로 발생했는지 판단


- [Si, vi]가 있으면 vi를 증가시킨다.
- 그렇지 않으면 새 항목 [Si, 1]를 만든다

⇒ 벡터 시계를 사용하면 어떤 버전 X가 버전 Y의 이전 버전인지(따라서 충돌이 없는지) 쉽게 판단할 수 있다. **버전 Y에 포함된 모든 구성요소의 값이 X에 포함된 모든 구성요소 값보다 같거나 큰지만 보면 된다.**

- 예를 들어 벡터 시계 D([s0, 1], [s1, 1])은 D([s0, 1], [s1, 2])의 이전 버전이다. 따라서 두 데이터 사이에 충돌은 없다.

- **충돌 해결**
    - Dynamo는 동시적으로 발생한 충돌을 자동으로 해결하려 하지 않고, 읽기 요청 시 클라이언트(애플리케이션 계층)에게 **충돌하는 여러 버전을 모두 전달하여 애플리케이션 논리에 따라 해결하도록 위임한다**

  ⇒ **응용 계층에서 충돌을 해결하도록 위임**함으로써 시스템 단의 단순성과 가용성을 유지


---

| 문제 상황 | Dynamo의 접근 | 효과 |
| --- | --- | --- |
| 서버 장애 시 요청 실패 | 느슨한 정족수 + hinted handoff | 요청 손실 없이 임시 저장 및 나중 복구 |
| 확장 시 전체 재분배 부담 | Consistent Hashing + Virtual Nodes | 점진적 확장 가능, 부하 균등 |
| 데이터 충돌 | Vector Clock + Application-level reconciliation | 고가용성 유지하면서 충돌 감지 |
| 중앙 관리 부하 | Gossip 기반 멤버십 관리 | 완전 분산 구조, SPOF 제거 |

> Dynamo는 “항상 응답하는 시스템(Always Writeable System)”이라는 Amazon의 요구를 충족시키기 위해 설계된 실용적인 분산 저장소다. **일관성보다는 가용성을 우선**한 설계 철학이 이후의 수많은 NoSQL 시스템(Cassandra, Riak, CouchDB 등)에 큰 영향을 주었다.
>
