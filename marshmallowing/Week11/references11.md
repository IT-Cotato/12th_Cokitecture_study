# 토스 증권의 Apache Kafka 이중화
[토스증권 Apache Kafka 데이터센터 이중화 구성 #1](https://toss.tech/article/kafka-distribution-1)

### 아파치 카프카 (Apache Kafka)

**대규모 실시간 데이터 전송 및 처리를 위한 분산 메시징 플랫폼**

**핵심 기능**

- **메시지 큐:** 한쪽에서 데이터를 생성(Producer)하면 카프카가 이를 보관하고, 필요한 쪽(Consumer)에서 가져가도록 연결
- **영속성:** 일반적인 메시지 큐와 달리 데이터를 디스크에 저장하여 나중에 다시 읽을 수 있는 능력이 있지만, 전통적인 SQL 데이터베이스(MySQL 등)처럼 데이터를 복잡하게 조회(Query)하는 용도는 아니다
- **분산 처리:** 데이터를 여러 대의 서버(브로커)에 나누어 저장하여 트래픽이 몰려도 터지지 않게 설계되어있다

### 쿠버네티스 (Kubernetes / K8s)

**애플리케이션의 배포, 관리, 자동화를 위한 컨테이너 오케스트레이션 도구**

→ 카프카 같은 애플리케이션(컨테이너)들을 어디에 배포하고, 어떻게 관리할지 결정

**핵심 기능**

- **서버 자동 관리:** 서버 한 대가 죽으면 그 위에서 돌아가던 프로그램을 다른 서버에서 자동으로 다시 살려줌
- **자동 확장:** 사용자가 많아지면 서버(컨테이너) 개수를 자동으로 늘린다
- **배포 관리:** 새로운 버전의 코드를 올릴 때 중단 없이 안전하게 올려준다

## **Kafka 이중화의 필요성**

**현재 토스증권의 서비스 서버 구성**

- 서버를 운영하는 Kubernetes는 두 곳의 데이터센터에 비슷한 스팩으로 구성되어 있음
    - 서로 다른 지역의 두 건물에 똑같은 서버 환경(Kubernetes)을 만들어 둠
- 두 데이터센터 모두 평시에 유저 트래픽을 처리하는 Active-Active 구성
    - Active-Active 구성: 두 곳의 데이터센터가 평상시에도 동시에 유저 트래픽을 받는 방식
- 서버는 MSA 구조로 설계되어 Kafka를 메시지 브로커로 활용
    - MSA(마이크로서비스 아키텍처): 전통적인 방식(모놀리틱)이 모든 기능을 하나의 거대한 덩어리로 만드는 것이라면, MSA는 큰 서비스를 기능별로 작게 쪼개서 운영하는 방식


![image.png](attachment:cd148514-baad-4a00-aa22-3b427b301563:image.png)

> 서비스 서버는 Active-Active로 구성돼 이중화 되어있다. 하지만 **MSA에서 메시지 브로커 역할을 하는 Kafka가 이중화 돼있지 않으면 실제 데이터센터 장애 상황 발생 시 서비스는 정상 동작하지 않을 것이 분명하다. 따라서  Kafka 역시 데이터센터 이중화를 해야 하는 상황이다.**
>

## **Kafka 이중화 아키텍처**

- **Active-Active 구성**

  ![image.png](attachment:dcb199c5-b9a3-4956-8890-bbf534fe2d07:image.png)

    - Active-Active 구성은 서로 독립적인 Kafka 클러스터를 양쪽 데이터센터에 각각 구성하는 방식
    - 독립된 두 Kafka 클러스터를 동기화해주기 위해 양방향 데이터 미러링이 필요
- **Stretched Cluster 구성**

  ![image.png](attachment:83049781-db44-4fb9-be47-6c1b71f9e900:image.png)

    - 양쪽 데이터센터에 분리된 노드를 하나의 Kafka 클러스터로 묶어서 구성
    - 물리적으로 분리돼있지만, 한 개의 Kafka 클러스터라 별도 데이터 미러링 작업이 필요 없이 동기화

<aside>
💡

두 방식의 가장 큰 차이점은 Kafka 클러스터를 하나로 구성할 것인지, 분리해서 두 개로 구성할지이다

</aside>

- **Active-Active 구성**

  독립적으로 구성된 두 개의 클러스터를 하나로 묶어주는 과정에서 운영부담이 크다는 단점 존재

    - 장점
        - Producer와 Consumer는 **지역성이 보장된 Kafka에 접근**하기 때문에 latency가 상대적으로 짧음
        - 양 클러스터가 독립적으로 구성됐으므로 한쪽 데이터센터 장애가 다른 쪽 클러스터에 영향을 주지 않음
    - 단점
        - 수백 개의 토픽을 **양방향 데이터 미러링을 구성하고 운영**해야 한다
        - 수천 개의 Consumer Group Offset을 **양쪽 클러스터에 동기화** 시켜야 한다
        - 토픽관리와 권한관리와 같은 **모든 작업을 두 배로** 해야 한다
- **Stretched Cluster**

  물리적으로 분리된 장비에 하나의 Kafka 클러스터로 구성하기 때문에 추가적인 운영부담없이 이중화가 가능하지만, 평시 성능저하와 Split-brain이 발생할 여지가 있음

    - 장점
        - 이중화를 하지 않은 Kafka 운영과 크게 다르지 않아서 운영이 편리
        - 데이터센터 장애 상황이 발생했을 때 클라이언트들이 안전한 데이터센터에 재연결 후 즉시 작업을 재개할 수 있음
    - 단점
        - Kafka의 전송 프로토콜 구조상 **매 요청마다 데이터센터 간 네트워크를 경유해야 하므로 latency 저하가 심함**
        - Kafka 클러스터의 상태를 관리하는 Zookeeper는 Stretched Cluster 구성 시 장애 상황에도 정족수를 충족시키기 위해 **삼중화 구성**이 필요한데, 데이터센터 간 네트워크가 단절되는 장애 상황에 Zookeeper에서 **Split-brain**이 발생해 Kafka 클러스터가 먹통이 될 가능성이 있음
            - 정족수와 삼중화
                - Kafka에서 Zookeeper는 관리자 역할(Leader)
                    - 브로커 목록 관리, 리더 파티션이 누구인지 결정, 장애 발생 시 새 리더 선출)
                    - **다수결의 원칙:** Zookeeper은 의사결정을 내릴 때 과반수의 찬성이 필요한데 이를 정족수라고 한다

                  > 서버가 3개일 때 1개가 고장 나도 남은 게 2개(66%)라 과반수를 충족해 시스템이 돌아가기 때문에, 따라서 장애 대비를 위해 최소 삼중화(3개 서버) 구성이 필수적이다
            >
            - Split-brain 현상

              Stretched Cluster구조에서 데이터 센터간 네트워크가 끊어지면, 서버 대수 배분이 꼬여서 누가 리더인지 합의하지 못하거나, 혹은 어느 쪽도 과반수를 확보하지 못하면 **전체 카프카 클러스터가 의사결정을 내리지 못하고 멈춰버리는** 상황이 발생

## **Active-Active 구성을 선택한 이유**

**아키텍처 선택에서의 고려 요소**

1. 다양한 장애 케이스에서 가용성을 확보할 수 있는가?
2. 가용성 확보를 위한 이중화 구성이 성능 저하를 유발하지는 않는가?
3. 클러스터 운영이 너무 어렵지는 않은가?

**Stretched Cluster의 큰 결함 두가지**

1. 데이터센터 간 네트워크 단절 장애 케이스에서 **Split-brain 발생**으로 가용성 확보 불가
2. 데이터센터 간 거리에 따라 **통신 latency**가 늘어나기 때문에 Kafka 성능 저하 발생

> Kafka 프로토콜 특성 상 leader가 요청을 받으면 replica 복제 완료 후 클라이언트로 응답을 전달하게 되는데, 매 요청마다 latency가 큰 DC간 네트워크를 경유해야 하는 Stretched Cluster 구성은 Kafka 성능을 크게 저하시키는 문제가 존재한다
>

## **이중화 구성 후 달성해야 할 목표**

1. Kafka 이중화 구성에 대한 이해가 깊지 않은 서비스 서버 개발자들도 쉽게 활용할 수 있어야 한다

   ⇒ 양쪽 Kafka 클러스터에 토픽명을 동일하게 가져가는 정책

   ⇒ 개발자가 이중화된 클러스터 어디에 배포가 되든 신경 쓰지 않아도 잘 동작하는 환경

2. 데이터센터 장애 발생 시 메인 센터 전환이 빠르고 편해야 한다

   ⇒ DNS 정책을 활용하고 Producer와 Consumer를 분리해서 정책을 설계

- **이중화 구성 후 트래픽의 흐름**

  ![image.png](attachment:681bd8f0-c574-4694-9bcb-f7a38338b8aa:image.png)

    <aside>
    💡

  Active-Active로 Kafka 이중화 구성을 하면 실시간으로 양방향 데이터 미러링으로 각각 100%를 맞춰줘야 하고, Consumer가 데이터센터를 옮겨갈 경우를 대비해서 Consumer Group Offset Sync를 맞춰야 한다.

    </aside>

    - **Producer 정책**

      Kafka에 접근할 때 **Split DNS를 활용해서 배포된 zone과 동일한 Kafka로 메시지를 전송**하도록 구성해 지역성을 확보

        - 각 zone으로 전송된 메시지는 실시간 양방향 데이터 미러링 도구가 상대 zone으로 메시지를 전달

      > 개발자는 어느 데이터센터에 서비스가 배포되는지 관심 가질 필요가 없고, 또 데이터센터 장애 상황 시 안전한 데이터센터로 최상단 트래픽만 모두 옮겨주면 정상적인 Kafka가 모든 메시지를 받게 된다
    >
    - **Consumer 정책**

      양방향 데이터 미러링으로 인해 양쪽 Kafka 클러스터에 데이터가 100%가 존재한다. 따라서 Consumer는 2배로 소비하는 것을 피하기 위해 **한쪽 Kafka 클러스터에서 소비를 하는 Active-Standby 구조**로 구성

        - 언제나 한쪽 Kafka만 바라보도록 통제하기 위해 GSLB(Global Server Load Balancing)라는 DNS를 활용하도록 구성
        - 서로 독립된 Kafka 클러스터는 Offset 정보가 서로 다르다(시작점이 달라서 생기는 차이)

          ⇒ 이 차이 때문에 Standby로 Consumer가 넘어갈 때는 Standby 클러스터 Offset 순서에 맞게 보정된 값이 저장돼있어야 한다

            <aside>
            💡

          데이터센터 1의 Consumer Group Offset을 데이터센터 2에 맞는 Offset으로 변환해서 저장해주는 도구가 **Offset Sync**이다

            </aside>
