> 분산 시스템 맥락에서는 정확히 한 번(Exactly-once) 메시지 전달은 불가능하다.
>
- 웹 브라우저 ↔ 서버 → 분산
- 서버 ↔ 데이터베이스 → 분산
- 서버 ↔ 메시지 큐 → 분산

분산 시스템에서 **메시지가 전달되었는지, 수신 확인 응답(ACK)이 전달되었는지**를 100% 확신할 수 없기 때문이다. 네트워크 지연, 파티션, 프로세스 충돌 등 비동기성(Asynchrony)과 **불확실성** 때문에 발신자는 메시지가 **'딱 한 번만'** 처리되었음을 보장할 수 없다.

## 메시지 전달 방식

- **At-most-once**
    - 최대 한 번 전달 → 중복은 없음, 대신 유실 가능
- **At-least-once**
    - 최소 한 번 → 재전송 가능, 중복 가능
- **Exactly-once**
    - 정확히 한 번 → *이론적 개념 (실현 불가능)*
    - 분산 시스템의 비동기적 특성상 100% 보장이 불가능

## Exactly-once가 불가능한 이유

- **두 장군 문제(Two Generals Problem)**
    - 두 장군이 계곡을 사이에 두고 공격 시간을 합의해야 하지만, 메시지 전달이 실패할 수 있어 상대방이 자신의 합의 메시지를 받았는지 **결코 확신할 수 없는** 문제 → ACK의 불확실성으로 인한 **무한 루프**에 빠짐
    - **메시지 전달과 ACK 전달 중 하나가 실패했을 때** 발신자가 재시도할지 말지를 결정할 수 없음을 보여줌
- **비잔틴 장군 문제(Byzantine Generals Problem)**
    - 노드가 **임의의 거짓 행동**을 할 수 있음
        - ex) 어떤 장군에게는 '오전 9시'를 전달하고, 다른 장군에게는 '오전 10시'를 전달
    - 신뢰할 수 없는 환경에서 일관성을 보장하는 것이 얼마나 어려운지를 보여줌
- **FLP 불가능성 이론(Fisher, Lynch, Paterson)**
    - 비동기 분산 시스템에서 프로세스 중 하나라도 결함이 발생할 가능성이 있다면, 모든 프로세스가 결정에 합의하는 것은 불가능하다는 것

<aside>
💡

메시지를 보낸 후 수신 확인(ACK)을 기다리는 과정에서 너무 많은 변수가 발생한다. 이러한 불확실성 때문에, 발신자는 재전송(At-Least-Once)과 유실(At-Most-Once) 중 하나를 선택해야 하며, 딱 한 번을 보장할 수는 없다.

- 처리 전에 ack (At-most-once)
    - 아주 빠르지만, 수신자가 처리 중 죽으면 데이터는 영영 사라진다
- 처리 후에 ack (At-least-once)
    - 신뢰성은 높지만, 수신자가 처리만 하고 ack 보내기 전에 죽으면 → **중복 처리 필요**

## Faking It

> 실무적 해법으로 Exactly-Once을 흉내내는 기법을 사용한다. (정확히 한 번처럼 보이도록 속인다)
>

**실제 분산 시스템**(Kafka, RabbitMQ)은 **At-Least-Once를** 기본으로 보장하고, **애플리케이션 계층**에서 **Exactly-Once**처럼 작동하도록 두 가지 핵심 전략을 사용한다.

- **멱등성 (Idempotency) 확보**
    - 메시지 또는 작업이 몇 번을 적용되어도 시스템의 최종 상태가 동일하게 유지되도록 설계
    - ex) 계좌 잔액에 100원을 더하라"는 메시지는 멱등성이 없음 (두 번 처리되면 200원이 더해짐). 하지만 "계좌 잔액을 500원으로 설정하라"는 메시지는 멱등성이 있음 (두 번 처리되어도 최종 상태는 500원)
    - **Zab 프로토콜:** ZooKeeper의 기반인 Zab 프로토콜은 상태 기계 복제(State-Machine Replication)를 사용하며, 작업을 **멱등적인 상태 변경**으로 만들어 At-Least-Once로 충분하도록 구현을 단순화
- **중복 제거 (Deduplication)**
    - 각 메시지에 고유한 ID를 부여하여, 수신 측이 이미 처리했던 ID의 메시지가 재전송되면 **무시**하고 버리는 방식
    - **복잡성:** 이 중복 제거 시스템을 구현하려면 **별도의 저장소**에 처리된 ID 목록을 저장하고, 이 목록에 대한 접근을 **강하게 일관되게** 관리해야 하는 복잡성이 따름
        - ex) Kafka가 ZooKeeper를 사용하여 중복 제거 조정을 처리

## **At-Most-Once와 At-Least-Once의 구현**

### At-Most-Once (최대 한 번)

- 수신자가 메시지를 **처리하기 전에** ACK를 발신자에게 보낸다
- **리스크:** 수신자가 ACK를 보낸 후 메시지 처리 중 충돌하면 **데이터가 유실**
- 메시지 브로커는 작업이 다른 워커에게 전달되는 것을 막기 위해 강한 일관성을 유지해야 한다

### At-Least-Once (최소 한 번)

- 수신자가 메시지를 **처리한 후에** ACK를 발신자에게 보낸다
- **리스크:** 수신자가 처리 후 ACK를 보내기 전에 충돌하거나 ACK가 유실되면, 발신자가 **재전송하여 중복 처리**가 발생한다
    - Exactly-Once를 흉내 내기 위한 **멱등성** 또는 **중복 제거**가 필요한 이유

## Kafka는 Exactly-once

> Kafka는 0.11 버전부터 **EOS(Exactly-Once Semantics)** 를 지원하는데, 이것은 *전달(delivery)* 이 아니라 **처리 결과(processing)** 가 정확히 한 번 발생하도록 보장하는 기능이다
>
- **Idempotent Producer**
    - Producer ID + Sequence Number로 브로커가 **중복 메시지를 자동 제거**
    - Producer는 최소 한 번 재전송할 수 있지만, Broker는 이를 한 번만 commit하도록 해준다
- **Transactional Producer**
    - EOS를 가능하게 하는 핵심 기능
    - `consume → process → produce → offset commit`

      이 모든 작업을 **단일 트랜잭션**으로 묶어 원자적으로 처리
